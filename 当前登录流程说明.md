# 当前登录流程说明

本文档基于当前仓库代码整理，目标是把“登录相关能力现在到底是怎么跑起来的”讲清楚，包括：

- 对外接口有哪些
- 各模块分别负责什么
- 密码登录、短信登录、邮箱登录分别怎么走
- 图形验证码、短信/邮箱验证码、会话是怎么存的
- 当前默认配置是什么
- 当前实现里有哪些需要特别注意的现状

本文档描述的是“当前代码已经实现的事实”，不是理想设计图。

## 1. 总体结论

当前登录体系由 3 个模块共同完成：

- `luoluo-system`
  - 对外暴露公共接口
  - 目前主要负责图形验证码和“公共验证码发送入口”
- `luoluo-member`
  - 负责登录编排、登录策略、会员资料读取、退出登录
- `luoluo-common`
  - 负责 Redis、通用验证码服务、认证会话服务、Sa-Token、统一响应封装

当前已经支持 3 种登录方式：

- `password`：账号密码登录
- `sms`：手机号 + 短信验证码登录
- `email`：邮箱 + 邮箱验证码登录

当前登录态使用的是：

- Sa-Token 负责真正的 token 登录态
- `sys_auth_session` 表负责把会话摘要落库

当前验证码使用的是：

- Redis 做短期校验和一次性消费
- `sys_verify_code_record` 表做留痕

## 2. 当前模块职责拆分

### 2.1 `luoluo-system` 做什么

它不是登录核心实现层，而是“公共入口层”。

当前主要暴露两个公共接口：

- `GET /luoluo/system/public/captcha/image`
- `POST /luoluo/system/public/member/auth/code/send`

这两个接口的职责是：

- 生成图形验证码
- 在发送短信/邮箱验证码之前做图形验证码校验
- 做同一目标（同手机号/同邮箱）的 Redis 限流

另外，`luoluo-system` 还暴露了一个公共登录入口：

- `POST /luoluo/system/public/member/auth/login`

但它本质上只是直接转调 `luoluo-member` 的登录服务，本身没有额外风控逻辑。

### 2.2 `luoluo-member` 做什么

它是登录业务主模块，负责：

- 登录请求编排
- 密码/短信/邮箱三种策略选择
- 用户查询
- 会话创建
- 退出登录
- 当前登录用户资料查询

当前主要接口：

- `POST /luoluo/member/auth/login`
- `POST /luoluo/member/auth/logout`
- `POST /luoluo/member/auth/send-code`
- `GET /luoluo/member/profile/me`

### 2.3 `luoluo-common` 做什么

它提供底层基础设施：

- `VerifyCodeService`
  - 通用验证码签发、缓存、核销
- `AuthSessionService`
  - 通用会话签发、落库、注销
- `RedisClient`
  - Redis 常用操作封装
- Sa-Token 相关配置
- 统一响应对象 `R<T>`

## 3. 当前对外接口清单

### 3.1 图形验证码

`GET /luoluo/system/public/captcha/image`

用途：

- 获取图形验证码图片
- 返回 `captchaKey`、`imageBase64`、`ttlSeconds`

说明：

- 返回的 `imageBase64` 实际上是 SVG 的 Base64
- 前端可直接拼成：

```text
data:image/svg+xml;base64,${imageBase64}
```

### 3.2 公共验证码发送接口

`POST /luoluo/system/public/member/auth/code/send`

用途：

- 给手机号发送短信验证码
- 给邮箱发送邮箱验证码

它会额外做两件事：

- 图形验证码校验
- 发送频率限制

### 3.3 登录接口

有两个路径，底层逻辑相同：

- `POST /luoluo/member/auth/login`
- `POST /luoluo/system/public/member/auth/login`

二者最终都会进入同一套 `MemberAuthService`。

### 3.4 退出登录

`POST /luoluo/member/auth/logout`

要求：

- 必须已登录
- 通过 `@MemberLoginRequired` 校验

### 3.5 当前登录用户资料

`GET /luoluo/member/profile/me`

要求：

- 必须已登录

### 3.6 统一响应格式

大多数登录相关接口返回统一结构：

```json
{
  "code": 0,
  "message": "xxx",
  "data": {}
}
```

其中：

- `code=0` 表示成功
- `message` 是提示文案
- `data` 是业务数据

## 4. 当前核心存储结构

## 4.1 用户表 `sys_user`

用途：

- 存系统用户主数据
- 登录成功后返回的用户摘要就是从这里来的

关键字段：

- `user_id`
- `username`
- `password_hash`
- `phone`
- `email`
- `display_name`
- `user_type`
- `status`
- `deleted`

当前查询登录用户时，只认：

- `status = ACTIVE`
- `deleted = 0`

### 4.2 验证码表 `sys_verify_code_record`

用途：

- 记录每次验证码签发
- 验证成功后把 `used` 更新为 `true`

关键字段：

- `biz_type`
- `target_type`
- `target_value`
- `channel`
- `provider`
- `request_id`
- `code_value`
- `expires_at`
- `used`
- `used_at`

当前登录验证码的典型取值：

- `biz_type = LOGIN`
- `target_type = sms` 或 `email`
- `subject_type = SYS_USER`

注意：

- 当前代码里验证码是明文写入 `code_value`
- Redis 里也是明文校验码
- 这和更严格的“只存摘要”设计不同，是当前实现现状

### 4.3 会话表 `sys_auth_session`

用途：

- 落库存档登录会话摘要

关键字段：

- `subject_id`
- `subject_type`
- `login_type`
- `token_digest`
- `expires_at`
- `revoked`
- `device_type`
- `client_ip`

注意：

- 表里存的是 `token_digest`，不是明文 token
- 明文 token 只在登录响应里返回给客户端

## 5. Redis Key 现状

当前登录相关 Redis Key 主要有 3 类。

### 5.1 登录验证码

格式：

```text
auth:verify_code:{namespace}:{bizType}:{targetType}:{targetValue}
```

当前登录场景实际类似：

```text
auth:verify_code:member_auth:LOGIN:sms:13800000000
auth:verify_code:member_auth:LOGIN:email:demo@example.com
```

特征：

- 只要校验成功就会被删掉
- 是一次性验证码

### 5.2 图形验证码

格式：

```text
auth:image_captcha:{namespace}:{captchaKey}
```

当前发送会员验证码时使用的命名空间是：

```text
member_send_code
```

### 5.3 验证码发送限流

格式：

```text
auth:send_limit:{namespace}:{targetType}:{targetValue}
```

当前发送会员验证码时使用的命名空间同样是：

```text
member_send_code
```

## 6. 密码登录流程

### 6.1 请求入口

客户端可调用任一登录接口：

- `POST /luoluo/member/auth/login`
- `POST /luoluo/system/public/member/auth/login`

典型请求体：

```json
{
  "grantType": "password",
  "deviceType": "web",
  "username": "demo_user",
  "password": "123456"
}
```

也支持用下面任一字段作为账号标识：

- `username`
- `email`
- `phone`

### 6.2 服务调用链

当前密码登录的核心调用链是：

1. Controller 接收请求
2. `MemberAuthApplicationService.login(...)`
3. `MemberAuthService.login(...)`
4. 根据 `grantType=password` 选择 `PasswordLoginStrategy`
5. `MemberUserService.findActiveByPasswordAccount(...)` 查用户
6. 用 `PasswordEncoder.matches(...)` 校验密码
7. 校验通过后 `MemberSessionService.createSession(...)`
8. `AuthSessionService.createSession(...)`
9. Sa-Token 登录并签发 token
10. 把 token 的 SHA-256 摘要落到 `sys_auth_session`
11. 返回登录响应

### 6.3 密码校验细节

当前密码策略支持两类校验：

- 正常情况：BCrypt 校验
- 兼容情况：允许明文密码直接匹配

是否允许明文密码由配置控制：

- `app.member.auth.allow-plain-password`

当前仓库提交的 `luoluo-admin/src/main/resources/application.yml` 中：

- `allow-plain-password: true`

这意味着当前代码默认保留了“本地演示明文密码兼容”的能力。

### 6.4 密码登录查人规则

密码登录时会按下面顺序解析账号输入：

1. `username`
2. `email`
3. `phone`

查询条件是：

- `username = account`
  或
- `phone = account`
  或
- `email = account.toLowerCase()`

并且必须：

- `status = ACTIVE`

### 6.5 密码登录成功响应

返回字段包括：

- `token`
- `tokenType`
- `expiresIn`
- `grantType`
- `user`

其中 `user` 是用户摘要：

- `userId`
- `username`
- `displayName`
- `phone`
- `email`
- `userType`

## 7. 短信登录流程

### 7.1 标准推荐流程

短信登录推荐按下面顺序调用：

1. 先获取图形验证码
2. 再发送短信验证码
3. 最后用手机号 + 短信验证码登录

### 7.2 第一步：获取图形验证码

调用：

```text
GET /luoluo/system/public/captcha/image
```

返回示意：

```json
{
  "code": 0,
  "message": "image captcha created",
  "data": {
    "captchaKey": "xxx",
    "imageBase64": "xxx",
    "ttlSeconds": 120
  }
}
```

### 7.3 第二步：发送短信验证码

推荐调用：

```text
POST /luoluo/system/public/member/auth/code/send
```

请求体示意：

```json
{
  "grantType": "sms",
  "phone": "13800000000",
  "captchaKey": "上一步返回的 captchaKey",
  "captchaCode": "用户看到并输入的图形验证码"
}
```

### 7.4 公共验证码发送入口内部做了什么

`PublicMemberAuthApplicationService.sendCode(...)` 的逻辑是：

1. 如果开启了图形验证码，则先校验 `captchaKey + captchaCode`
2. 解析 `grantType`
3. 提取目标值
   - `sms` 时取 `phone`
   - `email` 时取 `email`
4. 根据 `app.member.auth.verify-code-send-interval-seconds` 做限流
5. 调 `MemberAuthApplicationService.sendCode(...)`
6. 再由 `MemberSendCodeService` 根据 `grantType` 分派到对应发送策略

### 7.5 短信发送策略内部做了什么

`SmsSendCodeStrategy.send(...)` 的流程：

1. 校验手机号不能为空
2. 生成 6 位随机验证码
3. 读取验证码 TTL
4. 调用 `MemberSmsSender.sendLoginCode(...)`
5. 记录明文验证码日志（是否打印由配置决定）
6. 调 `MemberVerifyCodeService.recordAndCache(...)`
7. Redis 写入登录验证码
8. 数据库插入一条 `sys_verify_code_record`
9. 返回发送结果

### 7.6 短信通道当前支持情况

当前短信发送器有两种实现思路：

- Mock 短信发送器
- 阿里云短信发送器

从当前代码行为看：

- 如果没有启用阿里云短信，通常走 mock 发送
- 是否把验证码直接回传给前端，取决于：
  - `allow-mock-verify-code`
  - `receipt.debugCodeVisible`

当前仓库提交的正式配置里：

- `app.member.auth.allow-mock-verify-code: false`
- `app.member.sms.aliyun.enabled: false`

这意味着：

- 如果按提交态直接启动，又没有补齐真实短信配置，短信发送接口很可能返回 `SMS sender is not configured`

而 `.env.example` 里给的是另一套偏本地调试的值：

- `MEMBER_AUTH_ALLOW_MOCK_VERIFY_CODE=true`

所以“仓库当前提交配置”和“环境变量示例配置”是两种不同使用姿势。

### 7.7 第三步：短信验证码登录

调用：

```text
POST /luoluo/member/auth/login
```

请求体示意：

```json
{
  "grantType": "sms",
  "deviceType": "web",
  "phone": "13800000000",
  "smsCode": "123456"
}
```

### 7.8 短信登录内部做了什么

`SmsLoginStrategy.authenticate(...)` 的流程：

1. 校验 `phone`、`smsCode` 不能为空
2. 调 `MemberVerifyCodeService.verifyAndConsume("sms", phone, smsCode)`
3. 底层 `VerifyCodeService.verifyAndConsume(...)`
   - 先校验 mock code 是否放行
   - 再去 Redis 做 compare-and-delete
   - 再把数据库对应记录更新为 `used=true`
4. 验证码通过后，按手机号查用户
5. 找到用户后创建会话并返回 token

### 7.9 验证码核销的重要特征

当前短信/邮箱验证码核销是“一次性消费”：

- Redis 校验通过会立刻删除 key
- 数据库记录会被更新为 `used=true`

同一个验证码不能重复用。

## 8. 邮箱登录流程

邮箱登录和短信登录结构完全平行，只是发送通道不同。

### 8.1 发送邮箱验证码

推荐调用：

```text
POST /luoluo/system/public/member/auth/code/send
```

请求体示意：

```json
{
  "grantType": "email",
  "email": "demo@example.com",
  "captchaKey": "上一步返回的 captchaKey",
  "captchaCode": "用户输入的图形验证码"
}
```

### 8.2 邮箱验证码发送逻辑

`EmailSendCodeStrategy.send(...)` 会：

1. 校验邮箱不能为空
2. 统一转为小写
3. 生成 6 位随机验证码
4. 按模板拼装邮件正文
5. 调用 `CommonMailSender.sendText(...)`
6. 记录验证码日志
7. 记录验证码到 Redis 和 `sys_verify_code_record`
8. 返回请求流水、脱敏邮箱、有效期等

### 8.3 邮件发送器当前行为

`CommonMailSender` 的逻辑比较关键：

- 如果没有真实 `JavaMailSender` 或没有配置 `spring.mail.host`
  - 不真发邮件
  - 直接返回一个 mock receipt
- 如果有真实邮件配置
  - 通过 Spring Mail 发出

接着 `EmailSendCodeStrategy` 会判断：

- 如果不允许 mock 验证码，但当前 receipt 又是 debug/mock 可见的
  - 直接报错 `Email sender is not configured`

所以邮箱验证码发送与短信发送一样，也有“真实发送模式”和“本地 mock 模式”两种运行姿势。

### 8.4 邮箱登录

调用：

```text
POST /luoluo/member/auth/login
```

请求体示意：

```json
{
  "grantType": "email",
  "deviceType": "web",
  "email": "demo@example.com",
  "emailCode": "123456"
}
```

`EmailLoginStrategy.authenticate(...)` 会：

1. 校验 `email`、`emailCode`
2. 邮箱统一小写
3. 核销邮箱验证码
4. 按邮箱查用户
5. 成功后创建会话并返回 token

## 9. 会话创建与登录态管理

### 9.1 会话创建流程

任一登录策略认证成功后，都会走：

- `MemberSessionService.createSession(...)`
- `AuthSessionService.createSession(...)`

底层做的事情：

1. 校验 `subjectId`、`ttlSeconds`
2. 规范化 `loginType`
3. 决定 `deviceType`
   - 如果请求没传，就用 `loginType`
4. 调 `StpUtil.login(...)`
5. 读取 Sa-Token 当前 token 值
6. 计算 token 的 SHA-256 摘要
7. 插入 `sys_auth_session`
8. 返回明文 token 给前端

### 9.2 当前会话字段含义

当前会话记录里：

- `subject_type` 固定是 `SYS_USER`
- `login_type` 是 `PASSWORD/SMS/EMAIL` 语义值，但代码中写入时会被规范化成大写
- `device_type` 取请求里的 `deviceType`，如果为空则回退到登录类型
- `client_ip` 从请求头或 remote addr 里取

### 9.3 token 读取方式

当前 `sa-token` 配置：

- `token-name: Authorization`
- `token-prefix: Bearer`
- `is-read-header: true`
- `is-read-body: true`
- `is-read-cookie: false`

所以前端最标准的带 token 方式是：

```text
Authorization: Bearer {token}
```

## 10. 退出登录流程

调用：

```text
POST /luoluo/member/auth/logout
```

流程：

1. `@MemberLoginRequired` 先校验当前是否已登录
2. `MemberAuthApplicationService.logout()`
3. `MemberAuthService.logoutCurrentSession()`
4. `MemberSessionService.logoutCurrentSession()`
5. `AuthSessionService.logoutCurrentSession()`
6. 用当前 token 做摘要定位数据库会话
7. 把 `sys_auth_session.revoked` 更新为 `true`
8. 调 `StpUtil.logoutByTokenValue(...)`

效果：

- Sa-Token 登录态失效
- 数据库会话也被标记撤销

## 11. 当前登录用户资料查询流程

调用：

```text
GET /luoluo/member/profile/me
```

流程：

1. `@MemberLoginRequired` 保证用户已登录
2. `MemberSessionService.getCurrentLoginUserId()`
3. 底层从 Sa-Token 取当前 `loginId`
4. 按 `userId` 查询 `sys_user`
5. 返回当前登录用户资料

这也说明：

- 当前系统是“token -> Sa-Token loginId -> 用户表”的读法
- 不是每次都反查 `sys_auth_session` 来识别当前用户

## 12. 当前默认配置现状

这里需要特别说明，仓库里同时存在两套“默认感受”：

### 12.1 `luoluo-admin/src/main/resources/application.yml` 提交态

当前提交态关键配置是：

- `app.member.auth.image-captcha-enabled: true`
- `app.member.auth.plain-verify-code-log-enabled: true`
- `app.member.auth.allow-mock-verify-code: false`
- `app.member.auth.mock-verify-code: 123456`
- `app.member.auth.allow-plain-password: true`
- `app.member.auth.session-ttl-seconds: 86400`
- `app.member.auth.verify-code-ttl-seconds: 120`
- `app.member.auth.verify-code-send-interval-seconds: 60`

这意味着按提交态理解：

- 发验证码前需要图形验证码
- 会打印验证码明文日志
- 不允许直接使用 mock 验证码通关
- 仍然兼容明文密码登录

### 12.2 `.env.example` 示例态

`.env.example` 给的是更偏本地调试的一套环境变量示例：

- `MEMBER_AUTH_IMAGE_CAPTCHA_ENABLED=false`
- `MEMBER_AUTH_ALLOW_MOCK_VERIFY_CODE=true`
- `MEMBER_AUTH_MOCK_VERIFY_CODE=123456`
- `MEMBER_AUTH_ALLOW_PLAIN_PASSWORD=true`

所以本项目当前非常依赖“你实际怎么注入环境变量”。

`luoluo-member/src/main/resources/data.sql` 里提供了示例数据：

### 13.1 演示用户

- 用户名：`demo_user`
- 密码：`123456`
- 手机号：`13800000000`
- 邮箱：`demo@example.com`
- 昵称：`Demo User`

### 13.2 演示验证码记录

预置了两条 `LOGIN` 验证码记录：

- 手机验证码：`123456`
- 邮箱验证码：`123456`

注意：

- 这些是数据库初始化演示数据
- 实际运行时验证码校验优先依赖 Redis
- 如果 Redis 里没有对应 key，仅数据库里有记录并不能单独完成核销

也就是说：

- “只插数据库演示记录”不等于“运行时一定能成功登录”
- 真正能登录还取决于 Redis 中是否存在对应验证码缓存，或者是否开启 mock code 放行

## 14. 当前实现中的几个关键现状

下面这些不是猜测，而是从当前代码直接能看出来的现状。

### 14.1 公共登录接口和会员登录接口共用一套登录逻辑

当前：

- `/luoluo/system/public/member/auth/login`
- `/luoluo/member/auth/login`

最终都只是调用 `MemberAuthApplicationService.login(...)`。

也就是说：

- 公共登录入口没有额外风控
- 它不是另一套登录体系

### 14.2 登录本身不校验图形验证码

当前图形验证码只加在“发送短信/邮箱验证码”之前，不加在最终登录接口上。

### 14.3 存在一个可直接调用的内部验证码发送接口

当前 `luoluo-member` 里还有：

- `POST /luoluo/member/auth/send-code`

这个接口直接进入 `MemberSendCodeService`，不经过 `PublicMemberAuthApplicationService`，因此：

- 不做图形验证码校验
- 不做 Redis 发送频率限制

从“当前实现事实”来看，这是一个需要非常明确知晓的现状。

如果未来该接口会直接暴露给外部，通常需要额外考虑：

- 是否应该下线
- 是否应该只给内部调用
- 是否也补上风控

### 14.4 API 前缀配置目前没有直接体现在控制器路径里

虽然配置里有：

```yaml
app:
  api-prefix: /api/v1
```

但当前控制器上的 `@RequestMapping` 仍然是直接写死的：

- `/luoluo/member/auth/...`
- `/luoluo/system/public/...`

所以从当前代码事实看，接口路径不自动带 `/api/v1`。

## 15. 推荐的调用顺序示例

### 15.1 密码登录

```text
POST /luoluo/member/auth/login
```

```json
{

  "deviceType": "web",
  "username": "demo_user",
  "password": "123456"
```

```text
GET /luoluo/system/public/captcha/image
```

```json
{
  "grantType": "sms",
  "phone": "13800000000",
  "captchaKey": "xxx",
  "captchaCode": "ABCD"
}
```

3. 用短信验证码登录

```json
{
  "grantType": "sms",
  "deviceType": "web",
  "phone": "13800000000",
  "smsCode": "123456"
}
```

### 15.3 邮箱登录

1. 获取图形验证码
2. 发送邮箱验证码
3. 用邮箱验证码登录

登录请求体：

```json
{
  "grantType": "email",
  "deviceType": "web",
  "email": "demo@example.com",
  "emailCode": "123456"
}
```

## 16. 一句话总结

当前登录体系已经具备完整可运行的：

- 密码登录
- 短信验证码登录
- 邮箱验证码登录
- 图形验证码校验
- Redis 限流
- Sa-Token 会话签发
- 资料查询
- 退出登录

但它同时也保留了很明显的“演示/调试特征”：

- 明文密码兼容
- mock 验证码模式
- 公共登录和内部登录双入口
- 直接发送验证码的 member 内部接口
- 当前提交态与 `.env.example` 的默认行为并不一致

如果后续你要继续扩展前端联调、网关鉴权、风控治理或者安全加固，这份文档里的这些“现状边界”会非常关键。
