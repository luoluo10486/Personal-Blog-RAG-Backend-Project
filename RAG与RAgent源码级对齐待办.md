# RAG 与 RAgent 源码级对齐待办

## 1. 范围和当前结论

本文档用于记录当前项目 `D:\develop\Personal-Blog-RAG-Backend-Project` 与参考项目 `D:\develop\RAgent\ragent` 在 **排除登录、鉴权、用户体系** 后，距离“源码级一致”还剩哪些差异，以及后续应按什么流程继续补齐。

当前结论：**还没有达到除了登录外源码级一致**。上一轮已经把知识库文档、分块、定时刷新、pgvector 主链路、RustFS 配置等部分往 RAgent 靠近了，但仍存在较大的结构性差异，尤其集中在数据库 ID 模型、向量核心抽象、ingestion 模块形态、RAG 检索链路、意图分类链路、Controller/DTO 契约和模块/package 布局。

需要特别注意：当前项目不是 RAgent 的同构模块结构。当前项目主模块是 `luoluo-knowledge`、`luoluo-infra-ai`、`luoluo-rag-mcp` 等；RAgent 主模块是 `bootstrap`、`framework`、`infra-ai`、`mcp-server`。如果目标是真正“源码级一样”，就不只是补行为，还要尽量让类名、方法签名、DTO、包结构、SQL 字段、配置项都收敛。

## 4. 详细完善流程

### 阶段 0：冻结对齐口径

目标：避免一边改行为、一边改结构导致反复返工。

1. 明确“源码级一致”的最小范围：建议限定为 RAG/knowledge/ingestion/infra-ai/mcp-server 相关代码，继续排除登录、鉴权、用户、权限模块。
2. 建立对比基准：固定 RAgent 参考目录为 `D:\develop\RAgent\ragent`，固定本项目主目录为 `D:\develop\Personal-Blog-RAG-Backend-Project`。
3. 明确保留项：当前项目的非 RAG 业务模块，例如 admin、system、member、历史博客业务，不纳入源码级一致要求。
4. 建立差异清单脚本：按 `class basename`、`package path`、`endpoint`、`SQL table`、`application.yaml` 五类输出 diff，后续每轮改完都跑。

验收标准：

1. 有一份可重复执行的差异扫描命令或脚本。
2. 差异清单里登录、鉴权、用户相关项可以被稳定过滤。
3. 后续每次改造都有“本轮消灭了哪些差异”的记录。

### 阶段 1：数据库和 ID 模型收敛

目标：先解决最底层的结构差异，否则服务源码无法真正贴近 RAgent。

1. 逐表对比本项目 `script/sql/rag/baseline/B20260423__rag_schema.sql` 与 RAgent `resources/database/schema_pg.sql`。
2. 对每张 RAG 相关表输出字段 diff：字段名、类型、默认值、not null、索引、唯一约束、comment。
3. 重点处理 ID 类型：如果追求源码级一致，应将知识库、文档、分块、会话、消息、ingestion 等表逐步迁移到 RAgent 的 `VARCHAR(20)` ID 口径。
4. 重点处理 `t_knowledge_vector`：决定是否从当前丰富向量引用表改成 RAgent 的简洁 pgvector 表。
5. 同步修改 Entity、Mapper XML/注解、DTO、Controller 参数类型、Service 方法签名。
6. 设计数据迁移脚本：如果现有库已有数据，要提供 bigint 到 varchar 的兼容迁移或重建方案。

验收标准：

1. 除登录用户表外，RAG 相关 DDL 与 RAgent 字段级一致，或者只保留有明确说明的项目特有字段。
2. Entity 字段类型与 SQL 完全一致。
3. Mapper 查询不再出现 Long/String 混用。
4. `mvn -DskipTests compile` 通过。

### 阶段 2：向量核心抽象收敛

目标：让知识库、分块、检索都使用 RAgent 的向量接口，而不是本项目自定义接口。

1. 引入或迁移 RAgent 的 `VectorChunk`。
2. 引入或迁移 RAgent 的 `rag/core/vector/VectorStoreService` 接口，包含 `indexDocumentChunks`、`updateChunk`、`deleteDocumentVectors`、`deleteChunkById`、`deleteChunksByIds` 等方法。
3. 将当前 `KnowledgeVectorDocument`、`KnowledgeVectorSpace`、`KnowledgeVectorSpaceId` 的使用点逐步替换为 RAgent 抽象。
4. 按 RAgent 的 `PgVectorStoreService` 重写当前 pgvector 实现。
5. 明确 Milvus：当前主链路不用 Milvus，可以不接入运行配置；但如果要源码级一致，需要保留 RAgent 的 Milvus 实现类或提供同名空实现/条件装配。
6. 检查向量 metadata：确保 docId、kbId、chunkId、source、score 等字段与 RAgent 检索期望一致。

验收标准：

1. 知识库服务不再直接依赖本项目自定义 vector/ref 模型。
2. PgVector 读写方法签名与 RAgent 一致。
3. 上传文档后能在 pgvector 表中查到对应 embedding。
4. 删除文档、删除分块、禁用文档会同步删除或过滤向量。

### 阶段 3：知识库文档与分块服务收敛

目标：把当前“行为接近”的实现改成“源码结构接近”。

1. 对比 RAgent 的 `KnowledgeDocumentServiceImpl`，补齐同名私有方法和执行流：`runChunkTask`、`runChunkProcess`、`runPipelineProcess`、`persistChunksAndVectorsAtomically`。
2. 用 RAgent 的 `DocumentParserSelector`、`ChunkEmbeddingService`、`TransactionOperations` 思路替换当前分散的本地执行逻辑。
3. 将当前 `KnowledgeIngestionEngine.execute(...)` 在普通文档分块链路中的职责降级，只保留 pipeline 模式需要的部分，或者完全切换为 RAgent pipeline 内核。
4. 对齐 `KnowledgeChunkServiceImpl`：创建、更新、批量启停、删除时都走 RAgent 的 vector service 方法。
5. 对齐 content hash、token count、chunk index、enabled、deleted、status 的字段更新规则。
6. 对齐消息队列事件：文档上传、分块完成、向量化完成、失败事件的 topic/tag/payload 要逐项比对。

验收标准：

1. 文档上传、URL 导入、文本导入、重新分块、启用/禁用、删除文档行为与 RAgent 一致。
2. 分块新增、修改、删除、批量启停后，数据库和向量库状态一致。
3. 失败时状态、错误信息、日志记录与 RAgent 一致。
4. 单元或集成 smoke 测试覆盖上传到 RustFS、解析、分块、embedding、写 pgvector。

### 阶段 4：Ingestion 模块源码级重构

目标：补齐当前与 RAgent 差距最大的 ingestion 包。

1. 新增 RAgent 同款 `ingestion/controller/IngestionPipelineController` 和 `ingestion/controller/IngestionTaskController`。
2. 新增或迁移 `ingestion/controller/request`、`ingestion/controller/vo` 下的请求/响应模型。
3. 新增或迁移 `ingestion/service/IngestionPipelineService`、`ingestion/service/impl/IngestionPipelineServiceImpl`、`ingestion/service/IngestionTaskService`、`ingestion/service/impl/IngestionTaskServiceImpl`。
4. 新增或迁移 `ingestion/domain/context`、`ingestion/domain/pipeline`、`ingestion/domain/settings`、`ingestion/domain/enums`、`ingestion/domain/result`。
5. 将本项目已有 `knowledge/dto/ingestion` record 模型迁移为 RAgent 的 class/builder 风格，避免 DTO 契约不一致。
6. 将本项目 `service/ingest/node` 节点映射到 RAgent 的 `IngestionNodeType`、`NodeConfig`、`PipelineDefinition`。
7. 保证 pipeline 执行不是“写了定义但没按定义跑”，而是真正按 `PipelineDefinition` 逐节点执行、记录任务节点状态、支持失败中断。

验收标准：

1. RAgent ingestion 两个 controller 的 endpoint、method、request、response 基本一致。
2. `t_ingestion_pipeline`、`t_ingestion_pipeline_node`、`t_ingestion_task`、`t_ingestion_task_node` 的写入规则一致。
3. Pipeline 创建、更新、删除、详情、任务启动、任务详情、任务日志全部可用。
4. 本项目旧 ingestion DTO/Service 没有残留公共入口，或只作为兼容层。

### 阶段 5：RAG 检索链路收敛

目标：把当前本地检索器体系迁移到 RAgent 的多通道检索体系。

1. 引入或迁移 `MultiChannelRetrievalEngine`。
2. 引入或迁移 `VectorGlobalSearchChannel`、`IntentDirectedSearchChannel`。
3. 引入或迁移 `CollectionParallelRetriever`、`IntentParallelRetriever` 等并行召回策略。

6. 对齐 trace 节点：query rewrite、intent classify、retrieve、rerank、generate 每一步的 trace 记录字段与 RAgent 一致。

验收标准：

## 5. 建议执行顺序

1. 先做数据库和 ID 模型决策，这是最大分水岭。如果继续保留 bigint，很多 RAgent 源码无法直接落地。
2. 再做向量核心抽象，把 `VectorChunk` 和 RAgent `VectorStoreService` 放到所有知识库链路的中心。
3. 然后重构知识库文档/分块服务，让上传、分块、embedding、pgvector 写入与 RAgent 一致。
4. 接着补齐 ingestion controller/service/domain，让 pipeline 真正按 RAgent 的 `PipelineDefinition` 执行。
5. 再迁移 RAG 多通道检索和 `DefaultIntentClassifier`，让回答链路与 RAgent 一致。
6. 最后做 Controller/DTO/VO、配置、自动装配、SQL comment 的扫尾。

## 6. 风险和注意事项

1. ID 类型迁移风险最高：如果已有线上或本地数据，bigint 改 varchar 会影响外键、Mapper、前端参数和历史数据。
2. `t_knowledge_vector` 当前比 RAgent 更丰富：改成 RAgent 简洁结构会减少本项目已有的向量引用信息，需要确认是否接受。
3. Milvus 当前主链路不需要：不要因为 RAgent 有 Milvus 类就强行启用 Milvus；源码可保留，运行链路应继续走 pgvector。
4. 登录、鉴权、用户体系继续排除：不要为了类名一致把 RAgent 的 user/auth 整套搬进来，除非后续明确改变范围。
5. 当前项目混有非 RAG 业务模块：源码级一致应聚焦 RAG 相关模块，否则会变成重写整个项目。
6. 每轮改完必须重新与 RAgent 对比：不能只看编译通过，必须看类、方法、SQL、endpoint、配置是否继续收敛。
