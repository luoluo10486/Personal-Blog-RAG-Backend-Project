## 《AI大模型RAG项目》第 1 小节：调用大模型 API

### 1. 什么是大模型 API

大模型 API 可以理解为“远程推理服务入口”。  
应用程序本身不负责生成答案，而是把请求组织好，再把推理工作交给模型服务。

它和普通后端接口的相似点是：

- 都是一次标准请求
- 都有入参和返回值
- 都要考虑超时、鉴权、重试和异常处理

它和普通接口最不一样的地方在于：

- 返回内容并不是固定结构结果，而是模型生成结果
- 输出天然带有不确定性
- token 消耗和时延是重要成本

### 2. 为什么要先理解 API，再理解 RAG

因为 RAG 最终仍然要落回一次模型调用。  
无论前面做了多少检索、重排、工具调用，最后“把资料组织成答案”这一步，仍然需要 Chat API 来完成。

所以可以把整个系统拆成两层：

- 上游：负责找到资料
- 下游：负责基于资料生成回答

### 3. 当前阶段最重要的认知

- Chat 模型负责回答
- Embedding 模型负责向量化
- Rerank 模型负责候选排序

这三类能力都可能来自“大模型服务”，但它们的职责完全不同。

### 4. 常见误区

- 误区一：只要能调通 Chat API，就等于已经做了 RAG  
  实际上这只完成了“生成”，还没有完成“检索增强”。

- 误区二：Embedding 也是回答模型  
  实际上 Embedding 不负责回答，只负责表示语义。

- 误区三：API 调用只是工程细节  
  实际上 API 的超时、成本、稳定性会直接影响整条 RAG 链路能不能真正可用。

## 《AI大模型RAG项目》第 2 小节：Prompt 工程入门

### 1. Prompt 的本质

Prompt 不是“问模型一句话”，而是对模型行为施加控制。

一个完整 Prompt 往往至少包含：

- 角色设定：你是谁
- 能力边界：你回答什么，不回答什么
- 行为规则：你必须遵守什么
- 输出格式：你最后怎么表达

### 2. 为什么 Prompt 在 RAG 中格外重要

在普通聊天里，Prompt 主要影响“风格和倾向”；  
在 RAG 里，Prompt 更重要的是影响“可信度和可控性”。

它决定模型是否会：

- 只依据资料回答
- 在资料不足时明确拒答
- 输出引用编号
- 避免补全资料中没有的细节

### 3. Prompt 的常见作用层次

可以把 Prompt 理解成三层：

1. 角色层  
   定义模型站在什么身份回答。

2. 规则层  
   规定模型能不能扩写、能不能猜测、资料不足怎么办。

3. 输出层  
   规定回答格式、引用格式、语言风格。

### 4. 一个重要判断

如果一个 RAG 系统已经有检索结果，但回答仍然经常发散、胡编、乱引用，问题往往不只在模型本身，也很可能在 Prompt 设计不够严格。

### 5. 常见误区

- 误区一：Prompt 只影响语言风格  
  实际上 Prompt 也影响真实性、边界感和引用行为。

- 误区二：Prompt 越长越好  
  过长的 Prompt 会增加成本，也可能让关键约束被淹没。真正重要的是结构清晰、规则明确。

- 误区三：只要写了“不要胡编”，模型就一定不会胡编  
  幻觉抑制要靠 Prompt、上下文组织、检索质量共同作用。

## 《AI大模型RAG项目》第 3 小节：什么是 RAG

### 1. RAG 的定义

RAG 是 Retrieval-Augmented Generation，意思是“检索增强生成”。

它不是一个单点技术，而是一种组合式系统设计：

- 用检索解决“模型不知道私有资料”的问题
- 用生成解决“资料不会自己组织成答案”的问题

### 2. RAG 在解决什么问题

纯大模型的强项：

- 语言组织能力强
- 表达自然
- 推理能力较好

纯大模型的短板：

- 不知道你的内部资料
- 容易过时
- 容易在陌生知识上补全

纯检索系统的强项：

- 能找到资料
- 可追溯
- 相对稳定

纯检索系统的短板：

- 不会把结果组织成自然回答
- 用户需要自己读资料

RAG 的价值就在于把这两者接起来。

### 3. 标准 RAG 的最小链路

```text
用户问题
-> 检索相关资料
-> 组装参考上下文
-> 调用模型生成答案
-> 返回答案与证据
```

### 4. 核心认知

> RAG 不是让模型凭空更聪明，而是让模型在回答前先接触到更相关的资料。

### 5. 常见误区

- 误区一：RAG 的核心是“模型更强”  
  实际上 RAG 的核心是“系统把相关知识在正确时机送到模型面前”。

- 误区二：只要接了向量库就是 RAG  
  没有生成环节、没有上下文组装、没有回答约束时，最多只是“检索系统”。

- 误区三：RAG 一定比纯模型更好  
  如果检索质量差、上下文混乱，RAG 反而可能把模型带偏。

## 《AI大模型RAG项目》第 4 小节：用 Apache Tika 解析文档

### 1. 文档解析为什么是第一步

知识库的原始输入通常是文件，而检索和生成依赖的是文本。  
所以必须先完成一次“文件 -> 文本”的转换。

### 2. Tika 的角色

Tika 的作用不是回答问题，而是把不同格式文档里的内容抽出来，变成后续步骤能继续处理的文本和元数据。

### 3. 这一层真正解决的问题

```text
如何把 PDF / Word / 文本文件变成可继续切块和入库的原始材料
```

### 4. 为什么这一步不能跳过

- 向量检索检索的是文本表示，不是原始文件
- chunk 切分依赖正文内容
- metadata 提取依赖解析结果

### 5. 常见误区

- 误区一：文档上传就等于知识库构建完成  
  解析只是起点，后面还要切块、向量化、入库。

- 误区二：解析出正文就够了  
  metadata 同样重要，它关系到后续的来源展示和管理。

## 《AI大模型RAG项目》第 5 小节：数据分块 Chunk 策略与实践

### 1. 为什么要切 Chunk

整篇文档直接进入检索系统，会带来三个问题：

- 太长，噪声太多
- 太杂，主题不集中
- 太大，浪费上下文窗口

所以需要把文本切成适合检索和生成的基本单位。

### 2. 好的 Chunk 不是越小越好

Chunk 太大：

- 检索命中后包含无关信息太多
- 模型阅读成本高

Chunk 太小：

- 语义容易断裂
- 单条信息不足以回答问题

因此 chunk 的目标是：

> 在“足够短”和“语义完整”之间找到平衡。

### 3. 为什么结构化切块更优

如果切块只按固定长度截断，很容易把：

- 标题和正文分开
- 列表切断
- 表格拆碎
- 一段规则切成两半

而结构化切块会尽量尊重文档天然边界，比如：

- 标题
- 段落
- 列表
- 表格
- 代码块

### 4. Overlap 的意义

相邻 chunk 之间保留少量重叠，不是为了重复，而是为了减少语义断裂。

它解决的问题是：

- 一段内容刚好跨块
- 后半段需要前半段上下文才能理解

### 5. 常见误区

- 误区一：Chunk 就是把字数均分  
  真正重要的是语义边界，而不是机械长度。

- 误区二：Overlap 越大越好  
  重叠过多会导致冗余召回和上下文浪费。

- 误区三：切块是后续再优化的小问题  
  实际上 chunk 策略直接影响检索质量，是 RAG 的基础能力之一。

## 《AI大模型RAG项目》第 6 小节：元数据的作用与管理

### 1. 什么是元数据

元数据可以理解为“描述数据的数据”。  
正文负责承载事实，元数据负责说明这条事实是什么、来自哪里、属于哪类。

### 2. 元数据为什么重要

即使两段正文内容都能回答问题，系统仍然需要元数据来完成：

- 来源展示
- 引用对齐
- 分类过滤
- 文档管理

### 3. 元数据在 RAG 里的三种价值

1. 检索价值  
   用于分类过滤、范围收缩、后续扩展条件查询。

2. 展示价值  
   用于告诉用户“答案来自哪份资料”。

3. 管理价值  
   用于追踪知识来源、版本和归属。

### 4. 核心认知

> 没有元数据，系统可能还能回答；但没有元数据，系统通常就不够可解释、可追溯、可维护。

### 5. 常见误区

- 误区一：只要正文够强，元数据可以不要  
  这会导致系统失去证据链和管理能力。

- 误区二：元数据只是前端展示字段  
  实际上它也会反过来影响检索和过滤能力。

## 《AI大模型RAG项目》第 7 小节：从文本到向量之理解 Embedding

### 1. Embedding 的本质

Embedding 是把文本映射到一个高维向量空间。  
在这个空间里，语义相近的文本通常会更接近。

### 2. 为什么 Embedding 对 RAG 重要

因为用户问的问题，和文档里写的原话，往往不是一模一样的。

例如：

- 用户说“七天内还能退吗”
- 文档可能写“签收后 7 天内支持无理由退货”

如果只做字面匹配，可能命中不稳；  
Embedding 的价值就是帮助系统识别“语义相近”。

### 3. Embedding 解决的不是回答，而是召回

它回答的是这个问题：

```text
哪些资料和当前问题在语义上更接近
```

而不是：

```text
最终答案怎么说
```

### 4. 一个重要判断

如果 Embedding 质量不佳，即使后面的模型再强，也可能拿不到正确资料。  
这说明检索阶段的问题，无法完全靠生成阶段补回来。

### 5. 常见误区

- 误区一：Embedding 是缩小版聊天模型  
  它不是，它只负责向量表示。

- 误区二：只要有 Embedding 就一定能检索得准  
  还要看 chunk 质量、数据量、召回策略和后续排序。

## 《AI大模型RAG项目》第 8 小节：向量数据库的原理与选型

### 1. 为什么需要向量数据库

当知识条目规模变大时，不能每次都把所有向量逐条和 query 做全量比较。  
向量数据库就是为“大规模相似度召回”优化的基础设施。

### 2. 它在 RAG 中的职责

向量数据库不负责回答，它负责：

- 存向量
- 存 chunk
- 存 metadata
- 快速召回候选

### 3. 它和传统数据库的差异

传统数据库更擅长：

- 精确匹配
- 结构化条件查询
- 事务和关系处理

向量数据库更擅长：

- 相似度搜索
- 近邻检索
- 高维向量召回

### 4. 核心认知

> 向量数据库不是“更高级的数据库”，而是为相似度检索这一类问题专门优化的数据库。

### 5. 常见误区

- 误区一：有了向量库就不需要普通数据库  
  实际上很多系统仍然需要两者配合。

- 误区二：向量数据库会自动提高回答质量  
  它只解决“召回”，不直接解决“表达”和“真实性”。

## 《AI大模型RAG项目》第 9 小节：向量检索策略与召回优化

### 1. Dense 检索

Dense 检索依赖语义向量相似度。

适合：

- 同义改写
- 口语表达
- 自然语言问题

不够强的地方：

- 对订单号、编号、术语、精确条款不一定最敏感

### 2. Sparse 检索

Sparse 检索更强调字面匹配，常见代表就是 BM25。

适合：

- 编号
- 规则条款
- 专有名词
- 精确关键词

不够强的地方：

- 对语义改写不够友好

### 3. Hybrid 检索

Hybrid 是把 Dense 和 Sparse 结合。

它的意义在于：

- Dense 帮你理解“意思像不像”
- Sparse 帮你确认“字面中没中”

所以 Hybrid 特别适合现实业务问题，因为业务问题通常同时包含：

- 语义意图
- 精确关键词

### 4. 召回优化的核心目标

召回阶段最重要的不是“排出最终答案”，而是：

> 尽量把真正相关的资料先找回来，并且别漏掉。

### 5. 常见误区

- 误区一：召回分数最高的就是最好答案  
  召回分数只能说明“像不像”，不一定说明“最适合回答”。

- 误区二：只做 Dense 就够了  
  对很多带编号和术语的问题，Sparse 价值很大。

## 《AI大模型RAG项目》第 10 小节：RRF 与两阶段检索

### 1. 为什么需要融合

如果系统同时走 Dense 和 Sparse，两边都会给出排序结果。  
此时需要一个方法把多路结果融合起来。

### 2. RRF 的直觉

RRF 不执着于比较原始分数，而是更关注结果在各自检索器中的排名。

它背后的思想是：

- 如果一条结果在多个检索路径里都排得靠前，那它更值得信任

### 3. 为什么这很实用

因为不同检索器的分数体系常常不统一。  
相比直接硬比数值，基于排名的位置融合，通常更稳。

### 4. 两阶段检索的思想

第一阶段：

- 尽量别漏召回

第二阶段：

- 在候选里进一步排准

这说明一个很重要的系统设计思想：

> 检索并不追求一步到位，而是允许“先广后精”。

### 5. 常见误区

- 误区一：第一阶段必须就把顺序排到完美  
  实际上第一阶段更重要的是召回完整性。

- 误区二：融合就是简单平均  
  融合方法有讲究，不同方法对结果稳定性影响很大。

## 《AI大模型RAG项目》第 11 小节：重排序 Rerank 的作用

### 1. Rerank 到底在做什么

Rerank 不是从全库里找结果，而是在“已经召回的一小批候选”中做更细的判断。

因此它的职责是：

- 过滤误召回
- 拉开候选间差距
- 找出最贴题的几条

### 2. 为什么它重要

粗检索的结果常常会出现这种情况：

- 都相关
- 但相关程度不同
- 有些适合作为背景
- 有些才是真正直接回答问题的依据

Rerank 的作用就是进一步分辨这些细微差别。

### 3. 核心认知

> 检索负责“把相关的找回来”，Rerank 负责“把最适合回答的排上去”。

### 4. 常见误区

- 误区一：Rerank 是可有可无的小优化  
  当候选变多、相似条目变多时，它会明显影响最终答案质量。

- 误区二：有了 Rerank 就可以忽略召回质量  
  如果真正相关的内容在第一阶段就没被召回，Rerank 也无能为力。

## 《AI大模型RAG项目》第 12 小节：大模型生成策略、幻觉抑制与引用对齐

### 1. 生成阶段的真正目标

生成阶段不是把资料“复述一遍”，而是把资料变成：

- 简洁的答案
- 受控的答案
- 带证据的答案

### 2. 幻觉抑制

幻觉不是模型故意乱说，而是模型在信息不足或约束不够时进行“合理补全”。

抑制幻觉的基本方法包括：

- 限制只能基于参考资料回答
- 资料不足时明确拒答
- 禁止补充未出现的具体细节

### 3. 引用对齐

引用对齐的关键不在于“生成一个 `[1]`”，而在于：

- 让模型按规则引用
- 把引用编号映射回真实资料
- 让用户能看到证据链

这意味着：

> 引用不是装饰，而是答案可信度的一部分。

### 4. 答案约束

RAG 回答质量高，不只是因为“说得通顺”，更因为它：

- 不乱扩展
- 不随意发散
- 不把无关资料也塞进来

### 5. 常见误区

- 误区一：只要引用了编号，答案就一定可靠  
  还要看引用是否真的对应正确资料。

- 误区二：回答越丰富越好  
  在 RAG 里，过度扩展常常意味着约束不足。

## 《AI大模型RAG项目》第 13 小节：理解 Function Call

### 1. Function Call 的本质

Function Call 的意义不是“让模型会写代码”，而是让模型学会在回答前按需调用外部能力。

它体现的是一种能力升级：

- 从“只依赖一次性输入”
- 到“可以边看边取证再回答”

### 2. 为什么 Function Call 对 RAG 有价值

因为很多问题不是“看一眼目录摘要就能放心回答”的。

例如：

- 订单号
- 时间点
- 物流状态
- 条款细节
- 金额条件

这些都属于“需要核验”的信息。

### 3. 它和普通 RAG 的区别

普通 RAG 更像：

- 系统先把资料准备好
- 模型直接基于现成上下文回答

Function Call 增强后的 RAG 更像：

- 模型先看到资料目录
- 如果觉得细节还不够，就主动再取原文
- 再基于核验结果回答

### 4. 核心认知

> Function Call 让模型从“被动读取上下文”变成“主动取证后再作答”。

### 5. 常见误区

- 误区一：Function Call 只适合很复杂的外部系统  
  实际上哪怕只是“按编号读取一条资料原文”，也已经很有价值。

- 误区二：只要加了工具，模型就一定会调用  
  是否调用还取决于 Prompt 约束、上下文设计和问题类型。

## 《AI大模型RAG项目》第 14 小节：这些知识点之间是什么关系

到目前为止，这些知识点不是零散存在的，而是可以串成一条清晰链路：

```text
文档解析
-> 文本切块
-> 元数据整理
-> Embedding 向量化
-> 向量数据库存储
-> Dense / Sparse / Hybrid 检索
-> RRF / Rerank 排序优化
-> Prompt 约束生成
-> 引用对齐
-> Function Call 核验细节
```

如果从“每层解决什么问题”的角度看，可以这样理解：

- 文档解析：把文件变成文本
- Chunk：把文本变成检索单元
- 元数据：让资料可解释、可管理
- Embedding：把语义变成可计算表示
- 向量库：让相似检索可高效运行
- Dense / Sparse / Hybrid：决定怎么召回
- RRF / Rerank：决定怎么把结果排得更稳更准
- Prompt：决定模型如何被约束
- 引用对齐：决定答案如何可追溯
- Function Call：决定模型能否进一步核验细节

### 一个很重要的阶段性理解

RAG 的效果从来不是某一层单独决定的，而是整条链路共同决定的。

也就是说：

- 检索差，生成再强也会被带偏
- 资料对，Prompt 弱，仍然会发散
- 引用有，但对齐错，还是不可信
- 工具有，但不触发，也起不到核验作用

## 《AI大模型RAG项目》第 16 小节：MCP 协议入门与实践

MCP（Model Context Protocol）可以先把它理解成一套“让模型与外部能力标准化对接”的协议。

如果只从表面看，很容易把 MCP 误以为是“给大模型加工具调用”。但真正更重要的是：它不是某个项目里临时写出来的一组接口，而是在协议层统一规定了模型如何发现能力、如何读取上下文、如何触发执行。

也就是说，MCP 解决的并不是“能不能调某个方法”，而是“模型如何用统一方式接入外部世界”。

### 1. MCP 解决的核心问题是什么

在没有 MCP 之前，不同客户端、不同 Agent 框架、不同业务系统，往往都会各自定义一套工具接入方式。

这样会带来几个问题：

- 模型侧接入方式不统一
- 工具描述方式不统一
- 上下文读取方式不统一
- 一个系统里写好的能力，迁移到另一个系统时复用成本很高

MCP 的价值就在于：

- 用统一协议描述能力
- 用统一结构暴露给客户端
- 让客户端和服务端按同一套规则协作

所以它更像“模型能力接入标准”，而不只是“工具调用功能”。

### 2. MCP 不只有 Tool

初学时最容易把 MCP 只理解成 Tool。

其实 MCP 里至少要区分三类核心能力：

- Tool：可执行能力，强调“做一件事”
- Resource：可读取资源，强调“拿到一段内容”
- Prompt：可复用提示模板，强调“组织一套提示词结构”

这三个东西分别对应三种完全不同的职责：

- Tool 面向动作执行
- Resource 面向上下文供给
- Prompt 面向提示复用

如果只盯着 Tool，就会把 MCP 理解窄了。

### 3. 为什么 MCP 适合接入 RAG 体系

RAG 本质上就不是单纯的一问一答，而是一条“检索、组织上下文、生成答案、必要时再调用外部能力”的链路。

因此 MCP 很适合和 RAG 结合，原因在于：

- Tool 可以补充外部动作能力
- Resource 可以补充结构化上下文来源
- Prompt 可以把问答规则模板化

这样一来，RAG 不再只是“检索后拼 Prompt”，而是可以进一步变成“协议化地管理知识、上下文与能力”。

### 4. 核心认识

MCP 的重点不是“又多了一个接口层”，而是把模型与外部能力之间的协作方式标准化了。

记住这一点很关键：

- Tool 是能力暴露
- Resource 是内容暴露
- Prompt 是规则暴露
- MCP 是统一这些暴露方式的协议

### 5. 常见误区

- 误区一：把 MCP 等同于 Function Call
  Function Call 更像模型生成结构化调用意图的一种能力，而 MCP 是客户端与服务端之间的协议化协作方式。

- 误区二：把 MCP 只看成 Tool 容器
  实际上 MCP 还包含 Resource 和 Prompt，三者不能混在一起理解。

- 误区三：以为 MCP 只是某个客户端专属功能
  它本质上是协议层思路，不是某个具体客户端私有的业务写法。

## 《AI大模型RAG项目》第 17 小节：深入解析 MCP 中的 Resources 与 Prompts

很多人在刚接触 MCP 时，最容易理解的是 Tool，因为 Tool 很直观，就是“执行一个动作”。

但真正一旦往下做，就会发现 Resource 和 Prompt 才是 MCP 中特别值得单独理解的两部分，因为它们决定了“上下文从哪来”和“规则怎么复用”。

### 1. 什么是 Resource

Resource 可以理解成“可被客户端读取的资源内容”。

它强调的不是执行，而是提供内容。

所以 Resource 更像：

- 一份文档
- 一段结构化数据
- 一个可按 URI 访问的上下文片段
- 一个带参数的资源模板

它解决的问题是：

“当模型需要参考某些资料时，这些资料如何以标准化方式暴露出来？”

因此，Resource 的本质是内容供给，不是动作执行。

### 2. 什么是 Prompt

Prompt 则可以理解成“可复用的提示模板”。

它不是最终答案，也不是外部动作，而是把某一类任务所需的角色设定、约束规则、输出结构提前固化下来。

它解决的是：

“面对一类重复出现的问题，如何用统一模板组织提示词？”

所以 Prompt 更偏向于：

- 角色约束
- 回答规则
- 输出格式
- 参数化拼装

它的核心价值是复用和规范，而不是执行。

### 3. Tool、Resource、Prompt 三者怎么区分

可以用一句非常实用的话来区分：

- 要“做事”，用 Tool
- 要“拿内容”，用 Resource
- 要“定规则”，用 Prompt

如果继续细分：

- Tool 关注副作用和执行结果
- Resource 关注内容读取和上下文补充
- Prompt 关注提示组织和任务模板化

一旦这个边界清楚了，后面设计 MCP Server 时思路就会清晰很多。

### 4. 核心认识

Resource 和 Prompt 的价值，在于把“上下文供给”和“提示规则”从具体业务代码里抽离出来。

这样做的好处是：

- 上下文来源更清晰
- 提示模板更可复用
- 客户端看到的能力边界更明确
- 整体系统更容易维护

### 5. 常见误区

- 误区一：把 Resource 当成不会写入的 Tool
  它们的职责不同，Resource 核心是读内容，不是执行动作。

- 误区二：把 Prompt 当成普通字符串配置
  Prompt 在 MCP 里强调的是“模板能力”，不是简单文本片段。

- 误区三：什么都想做成 Tool
  如果本质只是提供资料或模板，硬做成 Tool 会让能力边界越来越乱。

## 《AI大模型RAG项目》第 18 小节：MCP 之官方 Java-SDK 深度解析

从代码层面看，MCP Server 的注册过程，最核心的不是“用了什么写法”，而是“客户端能看到什么定义、服务端最终执行哪个处理逻辑”。

也就是说，不管是 Tool、Resource 还是 Prompt，它们本质上都离不开两部分：

- 一部分是给客户端看的定义信息
- 一部分是真正被执行的处理逻辑

### 1. 注册的本质是什么

所谓“注册进去”，本质上就是在服务端建立一层映射关系：

- 客户端看到一个能力定义
- 当客户端触发这个能力时
- 服务端能准确找到对应的处理方法

所以注册不是“代码写在那就自动生效”，而是要让框架知道：

- 这个能力叫什么
- 它接收什么参数
- 它返回什么内容
- 真正由哪个处理器执行

### 2. 为什么不用注解也能注册

很多人一开始会觉得，不写注解就像“没注册”。

其实不是。

注解只是一种更方便的声明方式，它本质上是让框架帮你自动提取定义，再自动建立映射关系。

而手动注册的方式，则是你自己明确告诉框架：

- 我要暴露这个能力
- 它的定义是什么
- 它对应的执行逻辑是什么

所以两种方式的本质没有变，变的只是“谁来写这层映射”。

### 3. 注解注册与手动注册的差别

可以把两者对比着理解：

注解注册：

- 偏声明式
- 写起来更简洁
- 由框架扫描并生成注册信息
- 适合快速暴露标准能力

手动注册：

- 偏编程式
- 写法更明确
- 由开发者自己构造定义和处理器
- 适合版本差异较大或需要更细粒度控制的场景

所以“有没有注解”不是关键，关键是服务端最终有没有把定义和处理逻辑正确绑定起来。

### 4. 核心认识

理解 SDK 时最重要的一点是：

不管表面 API 怎么变，底层思想都没变，就是“定义”和“处理器”的绑定。

这套理解可以同时解释：

- Tool 为什么能被调用
- Resource 为什么能被读取
- Prompt 为什么能被客户端发现

因为三者都离不开“能力定义 + 执行入口”这个结构。

### 5. 常见误区

- 误区一：觉得只有加了注解才算注册
  实际上注解只是自动化注册手段，不是注册本质。

- 误区二：把 SDK API 名字当成核心
  真正要理解的是注册模型，而不是某个版本的具体类名。

- 误区三：只关注客户端能看到什么，不关注服务端真正执行什么
  如果只看定义不看处理逻辑，很容易在调试时找不到问题根源。

## 《AI大模型RAG项目》第 19 小节：MCP 客户端调用链与执行边界

当我们在客户端里输入一句“请调用知识库工具，回答某个问题”时，背后并不是 MCP Server 自己在“主动聊天”，而是一条分层很明确的调用链。

把这条链路看清楚，很多“为什么会这样返回”的问题就容易想明白了。

### 1. 一条完整调用链是怎么走的

可以把流程拆成下面几步：

1. 用户在客户端发起问题
2. 客户端里的模型先理解问题
3. 客户端模型判断是否要调用 MCP 能力
4. 如果决定调用，就向 MCP Server 发起能力请求
5. MCP Server 执行对应 Tool / Resource / Prompt 的处理逻辑
6. 如果 Tool 内部还接了检索、RAG、函数调用或其他模型，它会继续往下执行
7. MCP Server 把原始执行结果返回给客户端
8. 客户端模型再把结果组织成最终展示给用户的话术

所以用户看到的最终回答，往往已经不是 MCP Server 的“原始返回”了，而是客户端模型加工后的结果。

### 2. 为什么会出现“下面像总结过一层”的感觉

这是因为客户端模型通常会把工具返回结果重新整理成更自然的回答。

于是就会出现两层结果：

- 一层是 MCP Server 返回的原始结构化结果
- 一层是客户端模型面向用户生成的自然语言结果

如果客户端把原始 JSON 也展示出来，你就能明显看到这两层。

### 3. MCP 内部还能再调用模型吗

可以，而且这是理解执行边界时非常重要的一点。

客户端里的模型，和 MCP Server 内部真正执行业务逻辑时依赖的模型，不一定是同一个模型。

也就是说：

- 客户端模型负责“理解用户、决定是否调用、组织最终回复”
- MCP 内部模型负责“完成 Server 内部那段业务链路”

所以即使用户在客户端里看到的是一个模型名称，也不代表整个链路只用了这一种模型。

### 4. 核心认识

调试 MCP 问题时，一定要把边界拆开看：

- 是客户端模型没决定调用
- 还是 MCP Server 根本没启动成功
- 还是工具本身执行失败
- 还是工具内部依赖的检索、模型、配置出了问题
- 还是客户端对原始结果又做了一层总结

一旦分层清楚，问题定位就会快很多。

### 5. 常见误区

- 误区一：把客户端最终看到的话，全部当成 MCP Server 原样返回
  实际上很多时候中间还有客户端模型的重组。

- 误区二：以为客户端模型和 Server 内部模型必然相同
  两者完全可以不同，职责也不同。

- 误区三：只盯着聊天界面，不去区分协议层返回和展示层包装
  这样很容易误判问题到底出在哪一层。

## 《AI大模型RAG项目》第 20 小节：STDIO 型 MCP Server 的运行方式与排错思路

很多桌面客户端接入 MCP Server 时，常见方式是通过 STDIO 启动一个本地进程，然后靠标准输入输出和它通信。

这套方式看起来简单，但调试时特别容易混淆“进程启动问题”“协议通信问题”“业务执行问题”这三层边界。

### 1. 什么叫 STDIO 型 MCP Server

STDIO 的意思就是：

- 客户端启动一个本地进程
- 通过标准输入把请求发给它
- 通过标准输出接收它返回的协议消息

也就是说，这个 Server 往往不是先启动一个 Web 服务等别人来调，而是被客户端作为子进程直接拉起。

### 2. 为什么日志不能乱打到标准输出

因为在 STDIO 模式下，标准输出本身就是协议通道。

如果把普通日志、调试打印、启动信息一股脑打到 stdout，就可能污染协议消息，导致客户端解析失败。

所以一个很重要的经验就是：

- 协议消息走标准输出
- 普通日志尽量写文件，或者走标准错误输出

这也是为什么很多 MCP 配置里会专门把日志输出方式单独处理。

### 3. 常见问题应该怎么分层排查

遇到问题时，不要一下子把所有原因混成一团，而是按层排查：

第一层：客户端能不能成功拉起进程

要关注：

- `java` 路径对不对
- `JAVA_HOME` 是否正确
- jar 路径是否正确
- Windows 下 JSON 里的路径转义是否正确

第二层：Server 启动后，能力有没有注册成功

要关注：

- Tool / Resource / Prompt 是否真的暴露出来
- 注册逻辑是否生效
- 版本与依赖是否匹配

第三层：能力被调用后，内部业务链路是否能执行成功

要关注：

- API Key 是否存在
- 内部模型是否可用
- 检索与数据库配置是否正常
- RAG 流程本身是否报错

### 4. 核心认识

STDIO 模式下最重要的，不是“它有没有报错”，而是先判断错误落在哪一层：

- 启动层
- 协议层
- 能力注册层
- 业务执行层

只有先把层次拆开，排查才不会越查越乱。

### 5. 常见误区

- 误区一：客户端连不上，就立刻怀疑工具代码写错了
  其实很多时候连进程都没启动起来。

- 误区二：看到 Server 启动了，就认为工具一定能正常工作
  启动成功不等于注册成功，也不等于内部依赖都可用。

- 误区三：把普通日志打印到 stdout
  这在 STDIO 模式下很容易直接影响协议通信。
