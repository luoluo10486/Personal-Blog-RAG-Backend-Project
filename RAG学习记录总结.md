## 多轮对话 RAG 完整链路总览

结合前面的知识点，可以先把多轮对话 RAG 的完整链路建立成一个总图。

它不是“用户提问后直接去检索再生成”这么简单，而是一条层次很清楚的流水线：

1. 用户发起当前问题
2. 会话记忆模块读取历史上下文
3. Query 改写模块结合当前问题与历史上下文，生成更适合检索的查询
4. Retriever 使用改写后的 Query 去向量库 / 混合检索系统做召回
5. 从大规模文档中先拿到候选结果
6. Reranker 对候选结果再做精排
7. 选出最相关的少量片段作为上下文
8. LLM 结合系统提示词、对话历史、检索片段和用户原始问题生成答案
9. 将本轮用户消息和助手回复再写回会话记忆

### 1. 这条链路里每一层分别负责什么

会话记忆：

负责保留多轮对话历史，让系统知道当前这句话是在什么上下文中说出来的。

Query 改写：

负责把“用户原始问题 + 对话历史”整理成更适合检索的 Query。它服务的是检索，不是替代用户原问题。

Retriever：

负责大范围召回候选资料。它追求的是先把可能相关的内容找出来。

Reranker：

负责在召回结果里重新排序，把真正最相关的片段顶上来。

LLM：

负责基于最终上下文生成答案。它回答时仍然应该围绕用户原始问题，而不是围绕改写后的 Query 自说自话。

### 2. 为什么这条链路的顺序不能乱

这里最容易出错的是 Query 改写的位置。

正确顺序一定是：

- 先读会话记忆
- 再做 Query 改写
- 再发起检索
- 再做重排
- 最后生成答案

因为：

- 不先读历史，就没法做指代消解和上下文补全
- 不先改写就去检索，多轮问题很可能检索失败
- 不先重排，直接把大量候选都塞给模型，噪声会很大
- 不保留原始问题，生成阶段就可能偏离用户真正想问的内容

### 3. 这条链路最关键的一句总结

完整链路可以压缩成一句话：

> 提问 → 记忆 → 改写 → 检索（大库召回）→ 精排（候选收缩）→ 生成 → 写回记忆

而其中最关键的设计原则是：

> 检索使用改写后的 Query，生成围绕用户原始问题。

### 4. 为什么这是多轮 RAG 的核心骨架

因为多轮 RAG 的难点从来不只是“有没有知识库”，而是：

- 能不能理解当前问题和历史问题之间的关系
- 能不能把多轮表达转换成适合检索的查询
- 能不能从大规模文档中先召回、再精排
- 能不能在最终生成时仍然紧扣用户原始问题

所以这条链路，其实就是把“记忆、改写、检索、重排、生成”五层能力串起来的骨架。

## 《AI大模型RAG项目》第 1 小节：调用大模型 API

### 1. 什么是大模型 API

大模型 API 可以理解为“远程推理服务入口”。  
应用程序本身不负责生成答案，而是把请求组织好，再把推理工作交给模型服务。

它和普通后端接口的相似点是：

- 都是一次标准请求
- 都有入参和返回值
- 都要考虑超时、鉴权、重试和异常处理

它和普通接口最不一样的地方在于：

- 返回内容并不是固定结构结果，而是模型生成结果
- 输出天然带有不确定性
- token 消耗和时延是重要成本

### 2. 为什么要先理解 API，再理解 RAG

因为 RAG 最终仍然要落回一次模型调用。  
无论前面做了多少检索、重排、工具调用，最后“把资料组织成答案”这一步，仍然需要 Chat API 来完成。

所以可以把整个系统拆成两层：

- 上游：负责找到资料
- 下游：负责基于资料生成回答

### 3. 当前阶段最重要的认知

- Chat 模型负责回答
- Embedding 模型负责向量化
- Rerank 模型负责候选排序

这三类能力都可能来自“大模型服务”，但它们的职责完全不同。

### 4. 常见误区

- 误区一：只要能调通 Chat API，就等于已经做了 RAG  
  实际上这只完成了“生成”，还没有完成“检索增强”。

- 误区二：Embedding 也是回答模型  
  实际上 Embedding 不负责回答，只负责表示语义。

- 误区三：API 调用只是工程细节  
  实际上 API 的超时、成本、稳定性会直接影响整条 RAG 链路能不能真正可用。

## 《AI大模型RAG项目》第 2 小节：Prompt 工程入门

### 1. Prompt 的本质

Prompt 不是“问模型一句话”，而是对模型行为施加控制。

一个完整 Prompt 往往至少包含：

- 角色设定：你是谁
- 能力边界：你回答什么，不回答什么
- 行为规则：你必须遵守什么
- 输出格式：你最后怎么表达

### 2. 为什么 Prompt 在 RAG 中格外重要

在普通聊天里，Prompt 主要影响“风格和倾向”；  
在 RAG 里，Prompt 更重要的是影响“可信度和可控性”。

它决定模型是否会：

- 只依据资料回答
- 在资料不足时明确拒答
- 输出引用编号
- 避免补全资料中没有的细节

### 3. Prompt 的常见作用层次

可以把 Prompt 理解成三层：

1. 角色层  
   定义模型站在什么身份回答。

2. 规则层  
   规定模型能不能扩写、能不能猜测、资料不足怎么办。

3. 输出层  
   规定回答格式、引用格式、语言风格。

### 4. 一个重要判断

如果一个 RAG 系统已经有检索结果，但回答仍然经常发散、胡编、乱引用，问题往往不只在模型本身，也很可能在 Prompt 设计不够严格。

### 5. 常见误区

- 误区一：Prompt 只影响语言风格  
  实际上 Prompt 也影响真实性、边界感和引用行为。

- 误区二：Prompt 越长越好  
  过长的 Prompt 会增加成本，也可能让关键约束被淹没。真正重要的是结构清晰、规则明确。

- 误区三：只要写了“不要胡编”，模型就一定不会胡编  
  幻觉抑制要靠 Prompt、上下文组织、检索质量共同作用。

## 《AI大模型RAG项目》第 3 小节：什么是 RAG

### 1. RAG 的定义

RAG 是 Retrieval-Augmented Generation，意思是“检索增强生成”。

它不是一个单点技术，而是一种组合式系统设计：

- 用检索解决“模型不知道私有资料”的问题
- 用生成解决“资料不会自己组织成答案”的问题

### 2. RAG 在解决什么问题

纯大模型的强项：

- 语言组织能力强
- 表达自然
- 推理能力较好

纯大模型的短板：

- 不知道你的内部资料
- 容易过时
- 容易在陌生知识上补全

纯检索系统的强项：

- 能找到资料
- 可追溯
- 相对稳定

纯检索系统的短板：

- 不会把结果组织成自然回答
- 用户需要自己读资料

RAG 的价值就在于把这两者接起来。

### 3. 标准 RAG 的最小链路

```text
用户问题
-> 检索相关资料
-> 组装参考上下文
-> 调用模型生成答案
-> 返回答案与证据
```

### 4. 核心认知

> RAG 不是让模型凭空更聪明，而是让模型在回答前先接触到更相关的资料。

### 5. 常见误区

- 误区一：RAG 的核心是“模型更强”  
  实际上 RAG 的核心是“系统把相关知识在正确时机送到模型面前”。

- 误区二：只要接了向量库就是 RAG  
  没有生成环节、没有上下文组装、没有回答约束时，最多只是“检索系统”。

- 误区三：RAG 一定比纯模型更好  
  如果检索质量差、上下文混乱，RAG 反而可能把模型带偏。

## 《AI大模型RAG项目》第 4 小节：用 Apache Tika 解析文档

### 1. 文档解析为什么是第一步

知识库的原始输入通常是文件，而检索和生成依赖的是文本。  
所以必须先完成一次“文件 -> 文本”的转换。

### 2. Tika 的角色

Tika 的作用不是回答问题，而是把不同格式文档里的内容抽出来，变成后续步骤能继续处理的文本和元数据。

### 3. 这一层真正解决的问题

```text
如何把 PDF / Word / 文本文件变成可继续切块和入库的原始材料
```

### 4. 为什么这一步不能跳过

- 向量检索检索的是文本表示，不是原始文件
- chunk 切分依赖正文内容
- metadata 提取依赖解析结果

### 5. 常见误区

- 误区一：文档上传就等于知识库构建完成  
  解析只是起点，后面还要切块、向量化、入库。

- 误区二：解析出正文就够了  
  metadata 同样重要，它关系到后续的来源展示和管理。

## 《AI大模型RAG项目》第 5 小节：数据分块 Chunk 策略与实践

### 1. 为什么要切 Chunk

整篇文档直接进入检索系统，会带来三个问题：

- 太长，噪声太多
- 太杂，主题不集中
- 太大，浪费上下文窗口

所以需要把文本切成适合检索和生成的基本单位。

### 2. 好的 Chunk 不是越小越好

Chunk 太大：

- 检索命中后包含无关信息太多
- 模型阅读成本高

Chunk 太小：

- 语义容易断裂
- 单条信息不足以回答问题

因此 chunk 的目标是：

> 在“足够短”和“语义完整”之间找到平衡。

### 3. 为什么结构化切块更优

如果切块只按固定长度截断，很容易把：

- 标题和正文分开
- 列表切断
- 表格拆碎
- 一段规则切成两半

而结构化切块会尽量尊重文档天然边界，比如：

- 标题
- 段落
- 列表
- 表格
- 代码块

### 4. Overlap 的意义

相邻 chunk 之间保留少量重叠，不是为了重复，而是为了减少语义断裂。

它解决的问题是：

- 一段内容刚好跨块
- 后半段需要前半段上下文才能理解

### 5. 常见误区

- 误区一：Chunk 就是把字数均分  
  真正重要的是语义边界，而不是机械长度。

- 误区二：Overlap 越大越好  
  重叠过多会导致冗余召回和上下文浪费。

- 误区三：切块是后续再优化的小问题  
  实际上 chunk 策略直接影响检索质量，是 RAG 的基础能力之一。

## 《AI大模型RAG项目》第 6 小节：元数据的作用与管理

### 1. 什么是元数据

元数据可以理解为“描述数据的数据”。  
正文负责承载事实，元数据负责说明这条事实是什么、来自哪里、属于哪类。

### 2. 元数据为什么重要

即使两段正文内容都能回答问题，系统仍然需要元数据来完成：

- 来源展示
- 引用对齐
- 分类过滤
- 文档管理

### 3. 元数据在 RAG 里的三种价值

1. 检索价值  
   用于分类过滤、范围收缩、后续扩展条件查询。

2. 展示价值  
   用于告诉用户“答案来自哪份资料”。

3. 管理价值  
   用于追踪知识来源、版本和归属。

### 4. 核心认知

> 没有元数据，系统可能还能回答；但没有元数据，系统通常就不够可解释、可追溯、可维护。

### 5. 常见误区

- 误区一：只要正文够强，元数据可以不要  
  这会导致系统失去证据链和管理能力。

- 误区二：元数据只是前端展示字段  
  实际上它也会反过来影响检索和过滤能力。

## 《AI大模型RAG项目》第 7 小节：从文本到向量之理解 Embedding

### 1. Embedding 的本质

Embedding 是把文本映射到一个高维向量空间。  
在这个空间里，语义相近的文本通常会更接近。

### 2. 为什么 Embedding 对 RAG 重要

因为用户问的问题，和文档里写的原话，往往不是一模一样的。

例如：

- 用户说“七天内还能退吗”
- 文档可能写“签收后 7 天内支持无理由退货”

如果只做字面匹配，可能命中不稳；  
Embedding 的价值就是帮助系统识别“语义相近”。

### 3. Embedding 解决的不是回答，而是召回

它回答的是这个问题：

```text
哪些资料和当前问题在语义上更接近
```

而不是：

```text
最终答案怎么说
```

### 4. 一个重要判断

如果 Embedding 质量不佳，即使后面的模型再强，也可能拿不到正确资料。  
这说明检索阶段的问题，无法完全靠生成阶段补回来。

### 5. 常见误区

- 误区一：Embedding 是缩小版聊天模型  
  它不是，它只负责向量表示。

- 误区二：只要有 Embedding 就一定能检索得准  
  还要看 chunk 质量、数据量、召回策略和后续排序。

## 《AI大模型RAG项目》第 8 小节：向量数据库的原理与选型

### 1. 为什么需要向量数据库

当知识条目规模变大时，不能每次都把所有向量逐条和 query 做全量比较。  
向量数据库就是为“大规模相似度召回”优化的基础设施。

### 2. 它在 RAG 中的职责

向量数据库不负责回答，它负责：

- 存向量
- 存 chunk
- 存 metadata
- 快速召回候选

### 3. 它和传统数据库的差异

传统数据库更擅长：

- 精确匹配
- 结构化条件查询
- 事务和关系处理

向量数据库更擅长：

- 相似度搜索
- 近邻检索
- 高维向量召回

### 4. 核心认知

> 向量数据库不是“更高级的数据库”，而是为相似度检索这一类问题专门优化的数据库。

### 5. 常见误区

- 误区一：有了向量库就不需要普通数据库  
  实际上很多系统仍然需要两者配合。

- 误区二：向量数据库会自动提高回答质量  
  它只解决“召回”，不直接解决“表达”和“真实性”。

## 《AI大模型RAG项目》第 9 小节：向量检索策略与召回优化

### 1. Dense 检索

Dense 检索依赖语义向量相似度。

适合：

- 同义改写
- 口语表达
- 自然语言问题

不够强的地方：

- 对订单号、编号、术语、精确条款不一定最敏感

### 2. Sparse 检索

Sparse 检索更强调字面匹配，常见代表就是 BM25。

适合：

- 编号
- 规则条款
- 专有名词
- 精确关键词

不够强的地方：

- 对语义改写不够友好

### 3. Hybrid 检索

Hybrid 是把 Dense 和 Sparse 结合。

它的意义在于：

- Dense 帮你理解“意思像不像”
- Sparse 帮你确认“字面中没中”

所以 Hybrid 特别适合现实业务问题，因为业务问题通常同时包含：

- 语义意图
- 精确关键词

### 4. 召回优化的核心目标

召回阶段最重要的不是“排出最终答案”，而是：

> 尽量把真正相关的资料先找回来，并且别漏掉。

### 5. 常见误区

- 误区一：召回分数最高的就是最好答案  
  召回分数只能说明“像不像”，不一定说明“最适合回答”。

- 误区二：只做 Dense 就够了  
  对很多带编号和术语的问题，Sparse 价值很大。

## 《AI大模型RAG项目》第 10 小节：RRF 与两阶段检索

### 1. 为什么需要融合

如果系统同时走 Dense 和 Sparse，两边都会给出排序结果。  
此时需要一个方法把多路结果融合起来。

### 2. RRF 的直觉

RRF 不执着于比较原始分数，而是更关注结果在各自检索器中的排名。

它背后的思想是：

- 如果一条结果在多个检索路径里都排得靠前，那它更值得信任

### 3. 为什么这很实用

因为不同检索器的分数体系常常不统一。  
相比直接硬比数值，基于排名的位置融合，通常更稳。

### 4. 两阶段检索的思想

第一阶段：

- 尽量别漏召回

第二阶段：

- 在候选里进一步排准

这说明一个很重要的系统设计思想：

> 检索并不追求一步到位，而是允许“先广后精”。

### 5. 常见误区

- 误区一：第一阶段必须就把顺序排到完美  
  实际上第一阶段更重要的是召回完整性。

- 误区二：融合就是简单平均  
  融合方法有讲究，不同方法对结果稳定性影响很大。

## 《AI大模型RAG项目》第 11 小节：重排序 Rerank 的作用

### 1. Rerank 到底在做什么

Rerank 不是从全库里找结果，而是在“已经召回的一小批候选”中做更细的判断。

因此它的职责是：

- 过滤误召回
- 拉开候选间差距
- 找出最贴题的几条

### 2. 为什么它重要

粗检索的结果常常会出现这种情况：

- 都相关
- 但相关程度不同
- 有些适合作为背景
- 有些才是真正直接回答问题的依据

Rerank 的作用就是进一步分辨这些细微差别。

### 3. 核心认知

> 检索负责“把相关的找回来”，Rerank 负责“把最适合回答的排上去”。

### 4. 常见误区

- 误区一：Rerank 是可有可无的小优化  
  当候选变多、相似条目变多时，它会明显影响最终答案质量。

- 误区二：有了 Rerank 就可以忽略召回质量  
  如果真正相关的内容在第一阶段就没被召回，Rerank 也无能为力。

## 《AI大模型RAG项目》第 12 小节：大模型生成策略、幻觉抑制与引用对齐

### 1. 生成阶段的真正目标

生成阶段不是把资料“复述一遍”，而是把资料变成：

- 简洁的答案
- 受控的答案
- 带证据的答案

### 2. 幻觉抑制

幻觉不是模型故意乱说，而是模型在信息不足或约束不够时进行“合理补全”。

抑制幻觉的基本方法包括：

- 限制只能基于参考资料回答
- 资料不足时明确拒答
- 禁止补充未出现的具体细节

### 3. 引用对齐

引用对齐的关键不在于“生成一个 `[1]`”，而在于：

- 让模型按规则引用
- 把引用编号映射回真实资料
- 让用户能看到证据链

这意味着：

> 引用不是装饰，而是答案可信度的一部分。

### 4. 答案约束

RAG 回答质量高，不只是因为“说得通顺”，更因为它：

- 不乱扩展
- 不随意发散
- 不把无关资料也塞进来

### 5. 常见误区

- 误区一：只要引用了编号，答案就一定可靠  
  还要看引用是否真的对应正确资料。

- 误区二：回答越丰富越好  
  在 RAG 里，过度扩展常常意味着约束不足。

## 《AI大模型RAG项目》第 13 小节：理解 Function Call

### 1. Function Call 的本质

Function Call 的意义不是“让模型会写代码”，而是让模型学会在回答前按需调用外部能力。

它体现的是一种能力升级：

- 从“只依赖一次性输入”
- 到“可以边看边取证再回答”

### 2. 为什么 Function Call 对 RAG 有价值

因为很多问题不是“看一眼目录摘要就能放心回答”的。

例如：

- 订单号
- 时间点
- 物流状态
- 条款细节
- 金额条件

这些都属于“需要核验”的信息。

### 3. 它和普通 RAG 的区别

普通 RAG 更像：

- 系统先把资料准备好
- 模型直接基于现成上下文回答

Function Call 增强后的 RAG 更像：

- 模型先看到资料目录
- 如果觉得细节还不够，就主动再取原文
- 再基于核验结果回答

### 4. 核心认知

> Function Call 让模型从“被动读取上下文”变成“主动取证后再作答”。

### 5. 常见误区

- 误区一：Function Call 只适合很复杂的外部系统  
  实际上哪怕只是“按编号读取一条资料原文”，也已经很有价值。

- 误区二：只要加了工具，模型就一定会调用  
  是否调用还取决于 Prompt 约束、上下文设计和问题类型。

## 《AI大模型RAG项目》第 13.5 小节：工具调用的架构设计指南

工具调用是 RAG 系统从“只会查资料”走向“能干活”的关键一步。

Function Call 和 MCP 解决的是“模型怎么调用工具”，但工具本身定义得好不好，决定了整个系统的上限。

所以理解工具调用，不能只停留在“协议怎么发”，还要继续往下看：

- 工具应该怎么拆
- 参数应该怎么暴露
- 描述应该怎么写
- 返回值应该怎么规范

### 1. 设计阶段：先把工具定义对

一个好工具首先要解决的是“模型容易不会用错”。

设计时最重要的几条原则是：

- 单一职责：一个工具只做一件事，拆分通常比大而全更好
- 参数最小化：只暴露必要参数，能由后端补齐或推断的不要都甩给模型
- 幂等性：操作类工具最好带 `requestId` 之类的幂等标识，避免重复执行
- 返回值结构化：统一 JSON 结构，例如 `success + data + errorCode`

这里的核心思想是：

> 工具定义不是给人随便点着玩的，而是给模型自动决策和自动传参用的，所以越清晰、越收敛越好。

### 2. 描述阶段：让模型知道“什么时候该用它”

工具的 `name` 和参数定义只是最低要求，真正影响调用命中率的，往往是描述写得好不好。

描述阶段建议重点做好这几件事：

- 三要素齐全：写清功能说明、适用场景、参数说明
- 关键词优化：把用户可能的自然表达写进去，覆盖不同问法
- 突出差异：多工具并存时，用“优先用于”“仅当……时使用”拉开边界
- 避免歧义：参数描述尽量具体，并给出示例

也就是说，一个好描述不仅是在解释“这个工具能做什么”，更是在告诉模型：

- 什么情况下该调它
- 什么情况下不该调它
- 参数该怎么填

### 3. 返回值阶段：让模型能稳定消费结果

工具调用不是到“执行成功”就结束了，模型还要读懂返回值，再决定下一步怎么说、怎么继续调其他能力。

因此返回值设计也要讲规范：

- 信息密度适中：不要堆过多无关字段，也不要缺关键信息
- 错误码语义化：例如 `INVALID_PARAMETER`、`PERMISSION_DENIED`、`SYSTEM_ERROR`
- 错误信息友好：告诉模型应该怎么修正，而不是只返回一句“错了”

好的返回值应该支持模型完成三件事：

- 正确理解结果
- 正确处理异常
- 正确生成给用户的后续回复

### 4. 工具定义再好，也不代表线上一定稳

工具调用的协议层和定义层解决的是“能不能正确调用”，但生产环境里真正让系统稳定运行，还要继续补这些能力：

- 网络超时处理
- 第三方服务失败兜底
- 非法参数校验
- 权限控制
- 安全防护，例如注入、越权、恶意输入
- 测试验证与监控告警

所以可以把工具调用架构理解成三层：

- 协议层：Function Call / MCP 解决“怎么调”
- 定义层：工具设计解决“调什么、怎么传、怎么回”
- 运行层：异常处理、安全、测试、监控解决“线上能不能稳”

### 5. 一个很重要的判断标准

判断一个工具定义得好不好，不是看它“功能多不多”，而是看它是否同时满足：

- 模型容易选对
- 参数容易传对
- 返回值容易读懂
- 错误容易恢复

如果一个工具虽然功能强，但描述模糊、参数很多、返回混乱，那它对模型来说往往并不好用。

### 6. 这一部分最关键的结论

可以把这节压缩成一句话：

> Function Call 和 MCP 解决“调用机制”，而工具架构设计解决“调用质量”。

真正决定工具调用系统上限的，不只是协议选型，而是工具本身是否足够清晰、稳定、可恢复。

## 《AI大模型RAG项目》第 13.6 小节：生产级工具调用的完整链路

工具调用真正进入生产环境后，链路就不再是“模型决定调用 -> 工具执行 -> 返回结果”这么简单。

![5e31c00f-58dc-4808-9863-7ef231922365](D:\Wechat\xwechat_files\wxid_j2nnpd3x4n3122_4f70\temp\InputTemp\5e31c00f-58dc-4808-9863-7ef231922365.png)

在真实系统里，一次工具调用通常至少会经过这些层：

- Host：承接用户请求、管理工具调用流程
- LLM：做意图判断与工具选择
- 安全层：拦截越权、注入、非法参数
- 容错层：处理超时、重试、熔断、降级
- 下游业务系统：例如 HR、订单、CRM、支付系统
- 可观测层：日志、指标、链路追踪、告警

所以生产级工具调用，本质上是一条“决策 + 校验 + 保护 + 执行 + 观测 + 生成”的完整链路。

### 1. 第一步：LLM 先决定“要不要调哪个工具”

用户问题进入 Host 后，不会立刻直接打到业务系统，而是先进入模型判断阶段。

例如用户问：

- “我还剩几天年假？”

这时 Host 会把：

- 当前用户请求
- 可用工具列表
- 工具描述与参数要求

一起交给 LLM，由模型判断是否应该调用工具，以及调用哪个工具。

例如模型可能返回：

- `getUserAnnualLeave(userId)`

这一层解决的是：

- 要不要调用工具
- 该调哪个工具
- 参数应该怎么组织

### 2. 第二步：真正执行前必须先过安全校验

模型决定要调用工具，不代表就可以直接执行。

生产系统里，工具调用前通常必须先经过安全层，重点校验这些内容：

- 权限校验，例如 RBAC，确认当前用户能不能调用这个工具
- 防注入检查，避免恶意构造参数进入 SQL、命令行、模板或下游接口
- 参数格式校验，确保字段类型、长度、枚举值、必填项都符合要求

这一层的核心思想是：

> 模型可以提出调用意图，但是否允许执行，必须由系统规则决定。

### 3. 第三步：执行必须被容错层保护起来

安全校验通过后，Host 也不应该直接裸调下游系统，而是要把调用放进容错保护之下。

常见的保护动作包括：

- 设置超时时间，避免请求无限卡住
- 检查熔断器是否打开，避免持续把流量打向故障系统
- 针对瞬时故障做有限重试
- 连续失败后触发降级，而不是一直把异常往上抛

所以这一层的目标不是“让调用一定成功”，而是：

- 把失败控制在可恢复范围内
- 防止单个工具拖垮整条链路

### 4. 第四步：下游业务系统执行，结果可能成功、重试成功或降级返回

真正调用下游系统后，一般会出现三种典型结果。

执行成功：

- 下游系统正常返回业务结果
- Host 收到结果后继续后处理

首次失败但重试成功：

- 例如网络抖动、临时超时
- 容错层触发一次有限重试
- 第二次请求成功，仍然把结果返回上来

持续失败后降级：

- 下游持续超时或报错
- 熔断计数被更新
- Host 返回一个降级响应，例如“系统繁忙，请稍后再试”

这里最关键的认识是：

> 生产系统里的目标不是“绝不失败”，而是“失败时也要可控、可解释、可降级”。

### 5. 第五步：后处理不是可选项，而是生产必需

工具调用完成后，系统通常还要做一层后处理。

最常见的后处理包括：

- 敏感信息脱敏，避免把原始内部字段直接暴露给模型或用户
- 记录日志、指标、Span，形成可观测数据
- 检查告警规则，例如成功率、P95 延迟、熔断打开次数

这一步的意义在于：

- 让问题能被追踪
- 让风险能被发现
- 让系统状态能被量化

### 6. 第六步：模型基于工具结果再生成用户答案

工具返回给 Host 的，往往还是结构化结果，不一定直接适合展示给用户。

所以最后通常还会再经过一层模型生成或模板组装，把结果转成自然语言回复。

例如：

- 工具返回：`{ "leaveBalance": 7 }`
- 最终答案：`您还剩 7 天年假`

这也说明，工具调用链路里通常存在两层不同结果：

- 一层是工具的原始结构化返回
- 一层是面向用户的最终自然语言结果

### 7. 这一整条链路真正解决了什么

如果只看表面，工具调用像是在“帮模型查一次外部数据”。

但从生产视角看，它真正解决的是：

- 让模型能安全地调用外部能力
- 让调用失败时有重试、熔断、降级
- 让执行过程有日志、指标和告警
- 让最终回答既利用了工具结果，又不会把内部系统细节直接暴露给用户

### 8. 这一部分最关键的结论

可以把生产级工具调用链路压缩成一句话：

> 用户问题先由模型决定工具调用意图，再经过安全校验、容错保护和下游执行，最后结合可观测与后处理结果生成对用户可读的最终答案。

所以真正成熟的工具调用系统，核心不只是“模型会不会调工具”，而是：

- 调用前能不能拦住风险
- 调用中能不能扛住故障
- 调用后能不能看见问题

## 《AI大模型RAG项目》第 13.7 小节：工具调用最佳实践总表与检查清单

前面的几节分别讲了：

- 工具定义怎么设计
- 生产级链路怎么跑

这一节可以把它们压缩成一套“分阶段总表 + 检查清单”，用于做工具调用方案评审和上线前自查。

### 1. 分阶段总表：每一层最该关注什么

| 阶段 | 关键原则 | 具体措施 |
| --- | --- | --- |
| 设计阶段 | 单一职责、参数最小化、幂等性、返回值结构化 | 一个工具只做一件事；只暴露必要参数；操作类工具传 `requestId`；统一 JSON 返回 |
| 描述阶段 | 三要素齐全、关键词优化、突出差异、避免歧义 | 写清功能说明 + 适用场景 + 参数说明；覆盖用户常见说法；用“优先使用”“仅当”拉开边界 |
| 实现阶段 | 参数校验、超时控制、重试策略、降级方案、熔断机制 | 校验类型/格式/范围/权限；设置合理超时；指数退避重试；兜底信息或缓存；使用熔断器 |
| 安全阶段 | 权限控制、防注入、敏感信息脱敏、审计日志 | 基于身份和角色做校验；防 SQL/命令/XSS 注入；手机号/身份证脱敏；记录调用审计信息 |
| 运维阶段 | 日志记录、指标监控、链路追踪、告警机制 | 结构化日志；监控调用量/成功率/耗时/错误分布；traceId / Span；基于阈值告警 |

### 2. 设计阶段检查清单

- 工具名是否语义化，不使用晦涩缩写
- 工具是否符合单一职责原则，一个工具只做一件事
- 参数是否最小化，只保留必要参数
- 参数名是否语义化，不使用 `q`、`n`、`t` 这类难理解缩写
- 操作类工具是否支持幂等，例如传入 `requestId`

### 3. 描述阶段检查清单

- `description` 是否包含清晰的功能说明
- `description` 是否包含适用场景和关键词
- 每个参数是否都有详细说明，包括含义、格式、示例、默认值
- 多工具场景下是否明确写出了工具边界与差异

### 4. 实现阶段检查清单

- 是否做了参数校验，包括类型、格式、范围、权限
- 是否设置了超时控制，例如 5~30 秒
- 是否有重试策略，例如指数退避，且最大次数有限
- 是否有降级方案，例如兜底信息、缓存数据、引导用户
- 是否有熔断机制，例如 Circuit Breaker
- 是否有权限控制，基于用户身份决定能否调用
- 是否防住了注入攻击，例如 SQL 注入、路径穿越、XSS
- 敏感信息是否做了脱敏，例如手机号、身份证号
- 是否记录了审计日志，例如谁在什么时间调用了哪个工具、传了哪些关键参数

### 5. 测试阶段检查清单

- 是否有单元测试，覆盖参数校验、业务逻辑、错误处理
- 是否有集成测试，验证模型是否能正确选对工具、传对参数
- 是否有压力测试，验证高并发场景下的表现

### 6. 上线阶段检查清单

- 是否有结构化日志，例如 JSON 日志
- 是否有指标监控，例如调用量、成功率、耗时、错误分布
- 是否有链路追踪，例如 `traceId`、OpenTelemetry
- 是否有告警机制，例如成功率下降、耗时升高、错误率突增

### 7. 一个很重要的现实判断

很多工具调用方案失败，不是因为协议选错了，而是因为只做了“前两层”：

- 只设计了工具定义
- 只接通了模型调用

却没有继续补齐：

- 实现层的校验与容错
- 安全层的权限与防护
- 运维层的监控与告警

所以真正靠谱的工具调用系统，不是“能调起来”就够了，而是要做到：

- 模型容易用
- 业务容易稳
- 风险容易控
- 问题容易查

### 8. 这一部分最关键的结论

可以把这节压缩成一句话：

> 工具调用最佳实践不是某一个技巧，而是一套覆盖设计、描述、实现、安全、测试、上线全生命周期的系统性规范。

只有把这几层一起做好，工具调用才真的能从 Demo 走向生产。

## 《AI大模型RAG项目》第 14 小节：这些知识点之间是什么关系

到目前为止，这些知识点不是零散存在的，而是可以串成一条清晰链路：

```text
文档解析
-> 文本切块
-> 元数据整理
-> Embedding 向量化
-> 向量数据库存储
-> Dense / Sparse / Hybrid 检索
-> RRF / Rerank 排序优化
-> Prompt 约束生成
-> 引用对齐
-> Function Call 核验细节
```

如果从“每层解决什么问题”的角度看，可以这样理解：

- 文档解析：把文件变成文本
- Chunk：把文本变成检索单元
- 元数据：让资料可解释、可管理
- Embedding：把语义变成可计算表示
- 向量库：让相似检索可高效运行
- Dense / Sparse / Hybrid：决定怎么召回
- RRF / Rerank：决定怎么把结果排得更稳更准
- Prompt：决定模型如何被约束
- 引用对齐：决定答案如何可追溯
- Function Call：决定模型能否进一步核验细节

### 一个很重要的阶段性理解

RAG 的效果从来不是某一层单独决定的，而是整条链路共同决定的。

也就是说：

- 检索差，生成再强也会被带偏
- 资料对，Prompt 弱，仍然会发散
- 引用有，但对齐错，还是不可信
- 工具有，但不触发，也起不到核验作用

## 《AI大模型RAG项目》第 16 小节：MCP 协议入门与实践

MCP（Model Context Protocol）可以先把它理解成一套“让模型与外部能力标准化对接”的协议。

如果只从表面看，很容易把 MCP 误以为是“给大模型加工具调用”。但真正更重要的是：它不是某个项目里临时写出来的一组接口，而是在协议层统一规定了模型如何发现能力、如何读取上下文、如何触发执行。

也就是说，MCP 解决的并不是“能不能调某个方法”，而是“模型如何用统一方式接入外部世界”。

### 1. MCP 解决的核心问题是什么

在没有 MCP 之前，不同客户端、不同 Agent 框架、不同业务系统，往往都会各自定义一套工具接入方式。

这样会带来几个问题：

- 模型侧接入方式不统一
- 工具描述方式不统一
- 上下文读取方式不统一
- 一个系统里写好的能力，迁移到另一个系统时复用成本很高

MCP 的价值就在于：

- 用统一协议描述能力
- 用统一结构暴露给客户端
- 让客户端和服务端按同一套规则协作

所以它更像“模型能力接入标准”，而不只是“工具调用功能”。

### 2. MCP 不只有 Tool

初学时最容易把 MCP 只理解成 Tool。

其实 MCP 里至少要区分三类核心能力：

- Tool：可执行能力，强调“做一件事”
- Resource：可读取资源，强调“拿到一段内容”
- Prompt：可复用提示模板，强调“组织一套提示词结构”

这三个东西分别对应三种完全不同的职责：

- Tool 面向动作执行
- Resource 面向上下文供给
- Prompt 面向提示复用

如果只盯着 Tool，就会把 MCP 理解窄了。

### 3. 为什么 MCP 适合接入 RAG 体系

RAG 本质上就不是单纯的一问一答，而是一条“检索、组织上下文、生成答案、必要时再调用外部能力”的链路。

因此 MCP 很适合和 RAG 结合，原因在于：

- Tool 可以补充外部动作能力
- Resource 可以补充结构化上下文来源
- Prompt 可以把问答规则模板化

这样一来，RAG 不再只是“检索后拼 Prompt”，而是可以进一步变成“协议化地管理知识、上下文与能力”。

### 4. 核心认识

MCP 的重点不是“又多了一个接口层”，而是把模型与外部能力之间的协作方式标准化了。

记住这一点很关键：

- Tool 是能力暴露
- Resource 是内容暴露
- Prompt 是规则暴露
- MCP 是统一这些暴露方式的协议

### 5. 常见误区

- 误区一：把 MCP 等同于 Function Call
  Function Call 更像模型生成结构化调用意图的一种能力，而 MCP 是客户端与服务端之间的协议化协作方式。

- 误区二：把 MCP 只看成 Tool 容器
  实际上 MCP 还包含 Resource 和 Prompt，三者不能混在一起理解。

- 误区三：以为 MCP 只是某个客户端专属功能
  它本质上是协议层思路，不是某个具体客户端私有的业务写法。

## 《AI大模型RAG项目》第 17 小节：深入解析 MCP 中的 Resources 与 Prompts

很多人在刚接触 MCP 时，最容易理解的是 Tool，因为 Tool 很直观，就是“执行一个动作”。

但真正一旦往下做，就会发现 Resource 和 Prompt 才是 MCP 中特别值得单独理解的两部分，因为它们决定了“上下文从哪来”和“规则怎么复用”。

### 1. 什么是 Resource

Resource 可以理解成“可被客户端读取的资源内容”。

它强调的不是执行，而是提供内容。

所以 Resource 更像：

- 一份文档
- 一段结构化数据
- 一个可按 URI 访问的上下文片段
- 一个带参数的资源模板

它解决的问题是：

“当模型需要参考某些资料时，这些资料如何以标准化方式暴露出来？”

因此，Resource 的本质是内容供给，不是动作执行。

### 2. 什么是 Prompt

Prompt 则可以理解成“可复用的提示模板”。

它不是最终答案，也不是外部动作，而是把某一类任务所需的角色设定、约束规则、输出结构提前固化下来。

它解决的是：

“面对一类重复出现的问题，如何用统一模板组织提示词？”

所以 Prompt 更偏向于：

- 角色约束
- 回答规则
- 输出格式
- 参数化拼装

它的核心价值是复用和规范，而不是执行。

### 3. Tool、Resource、Prompt 三者怎么区分

可以用一句非常实用的话来区分：

- 要“做事”，用 Tool
- 要“拿内容”，用 Resource
- 要“定规则”，用 Prompt

如果继续细分：

- Tool 关注副作用和执行结果
- Resource 关注内容读取和上下文补充
- Prompt 关注提示组织和任务模板化

一旦这个边界清楚了，后面设计 MCP Server 时思路就会清晰很多。

### 4. 核心认识

Resource 和 Prompt 的价值，在于把“上下文供给”和“提示规则”从具体业务代码里抽离出来。

这样做的好处是：

- 上下文来源更清晰
- 提示模板更可复用
- 客户端看到的能力边界更明确
- 整体系统更容易维护

### 5. 常见误区

- 误区一：把 Resource 当成不会写入的 Tool
  它们的职责不同，Resource 核心是读内容，不是执行动作。

- 误区二：把 Prompt 当成普通字符串配置
  Prompt 在 MCP 里强调的是“模板能力”，不是简单文本片段。

- 误区三：什么都想做成 Tool
  如果本质只是提供资料或模板，硬做成 Tool 会让能力边界越来越乱。

## 《AI大模型RAG项目》第 18 小节：MCP 之官方 Java-SDK 深度解析

从代码层面看，MCP Server 的注册过程，最核心的不是“用了什么写法”，而是“客户端能看到什么定义、服务端最终执行哪个处理逻辑”。

也就是说，不管是 Tool、Resource 还是 Prompt，它们本质上都离不开两部分：

- 一部分是给客户端看的定义信息
- 一部分是真正被执行的处理逻辑

### 1. 注册的本质是什么

所谓“注册进去”，本质上就是在服务端建立一层映射关系：

- 客户端看到一个能力定义
- 当客户端触发这个能力时
- 服务端能准确找到对应的处理方法

所以注册不是“代码写在那就自动生效”，而是要让框架知道：

- 这个能力叫什么
- 它接收什么参数
- 它返回什么内容
- 真正由哪个处理器执行

### 2. 为什么不用注解也能注册

很多人一开始会觉得，不写注解就像“没注册”。

其实不是。

注解只是一种更方便的声明方式，它本质上是让框架帮你自动提取定义，再自动建立映射关系。

而手动注册的方式，则是你自己明确告诉框架：

- 我要暴露这个能力
- 它的定义是什么
- 它对应的执行逻辑是什么

所以两种方式的本质没有变，变的只是“谁来写这层映射”。

### 3. 注解注册与手动注册的差别

可以把两者对比着理解：

注解注册：

- 偏声明式
- 写起来更简洁
- 由框架扫描并生成注册信息
- 适合快速暴露标准能力

手动注册：

- 偏编程式
- 写法更明确
- 由开发者自己构造定义和处理器
- 适合版本差异较大或需要更细粒度控制的场景

所以“有没有注解”不是关键，关键是服务端最终有没有把定义和处理逻辑正确绑定起来。

### 4. 核心认识

理解 SDK 时最重要的一点是：

不管表面 API 怎么变，底层思想都没变，就是“定义”和“处理器”的绑定。

这套理解可以同时解释：

- Tool 为什么能被调用
- Resource 为什么能被读取
- Prompt 为什么能被客户端发现

因为三者都离不开“能力定义 + 执行入口”这个结构。

### 5. 常见误区

- 误区一：觉得只有加了注解才算注册
  实际上注解只是自动化注册手段，不是注册本质。

- 误区二：把 SDK API 名字当成核心
  真正要理解的是注册模型，而不是某个版本的具体类名。

- 误区三：只关注客户端能看到什么，不关注服务端真正执行什么
  如果只看定义不看处理逻辑，很容易在调试时找不到问题根源。

## 《AI大模型RAG项目》第 19 小节：MCP 客户端调用链与执行边界

当我们在客户端里输入一句“请调用知识库工具，回答某个问题”时，背后并不是 MCP Server 自己在“主动聊天”，而是一条分层很明确的调用链。

把这条链路看清楚，很多“为什么会这样返回”的问题就容易想明白了。

### 1. 一条完整调用链是怎么走的

可以把流程拆成下面几步：

1. 用户在客户端发起问题
2. 客户端里的模型先理解问题
3. 客户端模型判断是否要调用 MCP 能力
4. 如果决定调用，就向 MCP Server 发起能力请求
5. MCP Server 执行对应 Tool / Resource / Prompt 的处理逻辑
6. 如果 Tool 内部还接了检索、RAG、函数调用或其他模型，它会继续往下执行
7. MCP Server 把原始执行结果返回给客户端
8. 客户端模型再把结果组织成最终展示给用户的话术

所以用户看到的最终回答，往往已经不是 MCP Server 的“原始返回”了，而是客户端模型加工后的结果。

### 2. 为什么会出现“下面像总结过一层”的感觉

这是因为客户端模型通常会把工具返回结果重新整理成更自然的回答。

于是就会出现两层结果：

- 一层是 MCP Server 返回的原始结构化结果
- 一层是客户端模型面向用户生成的自然语言结果

如果客户端把原始 JSON 也展示出来，你就能明显看到这两层。

### 3. MCP 内部还能再调用模型吗

可以，而且这是理解执行边界时非常重要的一点。

客户端里的模型，和 MCP Server 内部真正执行业务逻辑时依赖的模型，不一定是同一个模型。

也就是说：

- 客户端模型负责“理解用户、决定是否调用、组织最终回复”
- MCP 内部模型负责“完成 Server 内部那段业务链路”

所以即使用户在客户端里看到的是一个模型名称，也不代表整个链路只用了这一种模型。

### 4. 核心认识

调试 MCP 问题时，一定要把边界拆开看：

- 是客户端模型没决定调用
- 还是 MCP Server 根本没启动成功
- 还是工具本身执行失败
- 还是工具内部依赖的检索、模型、配置出了问题
- 还是客户端对原始结果又做了一层总结

一旦分层清楚，问题定位就会快很多。

### 5. 常见误区

- 误区一：把客户端最终看到的话，全部当成 MCP Server 原样返回
  实际上很多时候中间还有客户端模型的重组。

- 误区二：以为客户端模型和 Server 内部模型必然相同
  两者完全可以不同，职责也不同。

- 误区三：只盯着聊天界面，不去区分协议层返回和展示层包装
  这样很容易误判问题到底出在哪一层。

## 《AI大模型RAG项目》第 20 小节：STDIO 型 MCP Server 的运行方式与排错思路

很多桌面客户端接入 MCP Server 时，常见方式是通过 STDIO 启动一个本地进程，然后靠标准输入输出和它通信。

这套方式看起来简单，但调试时特别容易混淆“进程启动问题”“协议通信问题”“业务执行问题”这三层边界。

### 1. 什么叫 STDIO 型 MCP Server

STDIO 的意思就是：

- 客户端启动一个本地进程
- 通过标准输入把请求发给它
- 通过标准输出接收它返回的协议消息

也就是说，这个 Server 往往不是先启动一个 Web 服务等别人来调，而是被客户端作为子进程直接拉起。

### 2. 为什么日志不能乱打到标准输出

因为在 STDIO 模式下，标准输出本身就是协议通道。

如果把普通日志、调试打印、启动信息一股脑打到 stdout，就可能污染协议消息，导致客户端解析失败。

所以一个很重要的经验就是：

- 协议消息走标准输出
- 普通日志尽量写文件，或者走标准错误输出

这也是为什么很多 MCP 配置里会专门把日志输出方式单独处理。

### 3. 常见问题应该怎么分层排查

遇到问题时，不要一下子把所有原因混成一团，而是按层排查：

第一层：客户端能不能成功拉起进程

要关注：

- `java` 路径对不对
- `JAVA_HOME` 是否正确
- jar 路径是否正确
- Windows 下 JSON 里的路径转义是否正确

第二层：Server 启动后，能力有没有注册成功

要关注：

- Tool / Resource / Prompt 是否真的暴露出来
- 注册逻辑是否生效
- 版本与依赖是否匹配

第三层：能力被调用后，内部业务链路是否能执行成功

要关注：

- API Key 是否存在
- 内部模型是否可用
- 检索与数据库配置是否正常
- RAG 流程本身是否报错

### 4. 核心认识

STDIO 模式下最重要的，不是“它有没有报错”，而是先判断错误落在哪一层：

- 启动层
- 协议层
- 能力注册层
- 业务执行层

只有先把层次拆开，排查才不会越查越乱。

### 5. 常见误区

- 误区一：客户端连不上，就立刻怀疑工具代码写错了
  其实很多时候连进程都没启动起来。

- 误区二：看到 Server 启动了，就认为工具一定能正常工作
  启动成功不等于注册成功，也不等于内部依赖都可用。

- 误区三：把普通日志打印到 stdout
  这在 STDIO 模式下很容易直接影响协议通信。

## 《AI大模型RAG项目》第 21 小节：对话记忆与上下文管理策略对比

在多轮对话系统里，一个很现实的问题是：上下文不可能无限增长。

随着对话轮数增加，历史消息会不断消耗 Token，如果没有管理策略，就会带来成本上升、响应变慢，甚至上下文超限的问题。

所以，“如何保留历史信息”本质上是一个上下文管理问题。

### 1. 常见上下文管理策略对比

| 策略 | Token 控制 | 信息保留 | 实现复杂度 | 额外 API 调用 | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| 完整历史 | 无控制 | 完整 | 极低 | 无 | 对话 ≤ 5 轮的简单场景 |
| 滑动窗口 | 按轮数控制 | 丢失早期 | 低 | 无 | 大多数客服场景 |
| Token 截断 | 按 Token 精确控制 | 丢失早期 | 中 | 无 | 消息长度差异大的场景 |
| 摘要压缩 | 大幅压缩 | 保留关键信息 | 高 | 每次压缩 1 次 | 长对话、需要长期记忆 |
| 混合策略 | 精确可控 | 长期摘要 + 短期完整 | 高 | 触发时 1 次 | 生产级系统（推荐） |

### 2. 每种策略的核心理解

完整历史：

最简单，直接把所有历史消息都带给模型。

优点是实现几乎没有成本，信息也最完整；缺点是 Token 会一直增长，所以只适合很短的对话。

滑动窗口：

只保留最近几轮消息，把更早的内容丢掉。

它的优点是实现简单、开销低，适合大量常规问答；但缺点也很明显，一旦早期信息很重要，就容易被窗口滑掉。

Token 截断：

不是按轮数裁剪，而是按 Token 上限控制。

这样相比滑动窗口会更精确，因为有些消息很长、有些消息很短，只按轮数并不稳定。但它仍然会丢失早期信息，只是控制方式更细。

摘要压缩：

把较早历史先总结成一段摘要，再把摘要和最近消息一起交给模型。

它的核心价值是：在有限上下文里保留长期记忆的关键部分。但代价是实现复杂，而且每次做摘要时通常都要额外调用一次模型。

混合策略：

把“长期摘要”和“短期原始上下文”结合起来使用。

也就是说，较早的信息压缩成摘要保留，最近几轮消息则原样保留，从而兼顾长期记忆和近期细节。

### 3. 为什么混合策略更适合生产环境

如果只用完整历史，成本会越来越高。

如果只用滑动窗口或 Token 截断，虽然控制住了成本，但长期重要信息容易丢。

如果只用摘要压缩，又可能在摘要过程中损失一些细节，而且每次压缩都会增加额外调用成本。

所以混合策略通常更平衡：

- 用摘要保存长期信息
- 用原始消息保留近期细节
- 用阈值控制何时触发压缩

这也是为什么它更适合作为生产级方案。

### 4. 核心认识

上下文管理没有绝对最优方案，只有“在成本、信息保留、实现复杂度之间做权衡”的方案。

可以这样记：

- 想要最省事：完整历史
- 想要简单可用：滑动窗口
- 想要更精细控制：Token 截断
- 想要长期记忆：摘要压缩
- 想要生产平衡：混合策略

### 5. 常见误区

- 误区一：以为只要保留的消息越多，效果就一定越好
  上下文过长不仅成本高，还可能让模型注意力分散。

- 误区二：以为滑动窗口已经足够解决所有问题
  一旦业务依赖早期信息，滑动窗口就可能丢关键上下文。

- 误区三：以为摘要压缩一定优于截断
  摘要虽然能保留关键信息，但也会引入摘要误差和额外调用成本。

- 误区四：忽视“近期细节”和“长期记忆”其实是两类不同需求
  真正稳定的方案，往往需要同时兼顾这两类信息。

## 《AI大模型RAG项目》第 22 小节：会话记忆的存储方案对比

当系统开始支持多轮对话后，除了要考虑“记忆保留策略”，还必须考虑另一个问题：这些会话记忆到底存在哪里。

因为记忆策略解决的是“保留多少、如何压缩”，而存储方案解决的是“这些记忆落到哪里、读写是否足够快、能不能持久保存、能不能支持生产环境扩展”。

所以会话记忆的设计，通常要同时回答两个问题：

- 用什么策略管理上下文
- 用什么介质存储上下文

### 1. 常见会话记忆存储方案对比

| 维度 | 内存 | Redis | 数据库（MySQL） |
| --- | --- | --- | --- |
| 读写性能 | 极快（纳秒） | 快（毫秒） | 较慢（毫秒到十毫秒） |
| 持久化 | 不支持 | 可选（AOF/RDB） | 天然支持 |
| 分布式 | 不支持 | 天然支持 | 支持 |
| 过期清理 | 需自己实现 | 原生支持（TTL） | 需定时任务 |
| 实现复杂度 | 极低 | 低 | 中 |
| 适用场景 | 开发调试 | 生产环境 | 审计 / 分析场景 |

### 2. 三种方案分别适合什么阶段

内存：

最简单，通常直接把会话记录放在应用进程里。

它的优点是速度极快、实现成本最低，特别适合开发阶段验证对话流程。但缺点也非常明显：

- 服务一重启，记忆就没了
- 无法跨实例共享
- 不适合多机部署

所以内存更适合开发调试或单机临时验证，而不适合正式生产环境。

Redis：

Redis 通常是会话记忆最常见的生产级方案。

它兼顾了几个关键点：

- 读写够快
- 支持分布式共享
- 支持 TTL 自动过期
- 可以按需开启持久化

这意味着它既适合保存短期会话状态，也适合在多实例服务之间共享用户上下文，因此非常适合大多数线上对话系统。

数据库（MySQL）：

数据库最大的优势不是速度，而是稳定持久和可追溯。

它更适合：

- 需要保留完整会话记录
- 需要做审计与分析
- 需要后续统计、报表、复盘

但如果直接把它当成高频实时会话记忆主存储，就会遇到响应慢、清理麻烦、实时性不如缓存系统的问题。

### 3. 为什么生产里常见“Redis + 数据库”组合

如果只用 Redis，实时性很好，但历史分析和长期存档能力不够强。

如果只用数据库，持久化很好，但高频读写和过期管理不够轻量。

所以在生产系统里，更常见的思路是分层：

- Redis 保存短期活跃会话记忆
- 数据库存长期历史记录和审计数据

这样做的好处是：

- 热数据访问快
- 冷数据可追溯
- 会话状态和历史归档职责分开
- 系统扩展性更好

### 4. 核心认识

会话记忆存储方案的选择，本质上是在权衡四件事：

- 实时性能
- 持久化能力
- 分布式支持
- 清理与维护成本

可以这样快速判断：

- 只是本地调试：用内存
- 要做正式生产会话：优先 Redis
- 要做审计、报表、长期留痕：要接数据库
- 要兼顾实时对话和长期归档：Redis + 数据库组合最稳

### 5. 常见误区

- 误区一：以为会话记忆只要能存就行
  真正的问题不只是“存下来”，还包括性能、过期、分布式和恢复能力。

- 误区二：以为数据库一定比 Redis 更适合作为主存储
  数据库更擅长长期保存，不一定更适合高频实时对话状态。

- 误区三：以为 Redis 有持久化，就能替代所有历史存档方案
  Redis 更偏实时会话层，长期审计与分析通常仍然需要数据库承担。

- 误区四：把“记忆策略”和“存储介质”混为一谈
  前者解决的是怎么保留上下文，后者解决的是把上下文放在哪里。

## 《AI大模型RAG项目》第 23 小节：会话记忆在生产环境中的注意事项

当对话系统从开发验证走向生产环境后，会话记忆就不再只是“能不能存下来”的问题，而要进一步考虑清理机制、数据安全和系统稳定性。

也就是说，生产环境里要关注的不只是功能可用，还要关注：

- 记忆会不会无限增长
- 敏感信息会不会被直接落盘
- 缓存失效后能不能恢复历史会话

### 1. 会话超时与清理为什么必须提前设计

不管使用哪种存储方案，都应该给会话设置明确的过期时间。

因为在真实系统里，用户可能关闭页面、长时间不发消息，或者已经结束当前对话。如果这些历史会话一直保留，就会不断占用内存、缓存或数据库资源。

因此，一个重要原则是：

- 会话不是永久存在的
- 无活跃行为的会话应当按规则自动清理
- 清理机制必须和存储方案一起设计

### 2. 不同存储方案的清理思路

内存存储：

通常需要自己实现清理机制，比如通过定时任务扫描，或者借助本地缓存框架的过期能力完成自动清理。

Redis 存储：

比较适合通过 TTL 直接设置过期时间，让 Redis 自动删除过期会话。这也是它在生产环境里很适合作为短期会话层的一个重要原因。

数据库存储：

数据库更适合保存长期记录，因此清理策略通常不是“自动立即删除”，而是通过定时任务清理已过期会话，或者把会话标记为“已关闭”。

### 3. 为什么不清理会带来真实风险

如果系统没有会话过期和清理机制，就会带来明显的资源风险。

例如：

- 内存存储会持续膨胀，最终可能引发 OOM
- Redis 会因为会话无限积累而出现内存不足
- 数据库里的历史会话也会不断堆积，增加维护和查询成本

所以清理机制不是“优化项”，而是生产环境中的基础能力。

### 4. 敏感信息处理为什么不能忽视

对话历史中，可能天然就会包含敏感信息，例如：

- 身份证号
- 手机号
- 银行卡号
- 密码或口令类信息

如果直接原样存储，会带来明显的数据泄露风险。因此，会话记忆在生产环境里必须考虑安全处理。

### 5. 常见的敏感信息保护方式

脱敏存储：

对敏感字段做掩码处理，例如手机号只显示部分内容，这样即使被查看，也不会直接暴露完整数据。

加密存储：

对整段消息或敏感字段加密，读取时再解密。这样可以降低底层存储泄露带来的风险。

访问控制：

不是所有人都应该能查看会话历史。生产系统通常还需要限制访问权限，并记录访问日志，确保谁看过、什么时候看过都能追踪。

### 6. Redis 与 MySQL 分层存储的补充理解

生产环境里，如果使用 Redis 保存活跃会话，即使缓存里的内容已经因为 TTL 被清理，只要用户没有主动删除该对话，系统仍然可以从 MySQL 中读取历史内容，再重新恢复或重建会话上下文。

这说明一个很重要的设计思路：

- Redis 更适合承载短期活跃上下文
- MySQL 更适合承载长期历史记录
- 两者结合可以兼顾实时性与可追溯性

### 7. 核心认识

会话记忆一旦进入生产环境，核心问题就会从“怎么实现”转向“怎么稳定、安全、可持续地运行”。

可以把重点记成三件事：

- 要有清理机制，避免资源无限增长
- 要有安全策略，避免敏感信息裸奔
- 要有分层存储思路，兼顾实时会话与历史恢复

### 8. 常见误区

- 误区一：只要功能跑通，就说明会话记忆设计没问题
  生产环境真正考验的是清理、安全和恢复能力。

- 误区二：以为 Redis 设置了 TTL，就不需要历史存储了
  TTL 解决的是活跃会话过期，不等于历史记录不再需要保留。

- 误区三：以为数据库天然持久化，就可以什么都原样存进去
  持久化不等于安全，敏感信息仍然需要脱敏、加密和访问控制。

## 《AI大模型RAG项目》第 24 小节：多轮对话场景下的查询改写策略对比

当 RAG 进入多轮对话场景后，用户当前这一句提问，往往并不是一个适合直接检索的完整查询。

因为真实对话里经常会出现：

- 代词引用前文
- 信息被省略
- 口语化表达过强
- 一个问题里混着多个意图
- 用户表达和文档表述不一致

所以在进入检索之前，通常需要先做查询改写（Query Rewrite）。

它的本质不是“把问题换个说法”，而是把用户的表达，改造成更适合检索系统理解的查询。

### 1. 常见查询改写策略对比

| 策略 | 解决的问题 | 依赖对话历史 | 实现复杂度 | 对检索的影响 | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| 指代消解 | 代词无法检索 | 是 | 低 | 必需，否则检索失败 | 多轮对话（最常用） |
| 上下文补全 | 省略信息无法检索 | 是 | 低 | 必需，否则检索不精准 | 多轮对话 |
| 口语化转正式 | 口语与文档的语义鸿沟 | 否 | 中 | 有提升，尤其对 BM25 | 单轮 + 多轮 |
| 多意图拆分 | 一次检索无法覆盖多个意图 | 否 | 高 | 有提升，但成本翻倍 | 复杂咨询场景 |
| 关键词扩展 | 同义词不匹配 | 否 | 低 | 对 BM25 有帮助 | 混合检索场景 |

### 2. 为什么指代消解和上下文补全几乎是多轮检索的基础能力

在多轮对话里，用户经常会说：

- “那它多久能到？”
- “这个支持退吗？”
- “如果换成另一个方案呢？”

这类问题如果脱离上文，往往根本无法检索，因为：

- “它”指的对象不明确
- “这个”缺少具体实体
- 当前句子省略了大量前文信息

所以在多轮场景里，指代消解和上下文补全通常不是可选增强，而是基础前置能力。没有这一步，多轮检索往往会直接失效，或者精度明显下降。

### 3. 为什么口语化转正式和关键词扩展能提升检索效果

很多用户提问是口语表达，但知识库文档往往是正式书面表达。

例如用户说法和文档写法之间，可能存在：

- 口语与术语不一致
- 同义词不一致
- 用户描述过于模糊

这时做口语化转正式、关键词扩展，就能让查询更贴近文档语料的表达方式。

这种提升对稀疏检索尤其明显，因为像 BM25 这类方法对词项匹配更敏感。

### 4. 为什么多意图拆分有效但成本更高

有些用户问题天然包含多个检索目标，例如：

- 既问退款规则，又问到账时间
- 既问发货时效，又问物流查看方式

如果只做一次整体检索，可能无法同时把多个子问题都覆盖完整。

这时把问题拆成多个子意图分别检索，通常能提高召回完整度。

但代价也很明显：

- 检索次数增加
- 排序和融合复杂度增加
- 推理与整体链路成本上升

所以它更适合复杂咨询场景，而不是所有请求都默认开启。

### 5. 实际使用时的优先级怎么判断

如果从投入产出比来看，可以这样理解：

- 指代消解：优先级最高，多轮对话必做
- 上下文补全：优先级同样很高，多轮检索基础能力
- 口语化转正式：建议做，收益通常比较稳定
- 关键词扩展：按检索方案决定，尤其适合 BM25 或混合检索
- 多意图拆分：按业务复杂度决定，不宜默认全量开启

也就是说，真正最值得先做的是“让问题变完整、变可检索”；然后再考虑“让查询更贴近文档表达”；最后才是“为复杂问题做更高成本增强”。

### 6. 核心认识

查询改写不是附属优化，而是多轮 RAG 中影响检索效果的重要前置层。

可以把这几类策略分成两组来记：

- 基础必需型：指代消解、上下文补全
- 效果增强型：口语化转正式、关键词扩展、多意图拆分

前一组解决“能不能检索”，后一组解决“能不能检索得更好”。

### 7. 常见误区

- 误区一：以为用户原问题可以直接拿去检索
  多轮对话里，原问题往往并不是完整查询。

- 误区二：以为所有查询改写策略都应该默认开启
  不同策略的收益和成本差异很大，应该按场景分层使用。

- 误区三：忽视 BM25 对词项表达非常敏感
  在稀疏检索或混合检索中，查询改写对召回效果影响会更明显。

- 误区四：把多意图拆分当成通用默认方案
  它虽然能提升复杂问题的覆盖度，但也会显著增加系统成本。

## 《AI大模型RAG项目》第 25 小节：Query 改写在 RAG 链路中的位置与职责

在多轮对话 RAG 中，Query 改写很容易被误解成“把用户问题改掉”。

但它真正的职责并不是替用户重新提问，而是把原始问题转换成“更适合检索系统理解”的查询表达。

所以一定要区分两个对象：

- 用户原始问题：决定最终答案应该回答什么
- 改写后的 Query：决定系统应该去检索什么资料

### 1. 为什么检索用改写后的 Query，而 Prompt 里通常放原始问题

这是因为检索和生成的目标并不一样。

检索阶段的目标是：

- 尽可能准确地找到相关资料
- 补全省略信息
- 消解代词指向
- 让查询更贴近知识库表达

所以这一阶段更需要“机器更容易检索”的表达形式。

而生成阶段的目标是：

- 直接回答用户真正问的问题
- 保持回答语义和用户意图一致
- 不因为技术性改写而偏离用户原意

所以 Prompt 里通常应该保留用户原始问题，因为模型最终面向的是“用户想问什么”，不是“检索层为了召回而生成的中间查询”。

### 2. 为什么不能直接拿改写后的 Query 代替用户问题

因为 Query 改写本质上是一种“面向检索”的技术处理，它可能会：

- 补充隐含上下文
- 展开代词指向
- 改成更正式的检索表达
- 拆成更利于召回的形式

这些操作有助于检索，但不一定适合作为最终回答时的原问题。

如果直接把改写后的 Query 当成用户问题，就可能出现两个风险：

- 回答语言风格变得机械，不像在真正回应用户
- 改写过程中加入的检索性补充，可能让模型偏离用户本来最关心的表达重点

所以更稳的做法通常是：

- 检索时使用改写后的 Query
- 生成时使用用户原始问题
- 必要时把“原问题 + 改写后 Query”一起放进 Prompt，作为辅助信息

### 3. 为什么“原问题 + 改写 Query 一起放入 Prompt”会更保险

有时候系统会担心两件事：

- 只放原问题，模型不知道检索时到底补全了哪些上下文
- 只放改写 Query，又可能偏离用户原始表述

所以一个折中且更稳妥的方式是：

- 明确告诉模型用户原始问题是什么
- 同时把改写后的 Query 作为“检索辅助表达”提供给模型

这样模型既能保持面向用户原始问题作答，又能理解检索阶段为什么召回了这些资料。

不过要注意，主问题仍然应该是原始问题，改写 Query 只是辅助，不应反客为主。

### 4. 为什么 Query 改写一定发生在“会话记忆读取之后、检索之前”

这个位置不能随便放，因为 Query 改写本身就依赖对话历史。

原因很简单：

- 指代消解要知道“它”“这个”“那件事”到底指什么
- 上下文补全要知道用户当前问题省略了哪些前文信息
- 多轮问题重写要把当前句和历史上下文拼成完整查询

如果在读取会话记忆之前就改写，那么改写器拿不到足够上下文，就无法完成正确补全。

而如果放到检索之后再改写，就更没有意义，因为检索已经用错查询执行完了。

所以正确顺序通常是：

1. 读取会话记忆
2. 结合当前问题做 Query 改写
3. 用改写后的 Query 发起检索
4. 将检索结果与原始问题一起交给模型生成答案

### 5. 这背后的本质到底是什么

本质上，Query 改写是连接“对话理解层”和“检索执行层”的中间桥梁。

它向前承接会话历史，向后服务检索召回。

所以它既不是纯记忆模块，也不是纯生成模块，而是检索前的查询理解层。

可以用一句话记住：

- 会话记忆负责提供上下文
- Query 改写负责把上下文整理成可检索表达
- 检索负责找资料
- 生成负责回答用户原问题

### 6. 核心认识

Query 改写的目标是“提升检索质量”，不是“改写用户意图”。

因此它在整条链路中的职责边界应该非常清楚：

- 它服务检索
- 它依赖会话历史
- 它不替代原始问题
- 它只是生成答案前的中间层能力

### 7. 常见误区

- 误区一：以为改写后的 Query 就是最终问题
  它只是面向检索的中间表达，不是最终回答目标。

- 误区二：以为 Query 改写可以脱离会话历史独立完成
  多轮场景下，大部分改写恰恰依赖历史上下文。

- 误区三：把 Query 改写放到检索之后
  检索已经结束后再改写，等于错过了它真正的作用位置。

- 误区四：只关注检索召回，不关注生成阶段是否仍然围绕原问题作答
  RAG 最终要回答的是用户的问题，而不是中间层生成的检索查询。

## 《AI大模型RAG项目》第 26 小节：Query 改写核心回顾

这一部分可以作为 Query 改写的阶段性总结，用来快速复盘它为什么重要、该怎么用、放在链路的什么位置。

### 1. 检索系统真正面临的问题是什么

在多轮对话里，会话记忆可以让模型具备上下文理解能力，但检索系统拿到的往往仍然是用户当前这句原始 Query。

而原始 Query 往往存在两个问题：

- 包含代词指向
- 包含被省略的信息

这会直接导致检索效果变差，严重时甚至检索失败。

### 2. Query 改写要解决的五类核心问题

当前阶段最值得记住的五类改写策略是：

- 指代消解：必做
- 上下文补全：必做
- 口语化转正式：推荐
- 多意图拆分：按需
- 关键词扩展：在混合检索场景里更有价值

这里可以把它们分成两组：

- 基础保障型：指代消解、上下文补全
- 效果增强型：口语化转正式、多意图拆分、关键词扩展

前一组决定“能不能检索”，后一组决定“能不能检索得更好”。

### 3. 为什么通常用大模型做改写

Query 改写本质上是一种对自然语言的理解与重写任务，因此非常适合交给模型来完成。

而且它通常不需要特别大的模型就能做好，很多改写场景用小模型也足够：

- 成本更低
- 延迟更小
- 更适合放在检索前作为高频前置步骤

所以改写层并不一定追求最强模型，而更强调稳定、便宜和响应快。

### 4. Prompt 设计为什么是改写质量的关键

改写这一步虽然交给模型，但并不意味着可以让模型自由发挥。

如果 Prompt 设计不清楚，模型就可能：

- 过度改写
- 擅自补充不存在的信息
- 把原意改偏

因此改写 Prompt 的关键要求通常是：

- 只做必要补全
- 不改变用户原意
- 不凭空新增事实
- 尽量输出适合检索的稳定表达

### 5. Query 改写在整条 RAG 链路中的位置

它应该放在：

- 会话记忆之后
- 检索之前

这是因为：

- 它需要历史上下文作为输入
- 它的输出要直接服务检索

所以最关键的一句可以记成：

- 检索用改写后的 Query
- 生成回答仍然围绕用户原始问题

### 6. 核心认识

Query 改写不是在“替用户换个问法”，而是在“替检索系统准备一个更可用的查询表达”。

所以它的本质职责是：

- 连接会话记忆与检索模块
- 提高多轮对话下的检索成功率与精准度
- 保证最终生成时不偏离用户原始问题

### 7. 常见误区

- 误区一：把 Query 改写理解成最终问题重写
  它服务的是检索，不是替代用户原问题。

- 误区二：以为改写必须用大模型才行
  实际上很多场景小模型就足够，重点是性价比和稳定性。

- 误区三：忽视改写 Prompt 约束
  改写质量往往不只取决于模型，还取决于提示词是否明确限制边界。

## 《AI大模型RAG项目》第 27 小节：意图分类与路由分发

当系统能力逐渐变多之后，用户输入的一句话就不再一定只对应“走 RAG 检索”这一条路径。

因为真实场景里，用户问题可能是：

- 想查知识库里的通用知识
- 想查实时数据或个人数据
- 只是打招呼或闲聊
- 问题本身不完整，需要先澄清

所以系统在真正处理问题之前，通常需要先做一层“意图分类”，再根据不同意图把请求路由到对应链路。

### 1. 为什么意图分类是系统入口层的重要能力

如果没有意图分类，系统很容易出现“所有问题都走同一条链路”的情况。

这会带来几个明显问题：

- 本来应该查知识库的问题，却走了工具调用
- 本来需要查实时数据的问题，却错误走了 RAG
- 本来只是闲聊，却被强行当作知识问答处理
- 本来信息不足的问题，没有先澄清就直接回答

所以意图分类的价值，不是在于“多加一步判断”，而是在于让后续处理链路选得对。

### 2. 四类最基础的意图类型

当前阶段最适合先记住四种基础意图：

| 意图类型 | 触发条件 | 处理流程 | 典型示例 |
| --- | --- | --- | --- |
| 知识检索 | 问知识库里的通用知识 | Query 改写 → 检索 → 生成答案 | 退货政策是什么 |
| 工具调用 | 查个人数据 / 实时信息 / 执行操作 | Function Call / MCP | 查我的订单状态 |
| 闲聊对话 | 打招呼 / 感谢 / 闲聊 | 模型直接回答 | 你好，谢谢 |
| 引导澄清 | 问题模糊、信息不足 | 反问用户补充信息 | 有什么推荐的 |

这四类虽然基础，但已经能覆盖很多通用系统的入口判断。

### 3. 为什么知识检索和工具调用必须分开理解

这两类最容易混淆，但它们解决的问题其实不同。

知识检索更适合回答：

- 相对稳定的通用知识
- 已经写进知识库的内容
- 不依赖当前实时状态的说明类问题

而工具调用更适合处理：

- 个人专属数据
- 实时动态信息
- 需要执行动作的请求

所以像“退货政策是什么”更像知识检索，
而“查我的订单状态”更像工具调用。

前者重点是找资料，后者重点是调用外部能力。

### 4. 为什么闲聊和澄清也必须作为独立路径

很多人刚开始设计系统时，只盯着“知识问答”和“工具调用”，但真实用户输入并不总是这么标准。

例如：

- 用户只是说“你好”
- 用户只是说“谢谢”
- 用户问“有什么推荐的”

这些输入如果强行走知识检索或工具调用，往往会显得生硬，甚至处理错误。

因此：

- 闲聊类问题，直接由模型自然回复即可
- 信息不完整的问题，要先澄清，再进入后续链路

这一步其实是在避免“系统过早进入错误流程”。

### 5. 为什么真实业务里意图分类会继续细化

基础四分类只是起点，不是终点。

在真实业务里，随着场景变复杂，意图分类通常会继续细分。例如电商、客服、办公等系统中，可能还会拆出：

- 投诉
- 催单
- 比价
- 售后
- 优惠咨询

但无论怎么细化，底层思路其实不变：

- 先识别用户到底想干什么
- 再把请求路由到正确处理链路

所以“分类”是入口判断，“路由”是后续执行。

### 6. 核心认识

意图分类不是可有可无的前置装饰，而是多能力系统的入口调度层。

可以把它理解成一句话：

- 先分类
- 再路由
- 最后按对应链路执行

只有入口判断对了，后面的 RAG、MCP、Function Call、闲聊回复、澄清追问才有可能整体表现稳定。

### 7. 常见误区

- 误区一：以为所有问题都应该优先走 RAG
  真实系统里，并不是所有请求都属于知识检索。

- 误区二：把工具调用和知识检索混成一类
  一个偏“找知识”，一个偏“查数据 / 做动作”，职责不同。

- 误区三：忽视闲聊和澄清路径
  这会让系统在面对非标准输入时显得很僵硬。

- 误区四：以为意图分类必须一开始就做得非常细
  实际上可以先从基础四分类开始，再根据业务需要逐步细化。

## 《AI大模型RAG项目》第 28 小节：意图分类的实现方案对比

当我们已经知道系统入口需要做意图分类后，下一个实际问题就是：这一步到底该怎么实现。

常见思路通常有三种：

- 规则匹配
- 大模型分类
- 混合方案

它们的差别，不只是“谁更聪明”，而是要从速度、准确率、成本和维护方式几个维度一起看。

### 1. 三种方案的核心对比

| 维度 | 规则匹配 | 大模型分类 | 混合方案 |
| --- | --- | --- | --- |
| 速度 | 极快（微秒） | 较慢（300~3000ms） | 快（规则命中时微秒，未命中时 300~3000ms） |
| 准确率 | 低（60%~70%） | 高（90%+） | 高（90%+） |
| Token 成本 | 零 | 每次请求都消耗 | 仅未命中规则时消耗 |
| 维护成本 | 高（不断加规则） | 低（改 Prompt） | 中（规则 + Prompt 两套） |
| 推荐场景 | 不推荐单独使用 | 消息量小、对延迟不敏感 | 生产环境（推荐） |

### 2. 为什么规则匹配很快但不适合单独作为长期方案

规则匹配的优点非常明显：

- 没有 Token 成本
- 响应速度极快
- 对特别明确的意图命中很高

例如一些非常固定的表达：

- “你好”
- “谢谢”
- “查订单状态”

这类输入用规则匹配往往就能快速命中。

但它的问题也很明显：

- 表达一旦变复杂，规则就容易漏
- 业务不断变化时，规则会越来越多
- 维护成本会随着时间快速上升

所以规则更适合处理“非常明确、非常稳定、非常高频”的意图，而不适合独立承担全部分类任务。

### 3. 为什么大模型分类更灵活

大模型分类最大的优势，不是速度，而是泛化理解能力。

它可以处理：

- 用户表达不规范
- 语义比较隐含
- 多种说法指向同一意图
- 规则难以穷举的复杂表达

所以它的准确率通常会更高，尤其适合复杂业务场景。

但它也有代价：

- 每次分类都要调用模型
- 会增加延迟
- 会持续消耗 Token

因此，如果请求量很大、对实时性要求高，完全依赖模型分类往往成本不够友好。

### 4. 为什么混合方案更适合生产环境

混合方案的核心思路是：

- 先用规则拦截最明确的意图
- 对规则无法覆盖的请求，再交给模型分类

这样做的好处很明显：

- 明确问题走规则，响应很快
- 复杂问题交模型，准确率更高
- 只有规则未命中时才消耗 Token
- 可以在准确率和成本之间取得平衡

所以混合方案往往是生产环境里更稳妥的选择。

### 5. 为什么混合方案的本质是“分层分类”

从本质上看，混合方案不是简单把两种方法拼在一起，而是在做分层处理：

- 第一层：低成本、高速度的规则过滤
- 第二层：高理解力、较高成本的模型判断

这就和很多系统设计一样，不追求所有请求都走最强能力，而是让不同复杂度的问题进入不同层级处理。

因此混合方案的关键思想是：

- 明确问题用低成本路径解决
- 模糊问题用高理解力路径解决

### 6. 随着业务发展，规则层为什么反而会越来越有价值

虽然规则匹配不适合单独使用，但在混合方案里，它会越来越重要。

原因是：

- 业务中总会有一些高频、稳定、可明确识别的请求
- 这些请求如果长期交给模型分类，会持续产生不必要成本
- 一旦把它们沉淀成规则，就能稳定减少模型调用比例

所以随着系统成熟，通常会逐步把高频明确意图沉淀到规则层，让模型层更多处理复杂边界场景。

### 7. 核心认识

意图分类方案的选择，本质上是在平衡四件事：

- 响应速度
- 分类准确率
- Token 成本
- 长期维护成本

可以这样记：

- 只靠规则：便宜、快，但不够稳
- 只靠模型：聪明、准，但成本和延迟较高
- 规则 + 模型：通常是生产环境最稳的平衡点

### 8. 常见误区

- 误区一：以为规则匹配完全没价值
  它在高频明确场景里非常有价值，只是不适合单独覆盖全部意图。

- 误区二：以为大模型分类一定是最优解
  它虽然准确率高，但不一定适合所有流量规模和时延要求。

- 误区三：把混合方案理解成简单叠加
  真正有效的混合方案，本质上是按复杂度分层处理请求。

- 误区四：忽视分类系统也需要持续演化
  无论是规则还是 Prompt，都需要随着业务变化不断调整。

## 《AI大模型RAG项目》第 29 小节：RAG 评估策略与 Bad Case 归因

当 RAG 系统开始具备“入口路由 + 检索 + 重排 + 生成 +（可选）工具核验”之后，就不能只凭主观感觉判断效果。

因为同一个“答错/答偏/乱编”，背后的根因可能完全不同：

- 意图路由错了，走错链路
- 检索没命中，资料根本没召回
- 重排没排好，正确证据排太后
- 生成没克制，基于不充分证据编造
- 知识库内容本身就不完整或过时

因此评估的关键不是“打一份总分”，而是建立一套能定位问题落点的分层评估框架。

![6b73770c-b5c8-4ad3-8566-c430d2450c4b](D:\Wechat\xwechat_files\wxid_j2nnpd3x4n3122_4f70\temp\InputTemp\6b73770c-b5c8-4ad3-8566-c430d2450c4b.png)

### 1. 评估为什么必须分层

RAG 本质是一条链路，不是单点模型能力。

所以评估也必须按链路拆开看：

- 入口层：这句话该走哪条链路
- 检索层：证据有没有被召回、排序是否合理
- 生成层：回答是否忠实证据、是否真正回答问题
- 端到端：用户视角是否“答对、可用、稳定”

只看端到端，很容易陷入“知道错了但不知道该改哪”的困境。

### 2. 评测集（EvalCase）怎么设计最实用

一条可落地的评测样本，建议至少包含四类信息：

- `query`：用户问题
- `expectedAnswer`：标准答案（或期望兜底话术）
- `relevantChunkIds / relevantDocIds`：这道题正确证据对应的 chunk/doc 标识
- `intent`：期望意图类别（knowledge/tool/chitchat/clarification）

这四个字段分别解决四个关键问题：

- 模型到底该回答什么（expectedAnswer）
- 检索到底该召回什么（relevantChunkIds）
- 入口到底该走哪条链路（intent）
- 出问题时能不能定位到哪一层（分层归因）

### 3. 入口层评估：意图分类与路由准确率

在具备多路能力后，评估必须先看路由是否正确。

最直接的指标是：

- `intent accuracy`：预测意图与期望意图是否一致

并且要关注“混淆方向”，例如：

- tool 被误判成 knowledge：会把个人数据问题误送进知识库
- clarification 被误判成 knowledge：信息不足时直接检索，容易答偏
- chitchat 被误判成 knowledge：闲聊也去检索，系统显得很僵硬

入口路由错了，后面的检索/生成评分往往没有意义。

### 4. 检索层评估：Hit Rate 与 MRR

对 `knowledge` 类样本，最实用的两个检索指标是：

- Hit Rate：Top-K 结果中是否包含正确证据
- MRR：正确证据第一次出现的位置（越靠前越好）

它们分别回答两类问题：

- Hit Rate：能不能把证据召回来
- MRR：召回了之后排得好不好

注意一个重要边界：

- 对“知识库本来就没有答案”的兜底样本，不应强行参与 Hit/MRR（否则会把系统的合理兜底算成检索失败）

### 5. 生成层评估：Faithfulness / Relevancy / Correctness

在 RAG 场景里，只评“对不对”是不够的。

更推荐把生成质量拆成三个维度：

- Faithfulness（忠实度）：回答是否严格基于检索证据，是否出现编造
- Relevancy（相关性）：回答是否真正回应了用户问题
- Correctness（正确率）：与标准答案语义是否一致

这三个维度解决的是三种不同失败模式：

- 忠实度低：典型幻觉/编造
- 相关性低：答非所问、跑题
- 正确率低：结论不对或关键信息缺失

实践中常见情况是：

- 检索命中了，但忠实度仍然低：说明生成阶段没被约束住
- 忠实度不低但正确率仍不高：说明证据本身不完整或标准答案覆盖更全

### 6. 端到端指标：正确率、兜底率、明显幻觉率

为了让评估对业务更直观，建议再做一些端到端统计：

- 答案正确率：例如 `correctness >= 4` 的占比
- 兜底率：回答包含“抱歉/找不到/没有找到/建议联系客服”等保守话术的比例
- 明显幻觉率：忠实度很低（例如 <=2）的比例

这三项分别对应：

- 系统是否真正可用
- 系统是否过度保守
- 系统是否存在高风险乱编

### 7. Bad Case 必须做归因，不只是列清单

Bad Case 最有价值的输出不是“哪题分低”，而是“为什么低”。

一套可复用的归因框架可以是：

- 路由问题：`intent` 预测与期望不一致
- 检索问题：未命中正确证据（Hit=false）
- 生成问题：命中证据但忠实度偏低（编造/过度推断）
- 知识库问题：命中证据且忠实度不差，但正确率仍不高（资料不完整/过时/切块丢信息）

这样归因的好处是：

- 每个 bad case 都能直接指向下一步优化的层级
- 评估结果能反过来驱动系统迭代，而不是停留在“打分”

### 8. 核心认识

RAG 评估的目标不是“证明系统很强”，而是：

- 用可重复的指标定位问题层次
- 用可解释的 bad case 归因指导改进方向
- 用回归评测保证每次改 Prompt/参数/模型都不会引入退化

只有评估体系建立起来，RAG 才能从“能跑通”走向“可持续优化”。

### 9. 评估与优化小结（从“能评”到“能优化”）

### 9.1 靠感觉优化为什么不行

如果没有评估体系，改完 Prompt、改完检索参数、换完模型之后：

- 不知道效果是变好还是变差
- 很难发现是否引入了回归问题
- 优化会变成“试错 + 玄学”

所以量化评估不是锦上添花，而是系统化优化的前提。

### 9.2 分层评估的目的：定位问题落在哪一层

评估不只看“答案对不对”，更关键是分层定位：

- 检索阶段：证据 chunk 找对了吗
- 生成阶段：答案是否忠实、有无编造
- 端到端：用户是否满意、是否可用

哪一层指标差，就优先优化哪一层。

### 9.3 检索指标怎么选

最常用、最直观的检索指标是：

- Hit Rate：命中了没有
- MRR：排在第几位

如果你的问题经常需要“多个 chunk 共同支撑才能回答”，再考虑：

- Recall / Precision：更适合多证据覆盖类场景

### 9.4 生成指标怎么选

生成阶段的两个核心指标是：

- 忠实度（Faithfulness）：有没有编造
- 相关性（Relevancy）：有没有答非所问

其中“明显幻觉率”可以作为系统级红线指标：

- 幻觉率 = 忠实度 <= 2 的占比

因为 RAG 的核心价值就是基于检索证据回答，如果幻觉率长期偏高，RAG 就失去意义。

### 9.5 评测数据集建设的建议

评估能不能跑起来，最关键的基础是评测数据集：

- 建议从 50~100 条起步
- 覆盖不同类型、不同难度、不同表达方式

提高性价比的常见做法是：

- 大模型辅助生成评测样本
- 人工抽查/校验关键样本（用于保证标注质量）

### 9.6 LLM-as-Judge 如何做自动化

用大模型当评委时，关键不是“调一次模型”，而是把它工程化：

- 设计好评分 Prompt
- 强制 JSON 输出
- 程序解析 JSON 结果

同时要保留“人工校准”机制：

- 定期用人工抽查对齐 judge 的一致性
- 防止 judge 漂移导致评分口径变了

### 9.7 评估驱动优化的闭环

一个有效的优化流程应该是：

- 先用指标定位瓶颈层（路由/检索/生成/知识库）
- 针对性改动（Prompt、检索参数、chunk 策略、rerank 等）
- 重新评测
- 确认“改进且无回归”

评估体系最终的目标是：让每次优化都有证据、有方向、可回归。

## 《AI大模型RAG项目》第 30 小节：SSE 字段速查表（流式输出基础）

SSE（Server-Sent Events）是一种基于 HTTP 的服务端单向推送协议。

它的“协议本体”非常简单：由多行 `key: value` 组成一条事件，事件之间用空行分隔（`\n\n`）。

### 1. SSE 字段速查表

| 字段 | 格式 | 作用 | 大模型 API 是否使用 |
| --- | --- | --- | --- |
| `data:` | `data: 内容` | 事件数据，核心字段 | 是，每个 chunk 都有 |
| `event:` | `event: 类型名` | 自定义事件类型，默认 `message` | 否，通常不使用 |
| `id:` | `id: 事件ID` | 事件标识，支持断线重连 | 否，通常不使用 |
| `retry:` | `retry: 毫秒数` | 指定客户端重连间隔 | 否，通常不使用 |
| `:` | `: 注释内容` | 注释行（常用于心跳保活） | 偶尔使用 |

### 2. 实战理解（为什么你看到的流式输出多是 data）

在大模型流式输出里，最常见的就是不断推送 `data:` 行，因为流式的本质是把“增量文本 chunk”持续发给前端。

而 `event/id/retry` 更多是 SSE 协议提供的通用能力，在大模型 API 的标准流式输出里通常不会默认用到。

### 3. 常见误区

- 误区一：以为 SSE 流式就是 WebSocket
  SSE 仍然是 HTTP 长连接，方向是“服务端 -> 客户端”单向推送。

- 误区二：把普通日志打到 SSE 输出通道
  SSE 输出必须保持协议格式，否则前端解析会失败。

- 误区三：认为 `event/id/retry` 是必须字段
  SSE 最小可用只需要 `data:`，其他字段是可选增强。

## 《AI大模型RAG项目》第 31 小节：长轮询 / SSE / WebSocket 的本质区别与选型

在需要“实时输出”或“流式返回”的系统里，常见的三种实现方式是：

- 长轮询（Long Polling）
- SSE（Server-Sent Events）
- WebSocket

它们看起来都能实现“实时”，但本质差别在于通信方向、协议形态、连接管理方式。

### 1. 三种方案分别是什么

长轮询（Long Polling）：

客户端发起请求，服务端如果暂时没有新数据就先 hold 住不返回，等有新数据再返回。客户端收到响应后立刻发起下一次请求。

本质上仍然是“一问一答”，只是每次答复的等待时间变长了。

SSE（Server-Sent Events）：

客户端发起一次 HTTP 请求，服务端保持连接，并持续向客户端单向推送数据。

它基于 HTTP，方向是“服务端 -> 客户端”单向推送。

WebSocket：

客户端与服务端通过一次 HTTP 握手升级为 WebSocket 协议，之后双方可以随时双向发送消息。

它是独立于 HTTP 的全双工协议（`ws://` / `wss://`）。

### 2. 详细对比表

| 对比维度 | 长轮询 | SSE | WebSocket |
| --- | --- | --- | --- |
| 通信方向 | 单向（模拟） | 单向（服务端 -> 客户端） | 双向 |
| 底层协议 | HTTP | HTTP | 独立协议（`ws://`） |
| 连接方式 | 每次都是新连接 | 一个长连接 | 一个长连接 |
| 自动重连 | 不支持 | 规范内置 | 需自己实现 |
| 浏览器原生支持 | 需手写轮询逻辑 | `EventSource` API | `WebSocket` API |
| Nginx/CDN 兼容性 | 天然兼容 | 天然兼容（HTTP） | 通常需要额外配置代理 |
| 实现复杂度 | 低 | 低 | 中 |
| 适用场景 | 兼容性要求高的老系统 | 服务端单向推送 | 双向实时通信 |

### 3. 在 RAG / 大模型场景里怎么选

如果目标是“服务端把模型输出一点点推给前端”（典型流式生成），通常优先考虑：

- SSE：最贴合“单向流式输出”，前端也比较好接

如果你的场景是“需要双向实时交互”（例如前端也频繁推事件给服务端、或者需要更复杂的会话状态同步），再考虑：

- WebSocket

如果你受限于某些环境或框架、必须保持最朴素的 HTTP 语义，或者对兼容性要求非常高，则：

- 长轮询也能做，但会有更多请求开销

### 4. 核心认识

三者最关键的本质区别可以一句话记住：

- 长轮询：还是 HTTP 请求，只是“等久点再回”
- SSE：一次 HTTP 长连接，服务端单向持续推
- WebSocket：升级成独立协议，双向随时通信

### 5. 常见误区

- 误区一：以为三者都一样，只是写法不同
  实际上它们的协议层形态不同，直接影响扩展性、成本和运维复杂度。

- 误区二：用 WebSocket 做纯单向推送
  能做但不划算，SSE 往往更简单、更符合需求。

- 误区三：忽略网关/代理层的配置差异
  SSE/长轮询更接近普通 HTTP，WebSocket 往往需要额外的代理配置与心跳策略。

## 《AI大模型RAG项目》第 32 小节：为什么大模型 API 更常选 SSE

在“流式输出”这个需求上，大模型 API（Chat Completions / Responses 的 streaming）普遍更倾向于 SSE，而不是 WebSocket 或长轮询。

这不是因为 SSE 更“高级”，而是它和大模型的输出形态天然匹配。

### 1. 方向匹配：大模型输出是典型的单向推送

大模型在流式生成时，本质上是：

- 服务端不断生成 token/chunk
- 客户端不断接收并渲染

而客户端在这条连接上通常不需要再持续把数据“反向推回服务端”。

因为用户的问题在第一次 HTTP 请求时就已经发完了。

所以 SSE 的“服务端 -> 客户端单向推送”刚好符合需求；WebSocket 的全双工能力在这个场景里反而用不上。

### 2. HTTP 原生：基础设施更友好

SSE 仍然是普通 HTTP 响应，只是 `Content-Type` 变成 `text/event-stream`，并按事件流格式持续输出。

这带来的好处是：

- Nginx/CDN/负载均衡/API 网关等 HTTP 基础设施天然支持
- 通常不需要额外配置

而 WebSocket 往往需要代理层支持协议升级，并可能需要额外配置（例如 Upgrade 相关 header），在某些企业网络环境里也更可能被拦截。

### 3. 自动重连：协议层自带能力

SSE 规范内置了重连机制，常见组合是：

- `Last-Event-ID`
- `retry:`

浏览器侧的 `EventSource` API 也会自动处理重连。

相比之下，WebSocket 断线后的重连策略通常需要你自己实现。

### 4. 实现简单：服务端输出文本流即可

SSE 的实现重点在于：

- 服务端按 SSE 事件格式输出文本
- 客户端按 SSE 事件解析并渲染

而 WebSocket 有自己的帧协议（frame），并且通常还需要处理：

- ping/pong 心跳
- 连接关闭与异常恢复
- 消息帧的编码解码

所以在“只需要服务端持续推送”的场景下，SSE 的实现复杂度通常更低。

### 5. 一个简单判断标准

可以用一句话快速选型：

- 如果你的场景是“服务端持续向客户端推送数据”，优先 SSE
- 只有当客户端也需要实时向服务端持续发送数据时，才更需要 WebSocket

例如：

- 大模型流式输出：SSE 优先
- 在线协作编辑、实时游戏、聊天室里双方频繁互发事件：WebSocket 更合适

### 6. 核心认识

大模型 API 选择 SSE 的本质原因不是“流式一定要 SSE”，而是：

- 单向推送匹配
- HTTP 基础设施友好
- 自动重连更省心
- 实现成本更低

这让 SSE 成为大模型流式输出的默认最优解。

## 《AI大模型RAG项目》第 33 小节：SSE 小结（从能用到懂原理）

这一节把 SSE 从“能跑通”提升到“能解释、能排障、能做稳”的程度。核心可以浓缩成下面几个结论。

### 1. SSE 协议完整规范要知道什么

SSE 常见字段包括：

- `data:`：事件数据，核心字段
- `event:`：事件类型名（可选）
- `id:`：事件 ID（可选，可用于断线重连）
- `retry:`：重连间隔（可选）
- `:`：注释行（常用于心跳保活）

大模型 API 通常主要使用 `data:`，但理解完整规范能帮助你：

- 更好排查流式输出为什么断、为什么卡、为什么粘包
- 更容易自建/转发一个规范的 SSE 服务

### 2. SSE vs WebSocket：为什么大模型 API 常用 SSE

大模型流式输出的典型形态是：

- 服务端持续生成并单向推送 token/chunk
- 客户端持续接收并渲染

所以 SSE 的优势在于：

- 方向匹配：单向推送刚好够用
- HTTP 原生：Nginx/CDN/网关更友好，通常无需额外升级配置
- 实现简单：按文本流输出事件即可

而 WebSocket 的全双工能力在纯“单向输出”场景中常常用不上，还会引入额外运维与实现复杂度。

### 3. 流式响应数据结构要抓住哪些规律

在大模型 SSE 流里，常见规律是：

- `delta` 是增量内容，会分 chunk 多次出现
- `finish_reason` 往往只在最后一个 chunk 出现
- `usage`（token 统计）存在“平台差异”：有的平台只在最后返回，有的平台会在流中带出，有的平台需要非流式才能拿全

所以做流式解析时要避免假设：

- “每个 chunk 都有 content”
- “每个 chunk 都有 finish_reason”
- “usage 一定会出现且结构固定”

### 4. 生产级 SSE 客户端需要补齐哪些工程要点

只要开始做真实系统，就要把“能读到内容”升级成“能稳定读、能恢复、能定位问题”：

- 回调接口：能把 `delta/usage/complete/error` 事件接入业务逻辑，而不是只 `System.out.print`
- 超时控制：服务端卡住不发数据时，客户端不能无限阻塞（常见 30~60 秒级别策略）
- JSON 容错：坏 chunk、空 delta、缺字段不应直接把整条流炸掉
- 错误处理不丢内容：连接中断时至少保留已收到内容，并能区分正常结束与异常 EOF

### 5. 一个关键视角：你既是 SSE 客户端，也是 SSE 服务端

实际项目里，Java 后端往往同时扮演两个角色：

- 作为调用方：它是“大模型 API 的 SSE 客户端”，消费上游流式输出
- 作为转发方：它又是“前端浏览器的 SSE 服务端”，把增量结果再推给前端

因此你需要做的不仅是“消费 SSE”，还要“把 SSE 正确地生产/转发出去”。

### 6. 预警：代理缓冲会让流式看起来像“不流”

一个常见的坑是：即使后端 SSE 实现完全正确，如果中间反向代理（例如 Nginx、网关）开启了响应缓冲，
前端看到的可能不是逐 token 输出，而是“攒一会儿一次性吐出来”。

遇到这种情况，优先排查链路上的：

- 代理是否开启 buffering
- 是否正确设置 `text/event-stream`
- 是否有额外日志/输出污染了 SSE 通道

### 7. SSE 协议视角：四个字段 + 注释心跳

从协议规范角度，SSE 的关键元素可以压缩成“四个字段 + 一种特殊行”：

- `data:`：事件数据（最核心）
- `event:`：事件类型（可选，但对前端按类型分发非常有用）
- `id:`：事件 ID（可选，断线重连时可配合 `Last-Event-ID` 做续传语义）
- `retry:`：建议的重连间隔（可选）
- `:`：注释行（常用于心跳保活，让链路不断、代理不回收连接）

### 8. 为什么大模型流式输出常选 SSE（本质原因）

大模型流式输出的形态几乎都是“服务端持续生成，客户端持续接收”，这是典型的单向推送。

因此 SSE 的优势在于：

- 单向推送匹配：不需要全双工也能很好完成“边生成边展示”
- HTTP 原生：更容易穿过 Nginx/CDN/网关/企业代理，部署改动更少
- 客户端简单：浏览器有 `EventSource`，服务端按文本流持续输出即可

### 9. 流式响应数据要点：delta / finish_reason / usage 的常见规律

对大模型 SSE 流来说，最关键不是“能读到内容”，而是要正确理解这三类字段在时间维度上的出现规律：

- `delta`：增量内容，会出现多次，且可能出现空 delta（不要把空 delta 当异常）
- `finish_reason`：通常只在最后出现（不要假设每个 chunk 都有）
- `usage`（token 统计）：平台差异很大，常见三种形态
- 只在最后一个 chunk 返回 usage
- 流中就持续带 usage（可能每 N 个 chunk 才出现一次）
- 只有非流式请求才给完整 usage（流式不给或不给全）

因此解析端的正确姿势是：

- 用“累加器”拼接 delta，而不是期待一次性 content
- finish_reason 缺失时不当成失败
- usage 作为可选信息，出现就更新，不出现就容忍

### 10. 健壮的 SSE 客户端应该补齐的能力清单（从能跑到能稳）

只要进入真实业务场景，就需要把“读到 token”升级成“稳定、可恢复、可观测”：

- 回调接口：把 `delta/usage/done/error` 事件做成明确回调，而不是在循环里直接打印
- 超时控制：常见 30~60 秒级策略，避免上游无输出导致无限阻塞
- JSON 解析容错：坏 chunk 只影响当前块，不影响整条流
- 空 delta 兼容：把它当成“无内容更新”，不要当成协议错误
- 错误处理不丢内容：断线时尽量保留已收到的内容，并给出可解释的错误信息

### 11. 服务端实战视角：为什么需要“后端中间层”

很多系统会选择“前端不直连大模型”，而是让后端做中转层，原因通常不是技术炫技，而是工程约束：

- API Key 安全：避免密钥暴露到浏览器
- 跨域限制：统一由后端处理 CORS、鉴权与访问控制
- 业务逻辑：在中转层做意图路由、RAG 检索、日志审计、风控限流、敏感信息处理

### 12. Spring Boot `SseEmitter` 的核心机制（线程模型与生命周期）

`SseEmitter` 的关键点是它把“建立连接”和“持续推送”解耦开了：

- Controller 线程可以很快返回（连接建立完成）
- 推送由独立线程/线程池异步执行（持续发送事件）
- 生命周期回调用于做超时/异常/结束的资源清理
- 常见回调：`onTimeout/onError/onCompletion`

### 13. SSE 服务端的三层实现路线（从最简到生产）

把服务端实现拆成三层，有助于你按阶段推进：

- 最简版：理解 `SseEmitter` 的基本套路（建连、send、complete）
- 完整版：接入大模型流式转发（上游 SSE -> 解析 -> 下游 SSE）
- 生产版：补齐心跳保活、断连检测、连接/线程池管理、超时策略与监控告警

### 14. 事件类型设计：token/done/error（以及一个常见扩展 usage）

事件类型设计的目标是让前端消费“稳定且可组合”：

- token（或 delta）：内容增量
- done（或 complete）：流结束信号 + 汇总信息
- error：错误通知（建议结构化而不是纯字符串）
- usage：token 统计（可选扩展，出现就展示/落库）

### 15. 前端对接：EventSource 与 fetch + ReadableStream

常见两种方式：

- `EventSource`：最符合 SSE 语义，自动重连，前端代码简单
- `fetch + ReadableStream`：可做更细粒度控制（自定义 header、鉴权、手动解析），但实现更复杂

### 16. Nginx 配置要点：关闭 `proxy_buffering` 是关键

只要链路中有反向代理，是否“真正实时”常常不取决于代码，而取决于代理是否缓冲：

- 不关闭 `proxy_buffering`，浏览器可能看到“攒一会儿再吐”，表现像“不流”
- 所以排障时要把“代理缓冲”当第一优先级检查项

### 17. 生产环境关注点（避免上线后才暴露）

和普通 HTTP 接口相比，SSE 在生产环境更容易踩坑的点主要集中在资源与链路稳定性：

- 连接数控制：SSE 是长连接，必须有上限与隔离策略
- 线程池配置：推送线程与业务线程隔离，避免互相拖死
- 心跳保活与断连检测：防止中间层回收连接或出现僵尸连接
- 内存泄漏防范：缓存 fullContent、会话上下文、订阅者列表等都要可控
- 负载均衡注意：如果需要粘性会话（sticky session），要提前评估与配置

## 《AI大模型RAG项目》第 34 小节：JSON-RPC 2.0 调用流程与消息规则小结

这一节可以看作 MCP 通信底层消息规范的收束总结。

如果说 MCP 解决的是“模型与外部能力如何标准化协作”，那么 JSON-RPC 2.0 解决的就是：

- 一次远程调用该怎么表示
- 成功与失败该怎么返回
- 通知类消息什么时候不需要响应
- 批量请求该怎么组合和返回

### 1. JSON-RPC 2.0 的本质

JSON-RPC 2.0 本质上是一套远程调用的消息格式规范。

它使用 JSON 来描述一次调用，但它只规定这些消息应该怎样表达：

- Request
- Response
- Notification
- Error
- Batch

它不绑定具体传输层，也不负责鉴权、服务发现、限流这些工程能力。

所以可以把它理解成：

> 用一套固定字段，把“调用哪个方法、传什么参数、成功返回什么、失败返回什么”表达清楚。

### 2. 一次标准调用由什么组成

最标准的一次 JSON-RPC 调用，本质上是：

- 客户端发送一个 `Request`
- 服务端返回一个 `Response`

其中最关键的几个字段是：

- `jsonrpc: "2.0"`：协议版本标识
- `method`：要调用的方法名
- `params`：调用参数
- `id`：请求标识，用于把响应和请求对应起来

而 `Response` 有一个非常重要的规则：

- `result` 和 `error` 必须二选一

不能同时出现，也不能两个都缺失。

### 3. 五种典型流程怎么理解

![cebcb1de-3872-4021-bcc7-74112370b7ce](D:\Wechat\xwechat_files\wxid_j2nnpd3x4n3122_4f70\temp\InputTemp\cebcb1de-3872-4021-bcc7-74112370b7ce.png)

把整个流程图压缩后，可以记成下面 5 种典型情况。

普通 Request（成功）：

- 客户端发送带 `id` 的请求
- 服务端先校验 JSON 和协议格式
- 再按 `method` 路由到对应业务处理器
- 业务成功后返回带 `result` 的响应
- 响应中的 `id` 与请求保持一致

普通 Request（失败）：

- 请求格式是合法的
- 但业务执行过程中抛异常或返回失败
- 服务端返回带 `error` 的响应
- 响应中的 `id` 仍与原请求一致

Notification：

- 本质上也是 Request
- 但因为没有 `id`
- 所以服务端只执行，不返回 JSON-RPC Response

协议错误：

- JSON 解析失败，或消息结构不符合 JSON-RPC 2.0 要求
- 这类错误属于协议层，还没进入业务执行
- 服务端会返回 Parse error / Invalid Request
- 此时 `id` 通常为 `null`

Batch Request：

- 客户端发送一个数组，里面可以混合普通 Request 和 Notification
- 服务端逐个处理数组元素，顺序处理或并发处理取决于实现
- 只有带 `id` 的元素才会进入最终响应数组
- 响应顺序不一定和请求顺序一致，真正的对应关系靠 `id`

### 4. Notification 为什么最容易被误解

Notification 很容易被误以为是“另一种协议消息”。

其实它本质上还是 Request，只是因为没有 `id`，所以客户端不期待服务端回包。

因此它的核心不是“长得不同”，而是“语义不同”：

- 普通 Request：执行后要回响应
- Notification：执行后不回响应

这一点在 MCP 场景里尤其重要，因为很多初始化、状态同步、事件推送类消息，本来就不要求返回业务结果。

### 5. Batch 的核心不是“打包”，而是“按元素决定是否响应”

Batch 最容易被误解成“发了一个数组，就一定回一个数组”。

实际上更准确的理解是：

- Batch 只是把多个调用放在一个数组里一起发
- 每个元素是否需要响应，仍然取决于它自己有没有 `id`

所以：

- 混合 Batch：只返回那些带 `id` 的结果
- 全 Notification Batch：可能根本不返回 JSON-RPC 响应

### 6. 协议错误和业务错误一定要分开看

这是实现时最容易混淆的一组概念。

协议错误解决的是“这条消息能不能被识别为合法 JSON-RPC 调用”。

例如：

- JSON 根本解析失败
- 缺少 `jsonrpc: "2.0"`
- `method` 类型错误
- 顶层结构不合法

业务错误解决的是“请求是合法的，但业务执行失败了”。

例如：

- 方法存在，但内部抛异常
- 参数语义正确，但业务校验不通过
- 数据库、外部接口、工具调用执行失败

因此不能把二者混为一谈，更不能把协议保留错误码当成普通业务异常码随便使用。

### 7. 核心规则可以记成这几句

- 一次标准调用由 Request 和 Response 构成
- Notification 本质上也是 Request，只是没有 `id`
- `result` 和 `error` 必须二选一
- Batch 只是把多个消息打包发送，不改变单条消息是否需要响应的规则
- 协议内置错误码处理的是协议层问题，不等于业务异常码

### 8. 实现视角下最重要的一句话

如果从编码实现角度去看 JSON-RPC 2.0，可以把它理解成：

> 先校验消息是不是合法协议对象，再按 `method` 路由到业务处理器，最后把执行结果重新包装成标准 JSON-RPC 响应。

这就是为什么一个完整服务端往往会天然分成几层：

- Transport 层：负责把消息送进来送出去
- Protocol 层：负责解析 JSON、校验字段、组装标准响应
- Router/Dispatcher 层：负责 `method -> handler`
- Business Handler 层：负责真正执行业务逻辑

### 9. 最容易踩的坑

实际编码时，最常见的问题通常不是“大方向不懂”，而是这些细节没处理严谨：

- 忘记带上 `jsonrpc: "2.0"`，导致消息不符合协议格式
- 把 Notification 当成普通请求处理，错误地返回了 Response
- 响应结构写错，让 `result` 和 `error` 同时出现，或者两者都缺失
- `method`、`params` 类型不符合规范，导致请求无法正确解析
- 使用命名参数时，字段名与服务端约定不一致，尤其容易出现在大小写或命名风格上
- Batch 请求里全部都是 Notification，却仍然错误返回一个空数组 `[]`
- 把业务失败误写成协议错误，比如滥用保留错误码

### 10. 和 MCP 的关系怎么理解

在 MCP 语境里，JSON-RPC 2.0 更像底层消息骨架。

也就是说：

- MCP 规定“客户端和服务端围绕 Tool / Resource / Prompt 怎么协作”
- JSON-RPC 2.0 规定“这些协作消息在协议层长什么样”

所以理解 JSON-RPC 2.0，不是为了背字段，而是为了在调试 MCP 时能快速分清：

- 是传输层没通
- 是协议层格式不对
- 是 method 路由没对上
- 还是业务处理器本身执行失败

### 11. 为什么 MCP 选择 JSON-RPC 2.0，而不是直接用 HTTP 或 gRPC

这个问题本质上不是“谁更强”，而是“谁更适合作为 MCP 这种跨语言、跨环境、强调互操作的协议基础”。

先把三者的定位分开：

- HTTP 更像传输协议
- gRPC 更像完整 RPC 框架
- JSON-RPC 2.0 更像消息格式规范

也就是说：

- HTTP 负责“怎么传”
- JSON-RPC 2.0 负责“消息长什么样”
- gRPC 同时把传输、接口定义、序列化、代码生成一起打包了

可以先用一张表把三者的差异压缩看清：

| 维度 | HTTP | gRPC | JSON-RPC 2.0 |
| --- | --- | --- | --- |
| 定位 | 传输协议 | 完整 RPC 框架 | 消息格式规范 |
| 消息格式标准化 | 无统一标准 | 通过 Protocol Buffers 统一 | 通过固定 JSON 结构统一 |
| 实现复杂度 | 需要自己设计消息格式 | 需要 `.proto` 和代码生成 | 只需解析 JSON |
| 人类可读性 | 取决于具体设计 | 较弱，通常是二进制 | 强，纯文本 JSON |
| 传输层灵活性 | 主要依赖 HTTP | 通常强依赖 HTTP/2 | 传输层无关 |
| 通知消息支持 | 需要自行设计 | 一般通过流式 API 变通 | 原生支持 Notification |
| 批量调用支持 | 需要自行设计 | 一般通过流式或自定义协议处理 | 原生支持 Batch |
| 类型安全 | 弱 | 强 | 弱，通常需要额外 Schema 约束 |
| 性能 | 取决于实现 | 高 | 中等，存在 JSON 序列化开销 |

### 12. 为什么不直接用 HTTP

HTTP 本身当然能承载 MCP，但 HTTP 只解决“请求怎么发、响应怎么回”，并没有替你统一规定一套远程调用消息格式。

HTTP 作为传输层协议，真正定义的是这些事情：

- 如何建立连接，例如 TCP / TLS
- 如何发送请求，例如 GET / POST / PUT / DELETE
- 如何返回响应，例如状态码、Header、Body

但 HTTP 不会替你回答这些“远程调用协议层”的问题：

- 方法名怎么表示，是放在 URL 路径里，还是放在 Body 里
- 参数怎么传，是 Query String、JSON Body，还是 Form Data
- 错误怎么表达，是靠 HTTP 状态码，还是 Body 里的错误对象
- 通知消息这种“不需要响应的调用”要怎么表示
- 批量调用要怎么组织、怎么返回

所以 HTTP 的问题不是不能用，而是：

> 它太底层了，MCP 如果只建立在 HTTP 上，不同实现之间很容易各写各的，最后缺少统一消息标准。

这也是为什么基于 HTTP 的 API 风格会出现明显差异：

- RESTful API：通常用 URL + HTTP 方法表达语义
- RPC-style API：通常用一个固定入口，再在 Body 里放 `method` 和 `params`
- GraphQL：又会走自己的 Query DSL

所以只说“基于 HTTP”是不够的，因为 HTTP 本身并不等于统一调用协议。

### 13. 为什么不直接用 gRPC

gRPC 的优势很明显：

- 强类型
- 高性能
- 基于 `.proto` 统一接口定义
- 代码生成成熟

但它不适合 MCP 这类协议作为“默认共同语言”的主要原因在于：

- 需要预先定义 `.proto` 文件，每个能力都要先写 Protocol Buffers 接口定义
- 强依赖代码生成，客户端和服务端通常都要生成代码，动态调用不够方便
- 二进制协议不如纯 JSON 直观，不能像 JSON-RPC 那样直接肉眼看懂请求响应
- 通常依赖 HTTP/2 生态，在某些宿主环境、代理环境里不如纯文本 JSON 灵活

换句话说，gRPC 的问题不是“不好”，而是它更像一套强类型、高性能、工程化很重的完整 RPC 框架。

所以 gRPC 更适合：

- 强类型内部服务调用
- 高性能微服务通信

但 MCP 更在意的是：

- 不同客户端和服务端之间容易互通
- 人能直接看懂请求和响应
- 调试和排障足够直接
- 在 STDIO、HTTP、WebSocket 等不同通道上都容易落地

### 14. JSON-RPC 2.0 为什么刚好合适

JSON-RPC 2.0 其实是在几种需求之间做了平衡：

- 标准化：统一了 Request / Response / Notification / Error / Batch 的表达方式
- 简单性：只需要解析 JSON，不要求额外 IDL 和代码生成链路
- 可读性：纯文本 JSON，调试和抓包都很直观
- 传输层无关：既可以跑在 HTTP 上，也可以跑在 WebSocket、TCP、STDIO 上
- 原生支持通知与批量：这些正好很适合协议交互场景

它的定位和 HTTP、gRPC 都不一样，它真正只定义“消息该长什么样”。

例如它会规定：

- Request 结构：`{"jsonrpc":"2.0","method":"...","params":{...},"id":1}`
- 成功响应结构：`{"jsonrpc":"2.0","result":{...},"id":1}`
- 错误响应结构：`{"jsonrpc":"2.0","error":{...},"id":1}`
- Notification 结构：`{"jsonrpc":"2.0","method":"...","params":{...}}`

同时它不绑定传输层，所以它可以跑在：

- HTTP 之上
- WebSocket 之上
- STDIO 之上
- 任何能稳定传输 JSON 文本的通道之上

它不是最强类型的，也不是性能最高的，但它非常适合做“跨实现之间的共同协议语言”。

### 15. 这个选择背后的核心权衡

可以把三者的取舍压缩成一句话：

- HTTP 太自由，缺少统一消息规范
- gRPC 太重，更适合强类型高性能内部 RPC
- JSON-RPC 2.0 刚好处在“标准化、简单性、灵活性”的平衡点上

所以对 MCP 来说，真正重要的不是追求极致性能，也不是追求最大自由度，而是：

> 让不同实现之间能够无缝通信、容易调试、快速迭代。

这正是 JSON-RPC 2.0 最适合 MCP 的原因。

### 16. JSON-RPC 2.0 的边界：它并不包办一切

理解 JSON-RPC 2.0 时，一个非常重要的认识是：

> 它解决的是“消息格式如何标准化表达”，不是“分布式系统里的所有问题”。

所以它适合做协议骨架，但它天然也有边界。

### 17. 缺陷一：它不包含传输层细节

JSON-RPC 2.0 只定义消息格式，不负责这些事情：

- 如何建立连接，是 TCP、HTTP、WebSocket 还是 STDIO
- 如何处理超时与重试
- 如何做负载均衡
- 如何做服务发现

这些都需要在 MCP 的实现层或基础设施层补齐。

所以它的优点是“传输层无关”，它的另一面也是：

- 传输层能力需要自己选
- 连接管理与可靠性策略需要额外设计

### 18. 缺陷二：它不包含鉴权机制

JSON-RPC 2.0 不定义如何鉴权，也不规定令牌、签名、会话这些安全机制怎么放。

这意味着：

- 如果跑在 HTTP 上，通常要靠 Header、Cookie、Bearer Token 等方式补上
- 如果跑在应用层，也可能在 `params` 或上下文字段里约定认证信息

所以 JSON-RPC 2.0 本身不是“不安全”，而是：

> 安全机制不在它的职责范围里，需要宿主系统自己补。

### 19. 缺陷三：它不包含强类型定义

JSON-RPC 2.0 的参数和返回值本质上都是动态 JSON，它不会像 gRPC 那样强制类型检查。

这会带来几个直接后果：

- 参数类型校验主要发生在运行时
- 编译期无法提前发现很多类型错误
- 跨实现协作时，必须额外约定字段结构

这也是为什么在真实系统里，往往还需要补一层 Schema 定义，例如 JSON Schema。

MCP 正是在协议层额外引入 Schema 来弥补这一点。

例如一个工具会额外声明 `inputSchema`：

```json
{
  "name": "get_weather",
  "description": "获取天气信息",
  "inputSchema": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称"
      }
    },
    "required": ["city"]
  }
}
```

这说明 JSON-RPC 2.0 负责“调用消息怎么传”，而 MCP 会继续补“这个调用期望什么输入结构”。

### 20. 缺陷四：性能不如二进制协议

JSON 是文本格式，所以序列化和反序列化开销通常大于 Protocol Buffers 这类二进制协议。

因此和 gRPC 相比，JSON-RPC 2.0 往往在这些方面不占优势：

- 消息体更大
- 编解码开销更高
- 极限吞吐性能通常不如二进制协议

不过这类缺点要结合场景看。

对于 MCP 这种场景，重点通常是：

- 工具调用
- 资源读取
- 协议互通
- 调试便利

而不是超高频、超低延迟的内部微服务调用。

所以它的性能缺点通常是“存在，但可以接受”。

### 21. 这一部分最关键的结论

可以把 JSON-RPC 2.0 的优点和缺点一起压缩成一句话：

> 它非常适合做跨实现的统一消息协议，但它并不负责传输、鉴权、强类型和极致性能，这些都需要上层协议或具体实现继续补齐。

这也是为什么在 MCP 体系里，经常会看到这样的分层：

- JSON-RPC 2.0 负责消息骨架
- MCP 负责能力语义与 Schema 约束
- HTTP / WebSocket / STDIO 等负责传输通道
- 具体宿主系统负责鉴权、超时、重试、日志、监控等工程能力
