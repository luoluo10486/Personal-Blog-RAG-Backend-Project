## 第一章 实习经历

### 1.1 项目总览

**问：这个广东省高质量数据集创新大赛平台，整体是在解决什么问题？**  
**答：** 它解决的是“赛事运营全流程数字化”的问题。`cde-gddatafair` 不是单独一个页面或者单独一个接口，而是把赛事展示、在线报名、作品提交、专家评审、定时导出、通知推送这些原本分散的动作串成一条稳定链路。  
如果从模块角度拆开看，它大致分成 `submission`、`user`、`admin`、`job` 和 `core` 几块：`submission` 负责报名和上传，`user` 负责参赛和评审相关能力，`admin` 负责后台导出和运营处理，`job` 负责定时任务，`core` 放公共工具和通用模型。  
我在实习里主要是围绕这几个模块做需求梳理、接口开发、表结构配合和联调测试。它不是那种“写完就结束”的 CRUD，而是要把报名、导出、评审和附件这些链路一起想清楚，保证高峰期也能稳定跑。

**追问：你在这里更偏业务还是偏基础设施？**  
**答：** 更偏业务落地，但里面包含了不少工程能力。比如报名入口的限流和幂等、短信验证码的 Redis 过期控制、导出的异步化和 ZIP 打包、评审附件的自动绑定与权限校验，这些都不是单纯堆业务字段。  
如果面试官继续追，我会把自己的定位说得更准一点：我不是只做一个功能页，而是在报名、导出和评审这几条最容易出问题的链路上做“能上线、能扛峰值、能排障”的工程处理。

**可替代说法：**  
如果面试官想听得更短，你就说：我主要负责 `cde-gddatafair` 里的报名、导出、评审和部分后台交付能力，重点是把高峰报名、批量导出和附件绑定这些最容易出问题的链路做稳。

### 1.2 报名模块

**问：报名模块为什么要拆表，而不是把所有字段塞进一张大表？**  
**答：** 因为报名场景本质上是“一个报名主体带多个从属实体”。主表负责核心报名状态，组织机构、团队成员、附件、赛道/赛题选择这些信息如果全塞一张表，后面会很难维护，也不利于查询、扩展和统计。  
我在设计时按“谁是主记录、谁是重复子项、谁是可复用附件”来拆分，这样后续做评审、导出、同步的时候，数据关系会清楚很多。再往细里说，拆表的好处不只是少字段，而是能把不同生命周期的对象拆开：主记录负责状态流转，成员负责结构化人员信息，附件负责 OSS 关联，这样任何一块出问题都能单独排查。

**问：你怎么处理重复提交和高并发报名？**  
**答：** 我用了两层保护。第一层是接口限流，限制短时间内同一用户或同一 IP 的请求频率，避免入口被打爆。第二层是业务幂等，防止重复点提交、重复刷新、重复请求把同一份报名数据写多次。  
代码里对应的是 `RateLimiter` 和 `RepeatSubmit`，你可以把它解释成“限流解决快，幂等解决重”，一个管入口流量，一个管业务唯一性。再进一步讲，限流是把洪峰挡在门外，幂等是把重复写入挡在事务前，这两个问题不是同一层面的，所以不能互相替代。  
关键代码可以看 [BopdaRegistrationController.java](D:/develop/Personal-Blog-RAG-Backend-Project/cde-base/cde-modules/cde-gddatafair/cde-gddatafair-submission/src/main/java/com/cde/gddatafair/submission/controller/BopdaRegistrationController.java)。

**追问：为什么不只靠数据库唯一索引？**  
**答：** 唯一索引只能兜住最终写入的一部分场景，挡不住前面的重复请求、重复日志、重复短信、重复业务判断。更稳的做法是“接口层限流 + 业务层幂等 + 数据层唯一约束”三层一起上。  
如果只靠数据库，用户体验会比较差，系统高峰时也更容易把压力往后面放大。你可以把它说成一个完整链路：前面先减少请求数量，中间减少重复操作，最后再用数据库兜底，这样才是生产环境里比较完整的保护方式。

**问：短信验证码你是怎么做的？**  
**答：** 流程是先校验图形验证码，再生成短信验证码并写入 Redis，设置过期时间，然后调用短信服务发送。发送成功和失败都会记录日志，方便排查“验证码没发出去”或者“用户没收到”的问题。  
验证码放 Redis 的原因也很简单：它是短生命周期、高频读写、天然需要 TTL 的状态，Redis 比较合适。它不是永久凭证，更像一次性票据，所以就应该在过期时间、查询速度和可回收性上都更友好。

**问：附件上传为什么限制单次最多 10 个？**  
**答：** 这是一个工程阈值，不是业务死规定。报名场景下附件数量通常不会特别多，限制单次上传数量可以防止一次请求把 IO、内存和对象存储写入同时打满。  
后端还会再做文件非空、文件类型和存储校验，所以这不是单纯前端限制，而是完整安全边界。限制数量的目的不是“少传几个文件”，而是控制请求体大小和失败排查成本。

### 1.3 导出与定时推送

**问：导出为什么不直接同步返回？**  
**答：** 因为报名数据量一大，同步导出会长时间占着请求线程，用户体验也差。更合理的做法是把导出拆成异步任务，任务参数通过 SnailJob 统一调度，执行时再按条件筛数据、分页读取、流式写出。  
你可以理解成：同步接口只负责“点一下就触发”，真正的重活交给后台任务去跑。这样还有一个好处是以后扩展导出逻辑会更轻松，比如全量导出、条件导出、按赛道拆分导出、发邮件通知，都可以沿着同一套任务参数继续往下演化。

**问：你们为什么要生成临时文件和 ZIP？**  
**答：** 因为导出的结果不一定只有一个文件。全量导出可能是一个 Excel，赛道-赛题拆分导出会变成多个 Excel，带附件时还会生成目录结构。先落临时目录，再统一打成 ZIP，结构最清晰，也最适合发邮件或者直接下载。  
关键代码可以看 [BopdaRegistrationAdminApplicationService.java](D:/develop/Personal-Blog-RAG-Backend-Project/cde-base/cde-modules/cde-gddatafair/cde-gddatafair-admin/src/main/java/com/cde/gddatafair/admin/application/BopdaRegistrationAdminApplicationService.java) 和 [ZipExportWriter.java](D:/develop/Personal-Blog-RAG-Backend-Project/cde-base/cde-modules/cde-gddatafair/cde-gddatafair-admin/src/main/java/com/cde/gddatafair/admin/support/ZipExportWriter.java)。临时目录是为了避免把所有文件一次性堆在内存里，ZIP 是为了把多个导出结果和附件打成一个可交付单元。

**问：SnailJob 里任务参数你是怎么设计的？**  
**答：** 我把参数拆成两部分：一部分是控制参数，比如是否全量导出、是否发邮件；另一部分是筛选参数，比如时间范围、赛道、赛题、组织机构。这样任务调度侧只要拿到一个 JSON，就能复现完整导出逻辑。  
对应的解析集中在 [RegistrationExportJobHelper.java](D:/develop/Personal-Blog-RAG-Backend-Project/cde-base/cde-modules/cde-gddatafair/cde-gddatafair-job/src/main/java/com/cde/gddatafair/job/RegistrationExportJobHelper.java)。这里最重要的不是“把 JSON 解析出来”，而是把任务参数和业务筛选条件分层，这样调度层和业务层职责才不会混。

**追问：为什么导出后还要异步发邮件？**  
**答：** 因为邮件发送不稳定，可能受 SMTP、网络和附件大小影响，不能阻塞导出主流程。我的做法是导出完成后再异步通知，导出本身成功就算成功，邮件失败单独记录和重试。  
这个思路很重要，因为它把“主任务”和“通知副作用”拆开了。导出链路的核心是把数据交出去，邮件只是通知手段，所以通知失败不应该回滚主任务。

### 1.4 初赛评审子系统

**问：评审子系统核心在解决什么？**  
**答：** 核心是“把评审员看到的内容和他本该看到的内容严格对齐”。评审员进入系统后，只能看到自己权限范围内的赛道、赛题和作品详情，不能越权查看别的内容。  
同时它还要支持作品打分、附件查看、历史记录回显和结果管理，这些都在 [TestIrReviewServiceImpl.java](D:/develop/Personal-Blog-RAG-Backend-Project/cde-base/cde-modules/cde-gddatafair/cde-gddatafair-user/src/main/java/com/cde/gddatafair/user/service/impl/TestIrReviewServiceImpl.java) 里。再往细里讲，这其实是“权限校验 + 业务加载 + 附件聚合”三件事一起做，不能只看某个列表接口有没有数据。

**问：你怎么防止重复评审提交？**  
**答：** 提交前先查 `registrationId + selectedTopic + reviewerId` 是否已经有有效记录，如果已经有，就直接拦住，不允许重复打分。  
这个判断比前端禁用按钮更可靠，因为前端只能防体验问题，不能当最终防线。重复提交在评审系统里不是“多点了一次”这么简单，它可能会影响成绩唯一性、结果公示和后续追责，所以必须在服务端拦住。

**问：附件自动绑定是怎么做的？**  
**答：** 这是我觉得最“像实战”的一块。线上和线下附件经常不一致，所以我做了一个批量绑定接口：先解压 ZIP，读取里面的 Excel 汇总表，再按报名 ID、赛道路径和规则去匹配附件文件，最后把匹配成功的文件上传 OSS 并回填绑定关系。  
这里用的是 ZIP 流式解压和规则匹配，不是靠人工一个个对。代码入口在 [TestIrWorkBindServiceImpl.java](D:/develop/Personal-Blog-RAG-Backend-Project/cde-base/cde-modules/cde-gddatafair/cde-gddatafair-user/src/main/java/com/cde/gddatafair/user/service/impl/TestIrWorkBindServiceImpl.java)。从工程角度看，这个过程本质上是在把“非结构化目录”转成“结构化绑定关系”，核心难点不是上传，而是正确匹配。

**追问：如果一个报名对应多个赛道怎么办？**  
**答：** 多赛道时我会优先根据路径和赛道名做精确匹配；如果无法明确判断，就宁可跳过，也不冒险错绑。因为评审附件一旦错绑，后面的打分和追责链路都会出问题。  
这个答案能体现取舍：宁可少绑，也不要乱绑。在评审类系统里，错绑的成本远高于漏绑的成本，所以我会优先保证准确性，而不是盲目追求覆盖率。

### 1.5 数据服务快速交付

**问：MagicAPI 这块你主要做了什么？**  
**答：** 我参与的是“快速交付数据查询接口”的方案落地。核心思路是通过 MagicAPI 实现多数据源路由和接口权限控制，把本来需要写很多 Controller、Service、Mapper 的查询接口，改成配置化交付。  
这样做的好处是接口上线更快，适合那种“需求变得快、数据口径又比较固定”的场景。更直白一点说，它把“写接口”的时间，从开发周期里往前压缩了很多，业务方要的只是几个查询能力，而不是完整的代码工程。

**问：为什么要单独做这一层，而不是让业务代码直接查库？**  
**答：** 因为有些查询接口本质上是数据服务，不应该强耦合到某个业务模块里。单独一层 MagicAPI 后，后面如果只是改 SQL、改返回字段、改权限范围，就不需要动主业务代码。  
你可以把它理解成“查询能力产品化”。这个思路很适合报表、统计、运营查询这类需求，因为它们往往变化快，但底层逻辑并不复杂，拆出来以后维护成本会低很多。

**问：有没有替代方案？**  
**答：** 有。可以直接写业务接口，也可以把查询放到报表服务里，甚至用低代码平台。但 MagicAPI 的优势是交付快、配置化强、权限控制相对集中。缺点是复杂业务逻辑还是得回到 Java 里处理，不能无限下沉。  
如果是高复杂度的事务型接口，我不会优先用 MagicAPI。它更适合“查询多、变更快、逻辑轻”的场景，而不是强事务强规则场景。

### 1.6 实习常见追问

**问：这段实习里你最有价值的贡献是什么？**  
**答：** 我觉得不是某一个按钮或者某一个页面，而是把几个最容易出问题的链路打通了：报名的限流和幂等、导出的异步和分发、评审的权限和附件绑定、数据接口的快速交付。  
这些功能都偏“能上线、能扛事”，而不是只做展示。

**问：如果再做一次，你会改什么？**  
**答：** 我会把参数结构、状态流转和导出任务进一步标准化，再把一些规则沉到配置里，这样后面扩赛道、扩评审规则、扩导出格式都会更轻。

## 第二章 RAG 智能体平台（Ragent）

### 2.1 项目总览

**问：这个项目到底是做什么的？**  
**答：** 这是一个围绕“企业知识问答 + 外部工具调用 + 长对话记忆”做的 RAG 智能体平台。它不是一个简单的“上传文档然后问一句答一句”的 Demo，而是一条完整的生产链路：文档入库、分块、向量化、检索、意图识别、工具调用、模型生成、会话记忆、限流和降级都在同一套体系里。  
从架构上看，项目没有把所有逻辑塞进一个服务，而是拆成了 `bootstrap`、`infra-ai`、`mcp-server` 三层。`bootstrap` 负责业务编排和主问答链路，`infra-ai` 负责模型能力适配、候选路由和故障转移，`mcp-server` 负责把天气、工单、销售这类外部能力包装成标准工具。这样拆分后，每一层的职责都很清楚，后续如果要替换模型、扩工具或调整链路，不会把整个系统一起带崩。  
从业务角度看，这个项目真正要解决的不是“能不能搜到文档”，而是“用户的问题经常不标准、上下文会延续、知识源还可能来自文档和外部系统，系统怎么在这种复杂场景下依然稳定给出可解释的答案”。所以它更像一个企业级知识与工具中台，而不是传统 FAQ。  
项目目标也不是“先跑起来就行”，而是让系统在并发高、问题复杂、模型波动、上下文增长的时候，仍然能保持可用、可观测、可扩展，并且在出错时能清楚知道卡在了哪一段链路。

**问：你在这个项目中的角色是什么？**  
**答：** 我主要负责的是主链路的业务编排和核心能力落地，覆盖知识入库、分块、向量检索、意图路由、会话记忆压缩、队列限流、模型路由和 MCP 对接。  
如果把用户发出一条问题后的流程拆开，我做的事情不是单点功能，而是把整条链路串起来：先决定这是不是一个要查知识的问题，再判断要不要拆子问题、要不要走 MCP 工具、要不要召回知识库、要不要引导用户澄清、要不要压缩上下文、要不要切换模型。  
这种回答方式比“我写了几个接口”更有面试价值，因为它说明你理解的是系统运行方式，而不是只记住了某个类名。面试官如果继续追，我也能顺着讲清楚每一层为什么要这么拆、出错以后怎么兜底。

**问：你们用了什么技术栈？**  
**答：** 后端核心是 Spring Boot + MyBatis Plus + PostgreSQL + Redis + Redisson + Apache Tika + Sa-Token。这个组合的核心不是“堆技术”，而是每一层都承担了很明确的职责：Spring Boot 负责业务编排，MyBatis Plus 负责数据库读写，PostgreSQL 存知识、对话、摘要和意图树这类结构化数据，Redis 和 Redisson 负责缓存、分布式锁和队列控制，Tika 负责文档抽取，Sa-Token 负责认证鉴权。  
模型侧通过统一的 `infra-ai` 封装 Chat、Embedding、Rerank，工具侧通过独立的 `mcp-server` 暴露 MCP 能力。当前默认向量维度是 1536，记忆默认配置是 `historyKeepTurns=4`、`summaryStartTurns=5`、`summaryMaxChars=200`，这些都能在 [application.yaml](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/resources/application.yaml) 和对应配置类里看到。  
如果面试官继续追问“为什么是这个组合”，你可以进一步解释：Spring Boot 解决服务编排，Redis 解决高频状态和临时协调，Redisson 解决分布式锁，PostgreSQL 解决持久化，Tika 解决文档类型兼容，MCP 解决外部能力接入。这个组合的重点不是“炫技”，而是让每个问题都落在最适合处理它的那一层，方便定位和扩展。

**问：这是直接套 Spring AI 或阿里那类 Agent 框架吗？**  
**答：** 不是直接套一个现成 Agent 框架，而是自研编排。主链路在 `bootstrap` 里组织，模型适配和故障转移在 `infra-ai`，工具协议在 `mcp-server`。  
这样做最大的好处是可控性高，关键行为都在自己手里。比如“什么时候先引导用户澄清”“什么时候直接答”“什么时候走 KB”“什么时候走 MCP”“模型失败后切哪个候选”“上下文太长怎么压缩”，这些行为都不是框架默认帮你做的，而是你得自己定义。  
如果面试官问“那为什么不用现成框架省时间”，你可以说：现成框架适合快速起步，但这个项目里真正难的部分不是简单的链式调用，而是多模型路由、队列限流、熔断降级、会话摘要、工具编排和错误兜底。等你把这些约束加进去以后，很多默认框架其实还是要改，不如一开始就按照自己的业务边界拆开。

### 2.2 知识入库与分块

**问：RAG 的具体入库流程是怎样的？**  
**答：** 流程大致是：文档上传 -> 文档元信息入库 -> 选择解析器 -> Tika 抽取文本 -> 按策略分块 -> 生成 embedding -> 写入 chunk 表和向量库 -> 更新文档状态和日志。  
如果展开讲，`upload` 这一步不是简单接收一个文件，而是先校验知识库是否存在、源类型是否合法、是否需要定时同步，然后把文件存储到对象存储里，顺便把文档记录写入数据库。真正重活在 `startChunk` 和 `executeChunk`：前者通过消息队列把“分块任务”异步化，后者再真正去做抽取、切分、向量化和持久化。这样做的好处是上传接口很快返回，分块失败也能单独重试，不会把主请求线程拖死。  
在代码里，[KnowledgeDocumentServiceImpl.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/knowledge/service/impl/KnowledgeDocumentServiceImpl.java) 负责整体编排，`runChunkTask` 会先写 chunk log，再按 `PIPELINE` 或普通 `chunk -> embed` 两种模式处理。处理成功后还会用事务把旧 chunk、旧向量、新 chunk、新向量和文档状态一起同步掉，这样能保证“分块结果”和“向量结果”始终一致。  
你答这个问题时最好强调“入库不是单步动作，而是一条可追踪流程”。因为后面如果某个 chunk 失败、某个索引异常、某个文档重跑，系统必须知道卡在哪一步，所以它把文档状态、处理模式、分块策略、耗时统计和失败原因都单独记录下来。

**问：你们存的知识是什么形式？**  
**答：** 源文件是对象存储里的原始文档，业务上支持多种常见格式，最终都先通过 Apache Tika 统一转成文本。数据库里存的是文档元信息、分块结果和向量索引信息，向量库里存的是 chunk 级别的 embedding。  
所以不是“把 PDF 直接扔进向量库”，而是“原文件 + 结构化文本 + 向量索引”三层分开。  
这样拆的好处是：原文件可以保留审计和重建能力，结构化文本方便后续重跑分块，向量索引则专门负责相似度检索。  
如果面试官继续追问“为什么不只存向量”，你就说向量只能解决找相似，不能替代原始文档的可追溯性和二次处理能力。原文件、文本和向量三层分开，后面才方便重跑和审计。

**问：你们为什么要做结构感知分块？**  
**答：** 因为简单按长度切块很容易把标题、代码块、列表、段落打散，召回时上下文就碎了。结构感知分块的目标，是尽量在自然边界切，保住语义完整性。  
比如一个章节标题下面跟着一段解释和一个代码块，如果硬切成两半，模型检索到其中一半时就很难判断这块内容到底在讲什么。  
具体实现看 [StructureAwareTextChunker.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk/strategy/StructureAwareTextChunker.java)：它把标题、代码块、图片/链接、普通段落先扫描成块，再按 min/target/max 去拼装，尽量不在块内部硬切。  
你如果要讲得更像工程实现，可以补一句：它不是追求“数学上的均匀切分”，而是追求“语义上的完整切分”，因为 RAG 的效果瓶颈经常不在模型，而在文本被切碎了。

**问：如果不用结构感知，直接固定长度切不行吗？**  
**答：** 可以，但检索质量通常会差一点。固定长度切块实现简单、稳定，适合纯文本、扫描 OCR 后的长文本或者格式比较糟糕的内容，但它的缺点是容易把一个逻辑段落切成几块，后续回答时模型需要自己拼。  
所以更稳的做法是：优先结构感知，必要时再用固定长度兜底。对应实现可以看 [FixedSizeTextChunker.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk/strategy/FixedSizeTextChunker.java)。它的价值不是“更智能”，而是“更朴素、更可靠”，在格式不可控的时候能保证至少能切出可用块。  
如果面试官继续问“那为什么不统一一种切法”，你就说：结构感知对格式好的文档效果更优，但对非常杂乱的文本未必稳定；固定长度虽然粗糙，但兜底能力强。工程上最好两者并存，而不是二选一。

### 2.3 多通道召回与重排序

**问：为什么要做多路召回，不直接向量检索就完了？**  
**答：** 因为复杂问题不是所有情况都适合同一种召回方式。有些问题更适合意图定向检索，比如用户已经说得很明确；有些问题更适合全局向量召回补底，比如用户问题表达比较模糊、但语义上和知识库里某一段内容很像。  
所以我把召回拆成两路：一条围绕意图走定向检索，一条围绕向量走全局召回，然后再做去重、重排和融合。这样既保证覆盖面，也保证结果稳定。配置上也不是拍脑袋，`vector-global` 走的是 `confidence-threshold=0.6`、`top-k-multiplier=3`，`intent-directed` 走的是 `min-intent-score=0.4`、`top-k-multiplier=2`，本质上就是根据通道置信度控制召回力度。  
更直白一点说，单通道向量检索在“问题表述很接近知识原文”时表现不错，但遇到多条件、跨主题、带上下文指代的问题，单路检索很容易漏。多路召回的价值，就是把“想得到的”和“可能漏掉的”一起抓回来，再让后处理做收敛。`IntentParallelRetriever` 会按意图节点动态计算每个子任务的 TopK，`MultiChannelRetrievalEngine` 再把多通道结果统一做后处理。

**问：为什么还要重排序？**  
**答：** 因为初召回拿到的只是“可能相关”的候选，不代表真正最相关。尤其是多通道召回时，重复 chunk、相似 chunk、不同通道召回的 chunk 会一起进来，如果不重排，模型很容易先看到一堆一般相关甚至重复的证据。  
所以重排序是把粗召回变成精筛的一层，属于 RAG 里很值的一步。它做的事情其实是“把候选压缩成更可信的证据集”，避免模型被前几条噪声误导。检索侧核心在 [RetrievalEngine.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/RetrievalEngine.java)。  
如果你要回答“为什么不直接把 topK 调大”，可以说：调大 topK 只是增加候选，不会自动提高排序质量，反而可能把噪声一起带进去。重排序的意义不是多拿，而是更准地选，所以它和 topK 不是替代关系，而是先扩召回、再收结果的两段式配合。

**问：单通道向量检索还需要重排吗？**  
**答：** 需要，但重要性会比多通道弱一些。单通道时数据更单纯，重排主要解决“相似度高但并不最相关”的问题；多通道时，重排还承担了“去重 + 融合 + 排序收敛”的作用。  
所以你可以这么说：单通道能跑，但多通道 + 重排更适合企业场景。  
如果面试官进一步问你“那是不是所有检索都必须重排”，你可以补一句：不是。简单 FAQ、短问题、强约束场景未必值得加太多链路；但在企业知识问答里，问题复杂度高、知识源异构、答案要求稳，重排通常值得。

**问：向量库里到底存了多少维？用的什么模型？**  
**答：** 当前配置是 1536 维，对应的是 `Qwen3-Embedding-8B` 这一类 embedding 模型，代码里既支持云端 embedding，也保留本地 `qwen3-embedding:8b-fp16` 的候选。  
这个维度不是单独为了“更大”而选，而是要和向量库、索引方式以及整体成本一起匹配。换句话说，embedding 负责把文本压成检索向量，维度太低容易损失语义，维度太高又会增加存储和检索成本，所以 1536 是一个比较折中的工程选择。  
这些配置在 [application.yaml](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/resources/application.yaml) 里能看到。

**问：你为什么选这个向量模型？**  
**答：** 我会从三个角度解释：第一，维度和向量库要匹配；第二，要适合中文检索场景；第三，要考虑成本和可替换性。  
所以这个选择不是拍脑袋，而是结合项目定位、模型效果和部署成本一起定的。你也可以补一句：我们保留了本地和云端两个候选，方便后续切换。

**问：向量模型和回答时用的大模型有什么区别？**  
**答：** 向量模型的职责是把文本映射成向量，方便做相似度检索；回答用的大模型负责理解上下文、推理和生成。  
前者更像“搜索编码器”，后者更像“生成器”，两者不是一类东西，不能混着讲。

### 2.4 查询改写、意图路由和歧义引导

**问：你们的 RAG 怎么把问题改写、拆分、路由串起来？**  
**答：** 主链路是先加载会话记忆，再做查询改写，把复杂问题拆成更适合检索的子问题，然后进入树形意图分类，区分 KB、MCP 和 SYSTEM。  
这里不是“先问模型，再顺手检索”这么简单，而是把每一步拆成独立阶段：先拿历史上下文，避免模型看不到前文；再改写问题，把口语化、代词化、合并式的问题拆成更利于检索的子问题；然后分类判断这道题应该走知识库、工具调用还是系统直答；最后再决定是否做歧义引导。  
如果意图不明确，就先做歧义引导；如果明确，就继续走检索或者直答。这个过程不是一坨 if-else，而是按阶段拆开的。可参考 [DefaultConversationMemoryService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/DefaultConversationMemoryService.java)、[IntentResolver.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/IntentResolver.java) 和 [DefaultIntentClassifier.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/DefaultIntentClassifier.java)。

**问：树形意图分类器是怎么实现的？**  
**答：** 我先从 Redis 读意图树，Redis 没有就回源数据库重建，然后把整棵树展开成叶子节点列表交给 LLM 分类。LLM 只需要在候选叶子里选，不需要理解完整树结构。  
这样设计有两个好处：第一，分类空间被压缩到叶子节点后，模型不需要在很大的标签树里做选择，稳定性会更高；第二，树结构可以把 KB、MCP、SYSTEM 三类能力都统一放进去，后面路由就更自然。  
分类结果会先做 JSON 解析，再按分数排序，并过滤掉低分项。`IntentResolver` 还会在多子问题场景下做总量封顶，避免结果太多；同时，节点的 prompt 模板、工具 ID、类别类型都被编码到树里，方便后续直接把意图映射到具体能力。

**问：为什么要做歧义引导？**  
**答：** 因为有些问题的几个候选方向分数很接近，如果硬选一个，很容易把检索带偏。歧义引导的作用是先把不确定性前置，让用户确认方向，再继续往下走。  
这比“先搜错再补救”要稳得多，也更符合企业问答场景。尤其是当两个候选方向只差一点点时，模型自己拍板很容易把问题带歪，所以我更倾向于先让用户选范围，再继续往下做精确检索。  
在实现上，它不是每次都弹引导，而是先看候选分数差距、再看问题里是否已经显式提到了某个系统名或领域名，只有真的不确定时才做澄清，避免过度打扰用户。

### 2.5 会话记忆与摘要压缩

**问：你们的会话记忆阈值是多少？**  
**答：** 当前配置里，`history-keep-turns` 是 4，`summary-start-turns` 是 5，`summary-max-chars` 是 200。  
也就是说，系统会保留最近 4 轮原文历史，超过 5 轮后就会启动摘要压缩。这个阈值的含义不是“只记四轮”，而是用最近短窗口保证上下文连续性，再用摘要承接更早的内容，避免对话越长越占 token。配置在 [application.yaml](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/resources/application.yaml) 里。

**问：这个阈值的标准是什么？**  
**答：** 主要是“对话轮数”，不是 token 数。原因很简单：轮数更容易做稳定控制，也更容易和业务交互节奏对应。  
当轮数超过阈值后，系统会用最近一段对话生成摘要，再保留最近几轮原文，形成“长期摘要 + 近期上下文”的组合记忆。

**问：摘要是怎么持久化到数据库里的？**  
**答：** 摘要生成后会写入摘要表，里面会记录摘要内容和 `lastMessageId`，方便下次只压新的消息，而不是每次都从头总结整段历史。消息表和摘要表都在关系型数据库里，项目主库配置是 PostgreSQL，但如果你面试时被问成 MySQL，也可以说“落到关系型数据库的摘要表里，原理一样”。  
摘要逻辑在 [JdbcConversationMemorySummaryService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/JdbcConversationMemorySummaryService.java)，读取和拼接在 [DefaultConversationMemoryService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/DefaultConversationMemoryService.java) 和 [JdbcConversationMemoryStore.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/JdbcConversationMemoryStore.java)。  
这里最重要的点其实是“增量压缩”：摘要不是一次性从全量历史里重新生成，而是借助 `lastMessageId` 只压新消息，这样既降低了重复成本，也能减少摘要越写越乱的问题。

**问：摘要超长了怎么办？**  
**答：** 当前实现先通过字符上限控制，`summary-max-chars` 只有 200，另外还有最近轮数裁剪，所以摘要本身不会无限长。  
如果摘要真的还不够短，下一步的优化思路就是把摘要拆成“硬约束”和“对话概括”两层：硬约束保留用户明确要求，比如语言、格式、角色；对话概括负责记录前文讨论了什么。这样做比单纯把摘要再压一遍更稳，因为它能保留真正不能丢的约束信息。  
如果你要回答“更极端的情况”，可以说后续可以把摘要拆成“硬约束”和“对话概括”两层，先保留用户明确要求，再压缩普通历史，必要时做二次摘要。

**问：自动摘要和对话过程是并行的还是串在一起的？**  
**答：** 逻辑上是集成在对话生命周期里，物理上是异步的。也就是说，主回答不会被摘要阻塞，摘要是在 assistant 消息落库后异步触发的。  
这点可以看 [DefaultConversationMemoryService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/DefaultConversationMemoryService.java)：`append` 先写消息，再交给摘要服务异步压缩。这样设计的好处是用户不会因为“后台在总结历史”而感到响应变慢，主链路和增强链路是解耦的。

**问：如果摘要丢了重要约束怎么办？**  
**答：** 这是记忆系统最容易被追问的点。我的优化思路是把硬约束单独存成结构化记忆，比如“语言要求”“格式要求”“角色要求”，不要只放在自然语言摘要里。  
当前代码是“摘要 + 最近历史”的方案，如果面试官继续追，你就说：对于必须保留的约束，最好单独做强制记忆槽，避免被摘要压缩掉。

### 2.6 模型路由、熔断和限流

**问：为什么要做多模型路由？**  
**答：** 因为不同模型在成本、速度、上下文能力和稳定性上都不一样。业务层不应该直接绑死某一个供应商，所以我在 `infra-ai` 里统一封装了模型路由和降级。  
这样上层只关心“我要 Chat / Embedding / Rerank”，不关心具体是哪个 provider 执行。更重要的是，路由层还能把“失败切换”“健康状态”“冷却时间”这些治理逻辑收拢起来，不让每个业务点自己写一遍。这样一来，后面换模型、加候选或者做灰度，都只需要改路由配置，而不是改主链路代码。

**问：熔断和降级怎么做的？**  
**答：** 路由层会维护每个模型的健康状态，调用失败达到阈值后进入熔断，冷却一段时间后再做半开探测。当前配置里，失败阈值是 2，打开时长是 30 秒，也就是说模型连续失败两次后会先暂停一段时间，再尝试恢复。  
调用成功就恢复健康，失败就切换到下一个候选。相关实现可看 [RoutingLLMService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/RoutingLLMService.java)、[ModelSelector.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelSelector.java)、[ModelRoutingExecutor.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelRoutingExecutor.java)。  
你如果要讲得更像工程设计，可以补一句：熔断不是为了“少调用”，而是为了把不稳定模型隔离出去，避免它把所有请求都拖慢；成功/失败状态由路由层统一记录，这样链路行为才可观测。

**问：请求高峰时你们怎么限流？**  
**答：** 我们不是简单拒绝，而是做了“排队 + 限流 + SSE 实时反馈”。  
底层用 Redis 信号量、ZSET 和 Pub/Sub 维护队列，前端可以实时看到排队进度；如果等待超时，就返回可读的繁忙提示，而不是让请求一直挂着。当前全局并发限制是 1，最大等待时间是 15 秒，许可证租期是 30 秒，轮询间隔是 200 毫秒，这些参数的目的就是让系统在高峰期宁可慢一点排队，也不要一下子把模型调用打爆。相关逻辑在 [ChatQueueLimiter.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/service/ratelimit/ChatQueueLimiter.java)。  
如果面试官继续追问“为什么不直接在网关层限流”，你可以说：网关限流只能控制入口流量，但不能解决业务层内部资源争抢；而排队限流可以把真正进入大模型的并发控制住，还能给用户一个明确的等待反馈。

**追问：为什么不直接在网关层限流？**  
**答：** 网关限流能挡一部分流量，但挡不住业务层内部的并发资源争抢。我的做法是：入口层控制节奏，业务层再做排队和并发许可控制，这样能更平滑地保护后面的模型调用。

### 2.7 MCP 工具调用

**问：MCP 在这个项目里起什么作用？**  
**答：** MCP 是把外部业务系统能力标准化成可调用工具，让模型不仅能查知识库，还能查实时数据、调业务系统。  
我这里是把意图路由和 MCP 工具绑定起来，模型抽参数后并行执行工具，最后把结果融合进回答上下文。这样做的意义是把“知识问答”和“能力调用”分开：知识库负责静态或半静态信息，MCP 负责需要实时执行、实时返回的能力，这样系统才能从“会答题”升级到“能办事”。

**问：为什么要单独做 MCP Server？**  
**答：** 因为工具能力本质上是“可调用的外部服务层”，和主问答链路职责不同。  
独立成 MCP Server 后，工具可以单独注册、单独扩展、单独替换，不会污染主业务问答代码。服务端入口是在 [McpServerConfig.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/mcp-server/src/main/java/com/nageoffer/ai/ragent/mcp/config/McpServerConfig.java) 里通过 `/mcp` 这个 servlet 端点挂出来的，实际工具列表则由 [DefaultMcpToolRegistry.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/DefaultMcpToolRegistry.java) 自动装配。  
也就是说，它不是“一个散乱的控制器写一堆接口”，而是一个标准协议入口 + 自动注册工具的组合，后面加天气、工单、销售这类工具时只需要实现 executor 即可。

**问：MCP 和 KB 的区别是什么？**  
**答：** KB 解决的是“知识检索”，MCP 解决的是“能力调用”。  
比如 KB 更适合查文档、查规则；MCP 更适合查实时状态、业务系统数据或者做结构化动作。两者是互补关系，不是替代关系。

**问：你是怎么把 MCP 和 RAG 融合的？**  
**答：** 先由意图分类把问题路由到 KB、MCP 或 SYSTEM，再由检索引擎分别执行。MCP 侧会先抽参数，再调用对应工具，最后由上下文格式化器把结果整理成模型可读的片段。  
检索主入口在 [RetrievalEngine.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/RetrievalEngine.java)。

### 2.8 这套项目的关键价值

**问：如果让你一句话总结你在 RAG 项目的贡献，你怎么说？**  
**答：** 我把“记忆、改写、意图、检索、工具、模型路由、限流、追踪”这些环节串成了一条能稳定上线的问答链路，而不是只做了一个能跑的 Demo。  
这句话比单独讲某个类更能体现你的价值。

## 第三章 截图追问专题

### 3.1 场景、职责、框架

**问：这个智能检索平台是在什么场景下做的？**  
**答：** 它是围绕企业级文档检索和问答场景做的，目标是让用户把内部文档、业务资料和动态工具数据统一接入到一个问答入口里。  
所以它不是普通 FAQ，而是“知识 + 工具 + 会话记忆”的综合系统。

**问：你在项目中的角色和职责是什么？**  
**答：** 我主要负责主链路的工程落地，包括知识入库、分块、检索、意图路由、记忆压缩、MCP 调用和模型路由。  
如果面试官追问你到底做了什么，就把“主链路编排”这几个字说出来，别只说“我写了接口”。

**问：你们用了什么框架开发这个 Agent 平台？**  
**答：** 没有直接依赖某个通用 Agent 框架来包办全部流程，而是用 Spring Boot 作为业务底座，自己把链路拆成 `bootstrap`、`infra-ai`、`mcp-server` 三层来编排。  
这样更灵活，也更容易把路由、熔断、降级和工具协议控制在自己的代码里。

### 3.2 RAG 主链路

**问：RAG 的召回、Prompt 组装、发送给模型，这个流程是怎么串起来的？**  
**答：** 先由记忆服务拿到上下文，再做查询改写和意图路由；然后走多通道检索，把 KB 和 MCP 的结果分别召回、去重、重排；最后由 Prompt 服务把 system、证据、历史和当前问题组装好，再交给统一模型服务发给大模型。  
这整条链路是自研编排，不是单纯把一个 Agent 框架往里一套就完事。

**问：你们构建 RAG 的具体流程是怎样的？**  
**答：** 可以按 9 步说：文档接入、文本抽取、分块、embedding、向量入库、问题改写、意图识别、检索融合、最终生成。  
如果对方继续追，你再补一句：中间还穿插了记忆压缩、歧义引导、模型路由和熔断降级。

**问：你们存储的知识是什么格式？**  
**答：** 源文件是对象存储里的原始文档，类型上覆盖 PDF、Word、文本类等常见格式，统一用 Tika 抽成文本。  
真正进入 RAG 的不是原始文件，而是切好的 chunk 和它对应的向量。

**问：从文档到向量数据库的步骤是什么？**  
**答：** 先上传文件并记录文档元信息，然后用 Tika 抽文本，再按策略分块，随后对每个 chunk 生成 embedding，最后把 chunk 元数据和向量写入数据库/向量库。  
如果重跑分块，会先删除旧 chunk 和旧向量，再写新结果，保证一致性。

### 3.3 分块、向量和模型

**问：Chunking 用了什么策略？**  
**答：** 我们同时支持结构感知分块和固定长度分块。结构感知适合 Markdown/结构化文本，固定长度适合作为兜底。  
策略选择在 [ChunkingStrategyFactory.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk/ChunkingStrategyFactory.java)，具体实现看 [StructureAwareTextChunker.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk/strategy/StructureAwareTextChunker.java) 和 [FixedSizeTextChunker.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk/strategy/FixedSizeTextChunker.java)。

**问：结构感知分块怎么实现的？**  
**答：** 它不是简单按字符硬切，而是先把标题、代码块、段落、图片/链接这些结构扫描成块，再按照 min/target/max 做拼装，尽量让 chunk 保持自然语义边界。  
这样做的本质是：保住上下文，减少切碎语义，提升召回后的可读性和可回答性。

**问：向量库里存了多少维？用什么模型生成的？**  
**答：** 1536 维。当前默认 embedding 模型是 `Qwen3-Embedding-8B` 一类配置，同时保留本地 `qwen3-embedding:8b-fp16` 作为候选。  
这部分在 [application.yaml](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/resources/application.yaml) 里可以直接看到。

**问：你怎么选这个向量模型的？**  
**答：** 主要看三点：维度是否匹配向量库、中文检索效果是否稳定、部署成本能不能接受。  
所以它不是“哪个热就用哪个”，而是根据项目场景、资源条件和检索效果综合选的。

**问：这个向量模型和回答用的大模型有什么区别？**  
**答：** 向量模型负责把语义压到向量空间里，用来算相似度；回答大模型负责理解上下文并生成答案。  
一个偏检索表征，一个偏文本生成，作用完全不同。

### 3.4 重排序和多通道召回

**问：为什么需要重排序？**  
**答：** 因为召回阶段拿到的是候选，不是最终答案。多通道召回时，重复片段和弱相关片段很容易一起回来，不重排的话，模型可能先看到一堆“看起来像相关、实际上不够准”的内容。  
重排序就是把粗召回变成精筛，属于很值的一层。

**问：如果只有单通道向量检索，还要重排序吗？**  
**答：** 也要，只是压力没那么大。单通道时重排主要解决“相似但不最相关”的问题；多通道时，它还要负责去重和融合收敛。  
所以我的答案通常是：能不重排吗？能，但效果和稳定性一般会差一截。

**问：你们是一开始就做双通道并行召回的吗？**  
**答：** 架构目标上是双通道并行，但落地上通常会先把主链路打通，再逐步加全局召回、意图定向和后处理。  
你答的时候不要说得太死，可以说“从设计上就是双路互补，从实现节奏上是分阶段补齐的”。

### 3.5 树形意图分类器

**问：树形意图分类器怎么实现的？**  
**答：** 先从 Redis 加载意图树，Redis 没有就回源数据库，然后把整棵树展开成叶子节点列表交给 LLM 做分类。  
分类结果会做 JSON 解析、分数排序和阈值过滤，再按子问题做总量封顶。核心实现看 [DefaultIntentClassifier.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/DefaultIntentClassifier.java) 和 [IntentResolver.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/intent/IntentResolver.java)。

**问：为什么要做树形，而不是平铺分类？**  
**答：** 因为企业问题通常有层级。树形可以先大类再细类，减少模型在大量标签里做选择的压力，也更便于后面做 KB / MCP / SYSTEM 路由。  
平铺分类也能做，但候选多了以后会更乱。

### 3.6 会话记忆

**问：会话记忆阈值是多少？摘要怎么持久化？**  
**答：** 当前是保留最近 4 轮原文，超过 5 轮后触发摘要压缩；摘要会写入摘要表，并带上 `lastMessageId`，下一次只压增量内容。这样做的原因是，前几轮对话通常还比较短，保留原文能让模型更稳地接上下文，等轮次变多以后再把旧内容压成摘要，能更好地平衡上下文完整性和 token 成本。  
这部分是异步的，不会阻塞主回答。代码见 [DefaultConversationMemoryService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/DefaultConversationMemoryService.java) 和 [JdbcConversationMemorySummaryService.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/memory/JdbcConversationMemorySummaryService.java)。

**问：这个阈值标准是什么？轮数还是 token？**  
**答：** 当前按轮数控制，更稳定也更容易工程落地。  
如果面试官继续问，你就补一句：轮数是基础阈值，真正执行时还会结合摘要字符上限和最近历史窗口一起控制。

**问：如果摘要本身也超出 token 限制怎么办？**  
**答：** 当前实现先用字符上限和历史窗口兜住，如果还是太长，下一步应该做层级摘要：先保留硬约束，再压缩普通历史。  
更激进一点，可以把“用户偏好/硬规则”单独做结构化记忆，不把它们完全交给自然语言摘要。

**问：自动摘要和会话过程是并行还是集成？**  
**答：** 逻辑上集成在对话流里，物理上异步执行。主回答先返回，摘要后台慢慢压，不会卡住用户。  
这个答案能体现你对“主链路”和“副链路”的分离意识。

**问：如果首轮有个重要约束，比如必须用 Java，后面摘要丢了怎么办？**  
**答：** 最好的优化不是“让摘要更聪明”，而是把硬约束单独建模。  
比如把语言要求、格式要求、角色要求单独存为结构化记忆，摘要只负责概括对话，不负责承载所有规则。这样就不会因为摘要压缩把关键约束吞掉。

### 3.7 MCP 和系统能力

**问：MCP 工具调用在项目里怎么接的？**  
**答：** 先由意图路由把问题分到 MCP，再由参数抽取器从自然语言里抽工具参数，最后通过 MCP 协议调用外部工具。  
MCP 服务端是独立模块，入口在 [McpServerConfig.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/mcp-server/src/main/java/com/nageoffer/ai/ragent/mcp/config/McpServerConfig.java)，而工具参数抽取和工具注册分别由 [LLMMcpParameterExtractor.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/LLMMcpParameterExtractor.java) 和 [DefaultMcpToolRegistry.java](D:/develop/Personal-Blog-RAG-Backend-Project/ragent/bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/mcp/DefaultMcpToolRegistry.java) 负责。  
这样拆开后，MCP 工具的定义、参数理解和执行逻辑是分层的：意图层决定“要不要调”，参数抽取层决定“怎么调”，执行器决定“具体怎么执行”，这就是它比简单反射调用更稳定的地方。

**问：为什么工具能力要独立成服务？**  
**答：** 因为工具是“能力层”，不是“文档层”。独立出来以后，工具可以单独注册、单独扩展、单独替换，也不会污染主问答链路。  
这样 KB 和 MCP 的职责会非常清楚：一个负责知识，一个负责行动。

**问：多模型路由和熔断为什么要单独做？**  
**答：** 因为模型不可避免会有抖动、超时和供应商差异。如果业务层直接绑死一个模型，问题一来就会全链路受影响。  
我这里把路由、健康检查、失败统计、降级和重试都收到了 `infra-ai` 里，上层只看到统一的 Chat / Embedding / Rerank 接口。

## 第五章 截图追问深挖版

### 5.1 场景与定位

**问：这个智能检索平台是在什么样的场景下需要做的？**  
**答：** 它主要面向企业内部文档检索、知识问答和业务工具调用场景。单纯的 FAQ 只能回答固定问题，但企业里用户经常会问“这是什么”“这怎么改”“这个状态为什么是这样”“能不能顺手帮我查一下别的系统”，所以系统不能只会搜文档，还要能理解上下文、判断意图、调用工具。  
所以这个项目不是单点检索 Demo，而是一个把知识、工具和会话记忆整合起来的问答底座。它的价值不在于“能不能回答”，而在于“在问题不标准、知识不统一、上下文会延续的情况下还能不能稳定答对”。

**问：你在这个项目里的角色和职责是什么？**  
**答：** 我负责的是主链路的工程落地，不是只写某一个孤立接口。具体包括知识入库、分块策略、检索链路、意图路由、记忆压缩、队列限流、多模型路由以及 MCP 工具接入。  
如果面试官追问“那你到底做了什么”，我会把重点放在“主链路编排”上，因为这代表我不只是会写功能，而是参与了系统从输入、处理、检索到输出的完整过程。

**问：你们用的是什么框架来开发这个 Agent 平台？**  
**答：** 项目没有直接依赖一个现成的 Agent 框架来包办所有流程，而是以 Spring Boot 为业务底座，把系统拆成 `bootstrap`、`infra-ai` 和 `mcp-server` 三层来做编排。  
这样做的好处是能把模型路由、故障转移、工具协议和业务逻辑分开，不会被某个框架的默认约束绑住。换句话说，我们不是为了“用了某个框架”而用了它，而是为了让整条链路更可控、更容易扩展。

### 5.2 RAG 主链路

**问：在召回、Prompt 组装、发送给模型的过程中，完整流程是怎么串起来的？**  
**答：** 我们的流程不是一股脑把文档丢给模型，而是先从会话记忆里取上下文，再做查询改写和意图路由，然后进入多通道召回，分别从 KB 和 MCP 两侧拿候选结果，接着做去重和重排，最后再由 Prompt 组装模块把 system、证据、历史和当前问题拼成最终输入。  
这个顺序很关键，因为它体现的是“先理解问题，再找证据，再让模型生成”。如果顺序反过来，模型就会更像在猜答案，而不是基于证据回答。

**问：你们构建 RAG 的具体流程是什么？**  
**答：** 可以按九步讲：文档接入、文本抽取、分块、生成 embedding、向量入库、问题改写、意图识别、检索融合、最终生成。  
这个流程里前半段解决的是“知识怎么进来”，后半段解决的是“问题怎么问、知识怎么找、结果怎么稳”。我一般会强调：RAG 不只是向量检索，它其实是一整条围绕上下文治理的链路。

**问：你们存储的知识是什么格式？**  
**答：** 源文件本身会保留在对象存储里，支持 PDF、Word、文本等常见格式；真正参与 RAG 的是经过 Tika 抽取和分块后的 chunk，以及它们对应的元数据和向量。  
这样拆分的原因是，原文件适合审计和重建，结构化文本适合重跑分块，向量适合检索。三层分开以后，后续无论是重建索引、重跑任务还是排查问题都会方便很多。

**问：从文档到向量数据库，具体步骤是什么？**  
**答：** 首先上传文档并保存元信息，然后用 Tika 抽取正文，再按策略分块，之后为每个 chunk 生成 embedding，最后把 chunk 数据和向量一起写入数据库或向量库。  
如果需要重跑分块，系统会先清理旧的 chunk 和旧索引，再重新写入，保证数据一致性。这里面最重要的不是“写进去”，而是“能重建、能追踪、能恢复”。

### 5.3 分块、向量和模型

**问：Chunking 用了什么策略？**  
**答：** 我们支持结构感知分块和固定长度分块两种方式。结构感知更适合 Markdown、技术文档、说明书这种层次清晰的内容，固定长度更适合格式杂乱或者没有明确标题层级的文本，属于兜底方案。  
这不是为了炫技，而是为了适配不同文档形态。结构好的文档，如果按自然段、标题、代码块切开，召回后的上下文会更完整；但如果是扫描件转换出来的文本、格式很乱的长文本，强行做结构识别反而可能引入更多噪声，这时候固定长度更稳。  
所以这个项目里不是“选一种算法就够了”，而是按文档特征去选最合适的策略，必要时让固定长度兜住最坏情况。

**问：能不能解释一下结构感知分块是怎么实现的？**  
**答：** 它不是单纯按字符长度硬切，而是先识别标题、代码块、段落、链接、图片说明这些结构，再按照 min、target、max 的范围做拼装，尽量让 chunk 落在自然语义边界上。  
这么做的核心目的不是让每块长度一样，而是让每块内容“能被单独理解”。因为 RAG 里真正影响回答质量的，经常不是“有没有检索到”，而是“检索到的内容是不是被切碎了”。如果标题和正文被分到两块，或者一个代码示例被切成前半句和后半句，模型拿到后很难判断这一段到底在讲什么。  
所以结构感知分块本质上是在保语义完整性，它比单纯追求平均长度更适合知识问答场景。

**问：向量数据库里存了多少维？用的是什么模型？**  
**答：** 当前配置是 1536 维，对应的是 `Qwen3-Embedding-8B` 一类 embedding 模型，同时保留了本地 `qwen3-embedding:8b-fp16` 作为候选。  
这里维度不是越大越好，关键是要和向量库、索引配置以及整体成本匹配。模型选型本质上是在“效果、成本、部署复杂度”之间找平衡，不是单看参数大小。

**问：你是怎么选这个向量模型的？**  
**答：** 我主要看三点：第一，维度是否能和向量库匹配；第二，中文检索效果是否稳定；第三，部署和调用成本能不能接受。  
所以这个选择不是拍脑袋，也不是追热点，而是结合项目定位和实际资源做的折中。对企业项目来说，稳定可用往往比追求极限效果更重要。

**问：向量模型和回答用的大模型有什么区别？**  
**答：** 向量模型负责把文本映射成向量，用来做相似度检索；回答用的大模型负责理解上下文、推理并生成答案。  
更专业地说，embedding 解决“找得准”，chat 解决“答得好”，rerank 解决“排得对”。这三者配合起来，才是一个完整的 RAG 链路。

### 5.4 多通道召回与重排序

**问：为什么要做重排序？**  
**答：** 因为召回阶段拿到的只是候选，不是最终答案。尤其是多通道召回时，重复 chunk、弱相关 chunk 和不同来源的候选会一起进来，如果不重排，模型很可能先看到一堆“看着像相关、实际上不够准”的内容。  
重排序的作用就是把粗召回变成精筛，让真正更相关、更可信的内容排在前面。它不是锦上添花，而是企业级 RAG 里很值的一层。

**问：如果只有单通道向量检索，还需要重排序吗？**  
**答：** 需要，只是必要性没那么强。单通道时重排序主要解决“相似但不最相关”的问题；多通道时，它还承担去重和融合收敛的作用。  
所以我一般会说：单通道能跑，但多通道加重排更适合企业场景，因为它对答案稳定性和覆盖率都更友好。

**问：你们是一开始就做双通道并行召回的吗？**  
**答：** 从设计目标上看是双通道互补，但从落地节奏上通常会先把主链路跑通，再逐步补上全局召回、意图定向和后处理。  
这个说法比较符合真实项目情况，因为很多系统都不是一开始就完美，而是先跑出一条可用链路，再逐步增强效果和稳定性。

### 5.5 意图分类与路由

**问：树形意图分类器是怎么实现的？**  
**答：** 我们先从 Redis 读取意图树，Redis 没有就回源数据库重建，然后把整棵树展开成叶子节点列表交给 LLM 做分类。模型不需要理解整棵树，只需要在候选叶子里做判断即可。  
分类结果会先解析 JSON，再做分数排序和低分过滤，最后由 `IntentResolver` 处理多子问题场景下的结果封顶。这个设计的关键不是“树形看起来高级”，而是把分类问题拆小，让模型在更有限的候选里做判断。

**问：为什么要做树形，而不是平铺分类？**  
**答：** 因为企业问题通常天然有层级。树形可以先大类后小类，减少模型在大量标签里选择的压力，也方便后面把问题路由到 KB、MCP 或 SYSTEM。  
如果直接做平铺分类，候选一多，模型就更容易选错，Prompt 也会变长，整体成本和不稳定因素都会增加。

**问：为什么要做歧义引导？**  
**答：** 因为有些问题的几个候选方向分数很接近，如果硬选一个，很容易把检索方向带偏。歧义引导的作用就是把不确定性提前暴露给用户，让用户先确认方向，再继续往下走。  
这比“先搜错再补救”稳得多，也更符合企业问答场景。很多时候，系统不是不知道，而是应该先问清楚再回答。

### 5.6 会话记忆与摘要压缩

**问：你们的会话记忆阈值是多少？为什么这么设？**  
**答：** 现在是保留最近 4 轮原文，超过 5 轮后触发摘要，摘要长度上限是 200 字符。这个阈值不是拍脑袋定的，而是为了平衡上下文保留和 token 成本。  
如果阈值太小，系统容易丢上下文；如果阈值太大，摘要机制就失去意义，长对话的 token 成本也会越来越高。所以这是一个典型的工程折中值，后续也可以根据真实对话长度再调。  
你面试时可以顺手补一句：我们不是只看单轮回答效果，而是看连续多轮对话时系统还能不能稳定保持语义连续性，所以阈值设置本质上是围绕“上下文连续性”和“成本控制”做权衡。

**问：这个阈值标准是轮数还是 token？**  
**答：** 当前以轮数为主。轮数控制更稳定，也更容易和业务交互节奏对应；如果改成纯 token 控制，虽然更精细，但每轮都要额外估算 token，而且不同模型的 token 计算方式并不完全一致。  
所以在工程上，轮数是更容易落地的第一版方案。它不一定最精确，但它足够稳定、足够可解释。

**问：摘要是怎么持久化的？**  
**答：** 摘要会写入摘要表，并记录 `lastMessageId`，这样下一次只需要压缩新增消息，而不是重新总结整段历史。消息表和摘要表都在关系型数据库里，主库是 PostgreSQL。  
这个设计的核心是增量压缩。它能避免重复摘要、减少调用成本，也让系统在长会话场景下更可控。

**问：如果摘要本身也超出 token 限制怎么办？**  
**答：** 当前实现先用字符上限和历史窗口兜住，如果还是太长，后续可以做层级摘要。也就是说，先保留硬约束，再压缩普通历史，最后必要时再做二次摘要。  
更稳一点的优化方式是把“语言要求、格式要求、角色要求”这类硬约束做成结构化记忆，不完全交给自然语言摘要，这样就算摘要压缩，也不容易把关键规则吞掉。

**问：自动摘要和对话过程是并行还是集成？**  
**答：** 逻辑上它是集成在对话生命周期里的，物理上是异步执行的。也就是说，主回答先返回，摘要在 assistant 消息落库后再后台处理。  
这样做是为了不拖慢主问答接口，因为摘要是增强能力，不应该成为用户等待答案的阻塞点。

**问：如果首轮有个重要约束，比如必须用 Java，后面摘要丢了怎么办？**  
**答：** 这就是记忆系统最容易被追问的边界问题。我的思路是把硬约束单独建模，做成结构化记忆，而不是只藏在自然语言摘要里。  
这样一来，摘要负责概括上下文，结构化记忆负责保存硬规则，两者分工明确，后续就算摘要被压缩，也不会把关键约束吞掉。

### 5.7 限流、路由和工具融合

**问：为什么要做多模型路由？**  
**答：** 因为不同模型在成本、速度、上下文能力和稳定性上都不一样，业务层不应该直接绑死某一个供应商。我们把模型路由做成统一抽象，就是为了把模型变化从业务逻辑里解耦出去。  
这样上层只需要关心“我要 Chat、Embedding 还是 Rerank”，不用管具体是哪个 provider 在执行。后续无论是切模型、做灰度还是做降级，改动都能收敛在路由层。

**问：熔断和降级是怎么做的？**  
**答：** 路由层会维护每个模型的健康状态，调用失败达到阈值后进入熔断，冷却一段时间后再做半开探测。成功就恢复健康，失败就切换到下一个候选。  
这样做的意义是把模型波动隔离在路由层，不让它直接拖垮主链路。重试适合短暂抖动，但如果某个模型已经持续异常，继续打它只会放大延迟和失败率，所以熔断是更合理的选择。

**问：请求高峰时你们怎么限流？**  
**答：** 我们不是简单拒绝，而是做了“排队 + 限流 + SSE 实时反馈”。底层用 Redis 信号量、ZSET 和 Pub/Sub 维护队列，前端可以实时看到排队进度；如果等待超时，就返回可读的繁忙提示，而不是让请求一直挂着。  
这个方案比纯拒绝更友好，因为用户知道自己在排队；同时它也比完全放开更稳，因为真正进入模型链路的并发是被明确控制住的。

**问：为什么不直接在网关层限流？**  
**答：** 网关限流能挡住一部分流量，但挡不住业务层内部的并发资源争抢。我的做法是入口层控制节奏，业务层再做排队和并发许可控制。  
换句话说，网关层解决的是“别进来太多”，业务层解决的是“进来以后怎么有序处理”，这两个层面不是替代关系，而是分工关系。

**问：MCP 在这个项目里起什么作用？**  
**答：** MCP 是把外部业务系统能力标准化成可调用工具，让模型不仅能查知识库，还能查实时数据、调业务系统。  
我这里是把意图路由和 MCP 工具绑定起来，让模型先决定要不要调用工具，再抽参数、并行执行工具，最后把结果融合进回答上下文。它的本质是把“能回答”升级成“能办事”。

**问：为什么要单独做 MCP Server？**  
**答：** 因为工具是能力层，不是文档层。单独拆出来以后，工具可以单独注册、单独扩展、单独替换，不会污染主问答链路。  
如果某个工具有自己的权限、超时或者调用格式变化，只要 MCP 协议层稳定，主链路就还能继续跑。这种拆分方式更符合真实企业系统的演进方式。

**问：MCP 和 KB 的区别是什么？**  
**答：** KB 解决的是知识检索，MCP 解决的是能力调用。KB 更适合查文档、查规则；MCP 更适合查实时状态、业务系统数据或者做结构化动作。  
如果面试官问为什么不都放知识库里，你可以直接说：知识库适合静态或半静态知识，而工具调用适合动态、结构化、需要实时执行的事情，这两类职责本来就不一样。

**问：你是怎么把 MCP 和 RAG 融合的？**  
**答：** 先由意图分类把问题路由到 KB、MCP 或 SYSTEM，再由检索引擎分别执行。MCP 侧会先抽参数，再调用对应工具，最后由上下文格式化器把结果整理成模型可读的片段。  
你可以把这个融合过程概括成三步：先判断这是不是知识问题，再判断是不是要调工具，最后决定把检索结果、工具结果还是两者一起喂给模型。这样面试官就能看出你对整条链路是有整体判断的。

### 5.8 收尾追问

**问：如果让你一句话总结你在这个项目里的贡献，你怎么说？**  
**答：** 我把记忆、改写、意图、检索、工具、模型路由、限流和降级这些环节串成了一条能稳定上线的问答链路，而不是只做了一个能跑的 Demo。  
这句话的重点不在于“我用了很多术语”，而在于我真正把这些环节组织成了一个能落地的系统。

**问：如果再给你一次机会重做，你会改哪里？**  
**答：** 我会进一步把硬约束和上下文记忆分层，把检索和重排的策略配置化，再把模型路由和工具编排的监控做得更细。  
因为真正上线之后，系统的瓶颈往往不是“有没有功能”，而是“在高并发、长对话和模型波动下还能不能稳定工作”。
