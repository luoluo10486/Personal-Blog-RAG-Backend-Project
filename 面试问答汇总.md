# 项目面试问答汇总

> 说明：
> 1. 这份文档按“面试官最可能追问的点”整理，答案是可直接口述的版本。
> 2. `Ragent` 对应仓库 `ragent`，实习项目对应仓库 `cde-base/cde-modules/cde-gddatafair`。
> 3. 里面的回答偏“稳”，你可以按自己的真实分工再微调措辞。

## 一、30 秒项目介绍

### 1. Ragent 项目怎么介绍？

**问：** 你这个项目是做什么的？

**答：** Ragent 是一个企业级 RAG 智能问答平台，覆盖知识库入库、检索、重写、意图识别、模型路由、MCP 工具调用、流式问答和管理后台。它不是简单接一个大模型接口，而是把企业里真正会遇到的检索、权限、容错、追踪和并发控制都做成了完整链路。

**代码对应：** `ragent/README.md`，`bootstrap`、`infra-ai`、`framework`、`frontend`。

### 2. 广东省高质量数据集创新大赛平台怎么介绍？

**问：** 你实习做的这个平台是什么？

**答：** 这是一个覆盖赛事展示、在线报名、作品提交、专家评审和运营管理的比赛平台，累计服务过 393 支报名团队、10 家发榜单位。我实习期间主要参与需求梳理、表结构设计、接口开发和联调测试，重点做了报名模块、导出推送、初赛评审和数据服务接口交付。

**代码对应：** `cde-base/cde-modules/cde-gddatafair/*`。

## 二、Ragent 项目高频问答

### 1. 为什么要做这个项目？

**问：** 为什么不做一个普通 CRUD 项目，而是做 RAG？

**答：** 因为 RAG 更能体现后端工程能力。普通 CRUD 面试官很难区分项目深度，但 RAG 会天然涉及检索、上下文、向量库、模型路由、流式输出、限流、观测和容错，这些点都能展开聊，也更贴近企业真实落地场景。

### 2. 这个项目解决的核心问题是什么？

**问：** 你这个平台到底解决了什么问题？

**答：** 核心是把企业知识从“散落在文档里”变成“可检索、可追踪、可调用、可持续维护”的服务。用户问答时，不是直接让模型瞎答，而是先做意图判断，再走检索、重写、工具调用和生成，保证答案更准、链路更稳。

### 3. 为什么要做分层和模块化？

**问：** 你为什么把项目拆成 `framework`、`infra-ai`、`bootstrap`、`mcp-server` 这些层？

**答：** 这样做是为了把通用能力和业务能力拆开。`framework` 放 Web、异常、分布式 ID、幂等、消息等公共能力，`infra-ai` 统一封装不同模型供应商，`bootstrap` 只管业务流程，`mcp-server` 独立处理工具能力。这样后面换模型、换存储、换业务规则时，不用大面积改代码。

### 4. 你的 RAG 链路是怎么走的？

**问：** 从用户提问到返回答案，中间有哪些步骤？

**答：** 大体是“问题改写 - 意图识别 - 多路检索 - 去重重排 - 上下文组装 - 模型生成 - SSE 流式返回”。如果判断为非知识检索，还会走 MCP 工具调用。整个链路上都有 trace 记录，方便排查是哪一步出了问题。

**代码对应：** `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/*`，`framework/src/main/java/com/nageoffer/ai/ragent/framework/trace/*`。

### 5. 多路检索为什么要并行？

**问：** 为什么不只用向量检索？

**答：** 纯向量检索解决不了所有问题。企业场景里，有些问题更适合意图定向检索，有些问题适合全局召回，所以我把检索拆成多个通道并行跑，再做去重、重排和后处理。这样可以兼顾召回率和精准度。

**代码对应：** `bootstrap/.../rag/core/retrieve/MultiChannelRetrievalEngine.java`，`.../channel/*`。

### 6. 问题重写和意图识别有什么价值？

**问：** 为什么要做问题重写和意图分类？

**答：** 因为用户提问通常不完整，尤其是多轮对话里，上下文缺失会让检索偏掉。问题重写负责补全上下文，意图识别负责判断该走知识库、业务工具还是其他链路。这样能避免“用户想问 A，系统拿着 B 去检索”的情况。

### 7. 文档入库流程怎么设计的？

**问：** 知识库入库流程你怎么做的？

**答：** 我把入库做成节点化 Pipeline，从抓取、解析、增强、切分到向量化，再写入 Milvus。每个节点单独配置、单独执行、单独记录日志，这样出问题时能定位到具体哪一步失败，而不是只看到“入库失败”。

**代码对应：** `bootstrap/.../knowledge/service/impl/KnowledgeDocumentServiceImpl.java`，`docs/examples/pdf-ingestion-example.md`。

### 8. 为什么要做模型路由和降级？

**问：** 生产环境为什么不能只接一个模型？

**答：** 因为单模型供应商不可能永远稳定。模型路由可以按优先级选择供应商，首包探测和健康检查可以提前判断响应是否正常，降级机制可以保证某个模型故障时服务还能继续跑，避免整条问答链路挂死。

**代码对应：** `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/*`，`infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/*`。

### 9. 你的并发限流怎么做的？

**问：** 高并发提问时怎么防止系统被打爆？

**答：** 我做的是队列式并发限流。请求先进入排队逻辑，再通过 Redis 的信号量、ZSET 和 Pub/Sub 做分布式排队与唤醒，控制最大并发数和等待时间。这样既能削峰，也能避免不同实例之间互相抢资源。

**代码对应：** `bootstrap/.../rag/aop/ChatQueueLimiter.java`，`bootstrap/.../rag/config/RAGRateLimitProperties.java`。

### 10. 为什么要用 SSE 流式输出？

**问：** 你为什么不用普通同步返回？

**答：** 因为大模型回答时间长，SSE 可以让用户先看到首包内容，体感更好，也能更容易做中途取消和状态展示。对话场景里，流式输出比一次性阻塞返回更接近真实产品体验。

**代码对应：** `framework/src/main/java/com/nageoffer/ai/ragent/framework/web/SseEmitterSender.java`，`bootstrap/.../rag/service/handler/*`。

### 11. MCP 工具是怎么接的？

**问：** 你怎么把业务工具和问答系统接起来？

**答：** 当意图判断结果不是知识检索时，系统会走 MCP 参数提取和工具调用链路。这样问答系统不是只会“聊天”，还能把业务能力挂进去，比如查票、查单、查业务信息，形成更完整的 Agent 能力。

**代码对应：** `bootstrap/.../rag/core/mcp/*`，`mcp-server/*`。

### 12. 全链路追踪有什么用？

**问：** 你为什么要做 trace？

**答：** RAG 链路很长，问题可能出在重写、召回、重排、模型调用任意一环。trace 的价值就是把每一步都记录下来，方便定位“到底是没召回到，还是召回到了但模型没用上”，这样调优会快很多。

**代码对应：** `framework/src/main/java/com/nageoffer/ai/ragent/framework/trace/*`，`bootstrap/.../rag/aop/RagTraceAspect.java`。

### 13. 这个项目你觉得最难的地方是什么？

**问：** 最难的问题是什么？

**答：** 最难的是把“能跑”做成“能稳定跑”。RAG 不是模型一接就完事，真正难的是检索质量、并发控制、模型容错和链路可观测性。我做这个项目时一直在平衡效果、稳定性和工程复杂度，这也是它最接近企业落地的地方。

## 三、实习项目高频问答

### 1. 你在实习里主要做了什么？

**问：** 你实习期间具体负责哪部分？

**答：** 我主要负责报名模块的表结构设计和接口开发、数据导出和定时推送、初赛评审核心流程、以及数据服务接口快速交付。除此之外，我也参与了需求梳理、联调测试和部分问题排查。

### 2. 报名模块的表结构为什么这样拆？

**问：** 报名模块你是怎么设计表的？

**答：** 我是按“报名主体 + 组织机构 + 团队成员 + 附件 + 关联记录”来拆的。这样做的好处是主信息和子信息分离，既便于报名流程扩展，也方便后面做评审、导出和统计，不会把所有字段堆在一张大表里。

### 3. 报名接口为什么要做幂等和限流？

**问：** 报名高峰期你怎么保证系统不出问题？

**答：** 报名场景天然会有重复提交和高并发，所以我在接口上做了幂等保护和限流。限流主要是防止短时间内频繁请求打爆系统，幂等和分布式锁主要是防止同一个用户重复提交造成脏数据或重复单据。

**代码对应：** `cde-gddatafair-submission/controller/BopdaRegistrationController.java`，`@RateLimiter`、`@RepeatSubmit`。

### 4. 你为什么同时用 Redis 分布式锁和 RateLimiter？

**问：** 这两个东西不是有点重复吗？

**答：** 它们解决的问题不一样。RateLimiter 更像入口流量控制，防止请求太密集；分布式锁或幂等键更像业务层保护，防止同一业务请求被重复执行。一个管“别太快”，一个管“别重复做”。

### 5. 报名接口里验证码发送流程怎么做？

**问：** 短信验证码是怎么校验的？

**答：** 先校验图形验证码，再生成短信验证码并写入 Redis，设置过期时间。发送成功后再记录短信日志，这样既保证校验链路完整，也方便后续排查短信发送失败的问题。

**代码对应：** `BopdaRegistrationController.sendSmsCode`。

### 6. 上传附件时你怎么控制文件数量和校验？

**问：** 附件上传怎么处理？

**答：** 我对空文件、空数组和单次上传数量都做了限制，避免用户一次性传太多文件导致资源问题。文件上传后统一走 OSS 服务，返回 OSS ID，后续报名和评审都通过这个 ID 做关联。

**代码对应：** `BopdaRegistrationController.uploadFiles`，`ISysOssService`。

### 7. 导出任务为什么要用 SnailJob？

**问：** 为什么不用直接同步导出？

**答：** 报名数据量大时，同步导出会占住请求线程，用户体验也差。用 SnailJob 之后，导出任务可以异步触发，任务参数还能做统一解析和控制，后续也方便扩展成全量导出、条件导出和按单位分发导出。

**代码对应：** `cde-gddatafair-job/RegistrationExportJobHelper.java`，`RegistrationPureExcelFullExportJob.java`，`RegistrationPureExcelDistributedExportJob.java`。

### 8. 导出参数你是怎么解析的？

**问：** 导出任务怎么知道导什么、要不要发邮件？

**答：** 我把任务参数拆成两部分：一部分是查询条件，一部分是控制参数。控制参数里会区分全量导出和是否发送邮件，解析时还兼容了不同格式的 JSON 输入，减少任务调度侧的出错概率。

**代码对应：** `RegistrationExportJobHelper.getControlArgsStr`、`parseControlArgs`、`parseQueryBo`。

### 9. 导出时为什么强调分页和流式处理？

**问：** 为什么不把所有数据一次性查出来？

**答：** 因为大表全量查出来很容易撑爆内存。分页和流式写入可以把内存占用控制住，尤其是 EasyExcel 这类工具适合边查边写，能更稳地处理大批量报名数据。

### 10. 为什么还要异步发邮件？

**问：** 导出完成后发邮件为什么不用主线程直接发？

**答：** 因为邮件发送本身不稳定，可能受网络、SMTP、附件大小影响。如果放在主流程里，会拖慢整个导出任务，所以我把发邮件放到线程池异步执行，让导出主链路尽快返回。

### 11. 初赛评审子系统的核心流程是什么？

**问：** 评审那块你具体做了什么？

**答：** 核心是评审员登录、查看待评审作品、查看作品详情、提交评分和结果管理。系统会根据评审员配置的赛道和赛题做权限控制，再把作品附件、基础信息和历史评审结果组装成详情页返回。

**代码对应：** `cde-gddatafair-user/service/impl/TestIrReviewServiceImpl.java`。

### 12. 你怎么防止评审重复提交？

**问：** 一个作品一个评审员为什么不会提交两次？

**答：** 提交前我会先查 `registrationId + selectedTopic + reviewerId` 这组关键条件是否已经存在有效记录。如果已经有记录，就直接拦掉，避免重复评分和脏数据。

**代码对应：** `TestIrReviewServiceImpl.submit`。

### 13. 评审详情里的附件怎么来的？

**问：** 评审页看到的附件是怎么组装出来的？

**答：** 我先从作品绑定表里查出该报名和赛题对应的 OSS ID，再通过 OSS 服务批量查文件元信息，最后按顺序组装成附件列表返回。这样评审端看到的是统一的文件信息，而不是直接暴露原始表结构。

**代码对应：** `TestIrReviewServiceImpl.resolveAttachments`、`workBindingMapper`、`ISysOssService`。

### 14. 线下附件和线上记录不一致的问题怎么处理？

**问：** 你说的批量绑定接口是怎么解决问题的？

**答：** 我做的是 ZIP 流式解压后，先读 Excel 汇总表，再按报名 ID 和路径规则去匹配作品文件。能明确匹配到赛道和赛题的，就只绑定对应项；如果是多赛道但无法明确识别，我宁愿跳过，也不做错误绑定，避免把文件错挂到别的报名上。

**代码对应：** `TestIrWorkBindServiceImpl.bindWorks`、`resolveTopicCodeFromPath`、`unzipToDirectory`。

### 15. ZIP 绑定里为什么要先查已有绑定？

**问：** 为什么不直接全量重新插入？

**答：** 因为现场数据可能已经有一部分绑定过了，重复插入会造成脏数据和重复关系。所以我先把已有绑定查出来，判断哪些报名已经完全绑定，哪些还缺文件，再只补齐缺失部分。

### 16. 路径匹配为什么不做得更激进？

**问：** 文件夹命名不标准时你怎么处理？

**答：** 我倾向于保守处理。只有当路径规则能明确匹配到报名 ID、赛道和赛题时才自动绑定；如果匹配不清楚，就记为失败或跳过，不做“猜测式绑定”。因为错绑的代价远比少绑更大。

### 17. MagicAPI 这块你具体做了什么？

**问：** 你说的快速交付接口是怎么落地的？

**答：** 我基于 MagicAPI 做了多数据源路由和接口权限控制，用配置化方式交付了 3 个查询接口。这样业务方需要新查询时，不一定要先走完整的 Java 开发流程，接口交付周期会短很多。

**代码对应：** `cde-base/cde-independent/cde-magicapi/README.md`。

### 18. 多数据源路由是怎么做的？

**问：** MagicAPI 怎么切不同库？

**答：** 我在代码层注册了多个数据源，并把 `main` 设成默认数据源。脚本里可以通过固定 key 或变量 key 去访问对应库，比如 `db.datahub`、`db['gddatafair']` 这种方式，适合不同业务库之间的查询隔离。

### 19. 你在这个实习项目里最有成就感的点是什么？

**问：** 哪块最能体现你的价值？

**答：** 我觉得是把报名高峰、导出分发、评审流转和离线文件绑定这些容易出问题的地方都做稳了。这个项目不是只写接口，而是要把流程、数据、调度和联调都接起来，能真正上线用，这一点比较有成就感。

## 四、面试官常见追问模板

### 1. 你是怎么做需求梳理的？

**答：** 我一般先把业务流程拆成主流程、异常流程和边界条件，再和产品或业务方对齐“谁发起、谁审核、谁查看、谁导出”。确认完流程后再落到表结构和接口设计，避免后面因为字段或状态流转不清楚反复改。

### 2. 你怎么做接口联调和问题定位？

**答：** 我会先看请求参数、返回值、数据库记录，再看日志和关键状态表，最后确认是不是调度、权限、缓存或者文件系统的问题。联调时我习惯把问题分成“入参不对”“数据没查到”“状态没流转”“外部服务失败”四类，排查速度会快很多。

### 3. 你做过什么取舍？

**答：** 我最大的取舍是“准确性优先于自动化率”。比如 ZIP 文件绑定这块，我宁可对不清楚的文件跳过，也不去猜赛题归属，因为一旦错绑，后面评审和导出都会跟着错。

### 4. 如果让你重做一次，你会改什么？

**答：** 我会更早把表结构、状态机和调度参数标准化，同时把一些规则抽成配置，减少硬编码。这样后面业务扩展到更多赛道或更多导出场景时，改动会更小。

## 五、背诵版总结

- `Ragent` 你要重点说：分层架构、多路检索、模型路由、队列式限流、SSE、MCP、trace。
- 实习项目你要重点说：报名表设计、幂等和限流、SnailJob 导出、EasyExcel 流式处理、评审流程、ZIP 自动绑定、MagicAPI 快速交付。
- 面试官最爱追问的是“为什么这么设计”和“如果出错怎么兜底”，答案尽量围绕稳定性、准确性和可扩展性来讲。
